## 要約（修正版）

ソフトウェア設計レビューは従来、ドキュメントベースの検査に依存してきた。本論文では Design-Gap Exploration (DGE) を提案する。DGE は、性格の明確なキャラクター原型を用いて対立的な会話劇を生成し、仕様のギャップを発見する対話駆動型の方法論である。**キャラクター選択はギャップのカバレッジを決定する設計判断である。不適切なキャラクター選択は盲点を生み、多様なキャラクターの組み合わせは単一の視点では発見できないギャップ型を発見する。**

5つの質問型（今泉メソッド）、技術的文脈設定のための語り手ロール、Observe-Suggest-Act パターンで方法論を形式化する。

2つのケーススタディと統制パイロットスタディで評価する。月間10億トランザクションを処理する本番パーサーフレームワークでは、5回の DGE セッションで108件のギャップを発見（偽陽性率6%）。AI エージェントオーケストレーションプラットフォームでは、11セッションで14,978行の設計文書と85件のギャップを生成（22%が人間のレビューで発見）。3名の開発者による統制交差パイロットスタディでは、DGE は同じ成果物に対して従来の仕様レビューの2.4倍のギャップと2.0倍のギャップ型を1.6倍の時間で発見した。

AI coding agent と統合可能なポータブル DGE ツールキット（12ファイル、957行）を公開する。**ツールキットのキャラクターカタログは出発点として扱うべきであり、チームは自身の文化的文脈からキャラクターを作成することを推奨する。**

## 本文の主要変更点

### 第5.3節（統制研究 — 限定句追加）

**3名の開発者によるパイロットスタディ**で2つの API 仕様を交差デザインでレビューし、DGE は1.6倍の時間で2.4倍のギャップを発見した。サンプルサイズは小さく、結果は予備的なものとして解釈すべきだが、全3名の開発者で一貫して DGE の方が多くのギャップを発見したこと、およびギャップ型の多様性が2.0倍増加したことから、より大規模な評価に値する方法論であることが示唆される。

**評価者間一致度:** Cohen's κ = 0.71（実質的一致）。生の一致: 56件のギャップ評価中47件が一致（84%）。

### 第3.3節（今泉メソッド — タグ付けの注記）

**タグ付け方法論に関する注記:** ギャップから質問型へのマッピングは独立した評価者ではなく論文の著者が実施した。確証バイアスの可能性がある。今後の研究では独立した評価者による分布の検証を行うべきである。

### 第4.2節（セットアップ — ツール互換性）

統合手順は Claude Code の `.claude/skills/` サポートを前提としている。他の AI ツール（Cursor, Copilot, Devin）を使用する開発者には、DGE ツールキットを参照文書として使用できる。開発者が `method.md` と `characters/catalog.md` を読み、AI ツールにキャラクターの説明とシーン構造を手動でプロンプトする。他ツールの拡張メカニズムとの統合は今後の課題である。

### 第5.4節（文化的移転可能性 — ガイダンス追加）

DGE ツールキットのキャラクターカタログは出発点として扱うべきであり、普遍的なセットではない。チームは `characters/custom-guide.md` を使用して自身の文化的文脈からキャラクターを作成することを推奨する。中国文学に馴染みのあるチームはヤン・ウェンリーよりも諸葛亮を自然に感じるかもしれないし、西洋のビジネス文化に馴染みのあるチームは鷲津キャラクターよりもウォーレン・バフェットのアーキタイプを好むかもしれない。**原理は不変: 「名前が interface」は LLM が豊富な学習データを持つ任意のキャラクターで機能する。**

### 第6節（議論 — キャラクター選択の強調、B6 対応）

### 6.3 設計判断としてのキャラクター選択

失敗事例 F3（第5.5節）は本論文の最も重要な実践的発見である: **キャラクター選択がギャップのカバレッジを決定する。** セキュリティレビューに建設的キャラクターのみを使用するとセキュリティギャップは生まれない。初期設計に対立的キャラクターのみを使用すると、創造なき破壊を生む。

推奨:
1. **初期設計セッション:** 建設的キャラクター2-3体（前提確認に今泉、スコープに僕、品質/簡素化にヤンまたは千石）
2. **レビューセッション:** 対立的キャラクター2-3体（ビジネスに大和田、隠れた問題にハウス、実装現実にリヴァイ）
3. **セッションあたり最低3体** で視点の多様性を確保
4. **セッションあたり最大5体** で会話劇のレビュー可能性を維持

キャラクター選択は一度限りの判断ではない。プロジェクトが設計から実装、保守へと進化するにつれ、適切なキャラクターは建設的から対立的、運用的へとシフトする。
