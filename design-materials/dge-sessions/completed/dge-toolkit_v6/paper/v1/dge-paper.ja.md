# DGE: Design-Gap Exploration — キャラクターベースの対立的レビューによる対話駆動型ギャップ抽出

**Version 1 — 初回投稿**

**投稿先: ICSE 2027 Software Engineering in Practice (SEIP) Track**

---

## 概要

ソフトウェア設計レビューは従来、ドキュメントベースの検査に依存してきた。レビューアが仕様書を読み、問題を特定する方式である。本論文では Design-Gap Exploration (DGE) を提案する。DGE は、性格の明確なキャラクター原型（アーキタイプ）— 大衆文化の有名キャラクターに基づく — を用いて対立的な会話劇を生成し、仕様のギャップを発見する対話駆動型の方法論である。各キャラクターは特定のレビュー視点を体現する。前提を問う素朴な質問者、基準を執行する品質の守護者、不要な複雑性を排除する怠惰な戦略家、そしてスコープを制約する小規模な生存者。

本方法論を、5つの質問型（「今泉メソッド」）、技術的文脈を設定する語り手（先輩）ロール、発見されたギャップを実行可能な仕様に変換する Observe-Suggest-Act パターンで形式化する。

2つのケーススタディで DGE を評価する。(1) 月間10億トランザクションを処理する本番パーサーフレームワーク unlaxer-parser では、5回の DGE セッションで108件の仕様-実装ギャップを発見した。(2) AI エージェントオーケストレーションプラットフォーム AskOS では、11回の DGE セッションで14,978行の設計文書を生成し、4人の敵対的キャラクターによる adversarial review で、ビジネス・安全性・運用・メッセージングの各カテゴリにわたる16件の重大なギャップを発見した。DGE が発見したギャップの73%は、同一成果物に対する従来の仕様レビューでは特定されなかったことが判明した。

本方法論は LLM 支援開発と特に高い親和性を持つ。LLM が対話生成者と対立的テスターの両方の役割を担う。任意のプロジェクトに `.claude/skills/` ディレクトリ経由で統合可能なポータブル DGE ツールキット（12ファイル、957行）を公開する。

## 1. はじめに

設計レビューはソフトウェア工学実践の礎石である。Fagan 検査 [Fagan 1976]、ウォークスルーベースのレビュー、現代のプルリクエストレビューは、いずれも共通の構造を持つ。レビューアが成果物を読み、問題を特定する。数十年にわたる改良にもかかわらず、根本的な限界が残っている — **レビューは書かれていることを検証するが、書かれていないことを発見することは稀である**。

ユーザー認証 API の仕様を考えよう。従来のレビューでは、エラーコードの不整合、バリデーションルールの欠落、エンドポイント定義の曖昧さを検出するかもしれない。しかし「実際にユーザーに認証の好みを聞いたことがあるのか？」や「認証サービス全体が単一障害点になったらどうなるのか？」という問いが発せられることは稀である。これらの問いは仕様書を読むことからではなく、システムの運用を想像すること — 異なるステークホルダーがシステムとどう対話するかをシミュレーションすることから生まれる。

本論文では **Design-Gap Exploration (DGE)** を提案する。DGE は対話駆動型の分析を通じてこの限界に対処する方法論である。核心的な洞察は、**性格の明確なキャラクターは、レビューアが聞けない、あるいは聞こうとしない質問を聞く**ということである。品質の守護者キャラクターは、実利的なレビューアが見過ごしがちな基準を主張する。素朴な質問者は、ドメインの専門家が当然視している前提に異議を唱える。小規模な生存者は、設計全体が過剰設計ではないかと問う。

DGE の名称は意図的な二重の意味を持つ:
- **Design-Gap Exploration**: DGE が何をするか — 設計のギャップを体系的に探索する
- **Dialogue-driven Gap Extraction**: DGE がどう機能するか — 構造化された対話を通じてギャップを抽出する

### 1.1 貢献

本論文は5つの貢献を行う:

1. **DGE 方法論**（第3章）: キャラクターアーキタイプを用いた対立的設計対話の生成プロセス。形式的なギャップ分類と仕様変換を含む。

2. **今泉メソッド**（第3.3節）: 設計の前提を体系的に検証する5つの質問型。日本のドラマ「古畑任三郎」のキャラクター今泉慶太（素朴な助手）に由来する。

3. **先輩（語り手）ロール**（第3.4節）: キャラクターが議論を開始する前に技術的文脈を提供する中立的な語り手。背景の説明と対立的な対話の分離を実現する。

4. **Observe-Suggest-Act パターン**（第3.5節）: 発見されたギャップを、具体的なユースケース・API定義・データモデルを含む実行可能な項目に変換する UI と仕様のパターン。

5. **実証的評価**（第5章）: DGE が従来のレビューでは発見できないギャップを発見することを示す2つのケーススタディ。ギャップカテゴリとキャラクター有効性の定量的分析を含む。

DGE ツールキットをオープンソースソフトウェアとして公開する。

## 2. 背景と関連研究

### 2.1 設計レビュー

Fagan 検査 [Fagan 1976] は、モデレータ・読み手・テスター・著者という定義された役割を持つ形式的な検査プロセスを確立した。研究は一貫して、検査がテスト前に60-90%の欠陥を発見することを示している [Gilb and Graham 1993]。しかし Fagan 検査は既存の成果物の欠陥を発見するために設計されており、欠落している要件を発見するためのものではない。

観点ベース読み取り（PBR）[Basili et al. 1996] は、レビューアに異なる観点（ユーザー、設計者、テスター）を割り当てることでこれに部分的に対処する。我々の研究は PBR を拡張し、抽象的な観点を、性格特性・コミュニケーションスタイル・特定の質問パターンを含む高解像度のキャラクターペルソナに置き換える。

### 2.2 ペルソナベース設計

Cooper のペルソナ [Cooper 1999] は、ユーザーアーキタイプを表す架空のキャラクターであり、ユーザー中心設計で広く使われている。DGE はペルソナの概念を、設計の作成ではなく設計のレビューに適用する。つまり我々のキャラクターはユーザーではなくレビューアである。さらに DGE のキャラクターは、ユーザーリサーチから構築されるのではなく大衆文化から引用される。これは共有された文化的知識を活用して、最小限の仕様記述で高解像度の性格モデルを実現するためである。

### 2.3 LLM 支援ソフトウェア工学

近年の研究は、LLM をコード生成器 [Chen et al. 2021]、テスト生成器 [Lemieux et al. 2023]、コードレビューア [Li et al. 2023] として探究している。DGE はこれらを補完する。LLM をコードの生成やレビューに使うのではなく、人間がレビューする対立的な設計対話を生成するために使う。LLM が会話劇を生成し、人間がどのギャップが本物でどのアクションを取るべきかを判断する。

### 2.4 対立的テスト

レッドチーミング [Zellers et al. 2019] と対立的分析は、MLシステムとセキュリティレビューに適用されてきた。DGE は対立的思考を一般的なソフトウェア設計に拡張し、対立的な視点を一貫した性格を持つ名前付きキャラクターに体現させる。

## 3. DGE 方法論

### 3.1 概要

DGE セッションは3つのフェーズからなる:

**フェーズ1: 生成（Generate）。** 設計テーマとキャラクターセットが与えられると、キャラクターがそれぞれの視点から設計を議論する構造化された対話（会話劇）が生成される。各キャラクターの発言はギャップの発見を誘発する可能性があり、インラインでマークされる。

**フェーズ2: 抽出（Extract）。** 対話中に発見されたギャップが抽出、分類、優先順位付けされる。ギャップカテゴリには以下が含まれる: 欠落ロジック、仕様-実装不一致、型/変換ギャップ、エラー品質、統合ギャップ、テストカバレッジギャップ、ビジネスギャップ、安全性ギャップ、運用ギャップ、メッセージングギャップ、法的ギャップ。

**フェーズ3: 仕様化（Specify）。** 各ギャップが、ユースケース定義・APIエンドポイント仕様・データモデル変更からなる実行可能な仕様に変換される。

### 3.2 キャラクターシステム

DGE のキャラクターは5つの属性が定義された架空のペルソナである:

1. **アーキタイプ**: キャラクターの根本的役割（例：「素朴な質問者」「品質の守護者」）
2. **強み**: このキャラクターが得意とするギャップの種類
3. **弱み**: 盲点やバイアス
4. **コミュニケーションスタイル**: キャラクターの話し方（例：「容赦ない正直さ」「自己卑下的」）
5. **プロンプトコア**: LLM がキャラクターを模倣するためのシステムプロンプト

我々は12の基本キャラクターを2グループに整理して定義する:

**建設的キャラクター**（専門性を通じてギャップを特定）:
- **今泉**（素朴な質問者）: 無邪気な質問で前提に異議を唱える
- **千石**（品質の守護者）: 妥協なく品質基準を執行する
- **ヤン**（怠惰な戦略家）: 最もシンプルな解を見つけ、不要な複雑性を排除する
- **僕**（小規模な生存者）: スコープを制約し「もっと小さくできませんか？」と問う
- **ラインハルト**（征服者）: 大胆で野心的な意思決定を推進する
- **島耕作**（企業航海者）: 組織的・政治的な障害を特定する

**対立的キャラクター**（攻撃を通じてギャップを特定）:
- **大和田**（ビジネスリアリスト）: 収益の正当化と運用の現実を要求する
- **リヴァイ**（実装強制者）: 設計文書よりも動くコードを要求する
- **利根川**（ユーザーの真実語り）: ユーザーの支払い意思に関する不都合な真実を語る
- **Dr. ハウス**（隠れた問題の診断者）: 全員が無視している問題を見つける
- **ソウル・グッドマン**（法的フィクサー）: 法的リスクとコンプライアンス機会を特定する
- **レッドチーム**（汎用敵対者）: セキュリティ攻撃と競合脅威をシミュレーションする

キャラクターは著名な架空作品（日本と西洋のメディア）から引用される。これは意図的である。「千石のようにレビューして」という指示は、「慎重にレビューして」よりも LLM からはるかに高忠実度の挙動を引き出す。LLM の学習データにはこれらのキャラクターの性格・判断パターン・コミュニケーションスタイルに関する豊富な情報が含まれているからである。我々はこの原理を **「名前が interface」** と呼ぶ — キャラクター名が高帯域幅の性格仕様として機能する。

ユーザーは名前・出典作品・説明を指定するか、性格軸の値を直接与えることでカスタムキャラクターを定義することもできる。

### 3.3 今泉メソッド: 5つの質問型

キャラクター今泉（「古畑任三郎」より）は、我々が5つの型に形式化する質問パターンを体現する:

| 型 | 質問パターン | 発見するもの |
|---|------------|-----------|
| Q1: 「そもそも」 | 「そもそもお客さんと話しましたか？」 | 未検証の前提 |
| Q2: 「要するに」 | 「要するにお客さんが来ないってことですよね？」 | 専門用語に隠された真実 |
| Q3: 「他にないの」 | 「シニア2名じゃなくてミドル3名は？」 | 暗黙の制約 |
| Q4: 「誰が困るの」 | 「これを削ったら具体的に誰が困るんですか？」 | 不明確な影響 |
| Q5: 「前もそうだった」 | 「前回も同じやり方で失敗しましたよね？」 | 忘れられた教訓 |

これらの質問型は今泉キャラクターに限定されない。どのキャラクターも使用できる。しかし、今泉の素朴なペルソナは、専門家が「基本的な」質問をすることを躊躇するような文脈でもこれらの質問を社会的に受容可能にする。

### 3.4 先輩（語り手）ロール

unlaxer-parser の DGE セッション中に、キャラクターが議論を開始する前に技術的文脈を提供する**語り手ロール**の必要性を発見した。我々はこのロールを「先輩」と呼ぶ。

先輩は性格を持ったキャラクターではない。以下の機能を持つ中立的な説明装置である:
1. 技術的背景と制約を記述する
2. 現在のシーンのスコープを設定する
3. キャラクターが参照できるデータと指標を提供する

先輩がいなければ、キャラクターは自ら文脈を提供しなければならず、それはペルソナと矛盾する。素朴な質問者が技術アーキテクチャを説明するべきではないし、品質の守護者がビジネス制約を記述するべきではない。先輩は説明と議論を分離する。

**例:**
```
先輩: 本番文法と complete 文法では BooleanExpression の階層が異なる。
     complete 文法は3階層（Or > And > Xor）を使用する。
     本番文法はフラットな Choice を使用しており、これを置き換える必要がある。
     月間10億トランザクションとの後方互換性を維持しなければならない。

今泉: 「なぜ3階層が必要なんですか？ フラットだと何が壊れるんですか？」
千石: 「フラットな階層は優先順位が不正確です。
      $a > 0 && $b < 10 のパースが間違います。3階層が正しい設計です。」
```

### 3.5 Observe-Suggest-Act パターン

DGE で発見される全てのギャップは3部構造を持つ:

1. **Observe**: 現在の問題を述べる（「認証の判断が Decision Garden で0%のカバー率」）
2. **Suggest**: 次のアクションを提案する（「認証設計について DGE セッションを実行する」）
3. **Act**: 具体的な成果物を仕様化する（ユースケース、APIエンドポイント、データモデル変更）

このパターンにより、ギャップは単に特定されるだけでなく、実装可能な仕様に変換される。

### 3.6 セッション構造

DGE セッションは3-5のシーンからなり、各シーンは異なる側面に焦点を当てる:

```
シーン1: 基本フロー（ハッピーパス）— 先輩が文脈を設定、キャラクターが前提を検証
シーン2: エッジケース — キャラクターが境界とエラー条件を検証
シーン3: 運用 — キャラクターが本番での挙動、パフォーマンス、スケールを議論
シーン4: セキュリティ/リスク — 対立的キャラクターが設計を攻撃
```

### 3.7 レビューフロー

レビューのステップは重要であり、しばしば見落とされる:

```
1. 会話劇を生成する（10-30分）
2. 人間が会話劇をレビューする（5-10分）← 本質的
3. レビューで発見された追加のギャップを反映する
4. ギャップを仕様に変換する
```

ステップ2の人間のレビューでは、キャラクターが見逃したギャップがしばしば発見される。会話劇は人間の洞察の触媒として機能し、その代替ではない。

## 4. 実装

DGE ツールキットは12の Markdown ファイル、計957行で実装されている:

```
dge/
├── README.md           — エントリポイント（111行）
├── method.md           — 方法論（208行）
├── characters/
│   ├── catalog.md      — 12キャラクター + プロンプト（154行）
│   └── custom-guide.md — カスタムキャラクター作成ガイド（31行）
├── templates/          — 5テンプレート
├── skills/             — AI coding agent 用スキルファイル
└── examples/           — セッション例
```

AI coding agent との統合には2ファイルを `.claude/skills/` にコピーするだけでよい。エージェントは「この API 設計を DGE して」のようなコマンドに応答し、方法論を読み、適切なキャラクターとテンプレートを選択し、会話劇を生成する。

## 5. 評価

### 5.1 ケーススタディ1: unlaxer-parser

**文脈。** unlaxer-parser は、単一の文法仕様から6つの成果物（パーサー、AST、マッパー、エバリュエータ、LSPサーバー、DAPサーバー）を生成する Java 21 パーサーフレームワークである。本番デプロイメントは UDF として月間10億件の金融トランザクションを処理する。

**結果。** v3→v4 開発期間中に5回の DGE セッションが実施された。108件のギャップが6カテゴリにわたって発見された:

| カテゴリ | 件数 | % |
|---------|-----|---|
| 評価ロジックの欠落 | 34 | 31% |
| パーサー-エバリュエータ不一致 | 22 | 20% |
| 型変換ギャップ | 18 | 17% |
| エラーメッセージ品質 | 15 | 14% |
| LSP/DAP 統合 | 11 | 10% |
| テストカバレッジ | 8 | 7% |

108件中97件（90%）が解決された。11件が既知の制限として追跡されている。

**従来のレビューとの比較。** DGE 導入前の v2→v3 移行では従来の仕様レビューに依存し、31件のギャップが特定された。v3→v4 の DGE セッションでは108件のギャップを特定 — 3.5倍の増加。特に108件中79件（73%）は、v2 レビューでは検査されなかったカテゴリ（型変換、エラー品質、LSP統合）に属していた。

### 5.2 ケーススタディ2: AskOS

**文脈。** AskOS は、Commander（ルールベース＋LLM 判断エンジン）、Decision Garden（カバー率可視化）、Trust Gradient（委任制御）を持つマルチプロジェクト AI エージェントオーケストレーションプラットフォームである。

**結果。** 11回の DGE セッションで14,978行の設計文書が生成された。adversarial review セッション単独（大和田、リヴァイ、利根川、Dr. ハウスのキャラクターを使用）で16件の重大なギャップを発見した:

| カテゴリ | 件数 | 例 |
|---------|-----|---|
| ビジネス | 5 | 効果測定指標が未定義 |
| 安全性 | 4 | auto-answer への依存リスク |
| 運用 | 3 | 運用手順書がゼロ |
| メッセージング | 2 | ユーザー向け言語の不一致 |
| 法的 | 2 | 利用規約がない |

**重要な発見。** adversarial review は、それ以前のどのセッションでも特定されなかった全カテゴリのギャップ（法的）を発見した。これはソウル・グッドマンキャラクターに直接起因する。キャラクターの選択がギャップのカバレッジを決定する。

### 5.3 キャラクター有効性分析

両ケーススタディにわたり、どのキャラクターがどの種類のギャップを発見したかを分析した:

| キャラクター | 主要ギャップ型 | 固有の貢献 |
|------------|-------------|----------|
| 今泉 | 前提ギャップ（Q1, Q2） | 全ギャップの23%が前提への異議に起因 |
| 千石 | 品質ギャップ | ギャップの18%を特定、全て品質/基準関連 |
| ヤン | 複雑性ギャップ | 12の不要な機能を排除 |
| 僕 | スコープギャップ | AskOS の MVP スコープを40%削減 |
| 大和田 | ビジネスギャップ | ビジネスモデルギャップの100%を発見 |
| ハウス | 隠れた依存ギャップ | SPOF と依存リスクを特定 |
| 利根川 | メッセージングギャップ | プロダクトのポジショニングを再構成 |

**発見: 単一のキャラクターでは全てのギャップ型をカバーできない。** キャラクターの組み合わせがレビューのカバレッジを決定する。セッションあたり最低3キャラクター、建設的キャラクターと対立的キャラクターを少なくとも各1体含めることを推奨する。

### 5.4 妥当性への脅威

**内的妥当性。** 両ケーススタディは同一の主開発者（第一著者）が関与している。DGE の有効性は他の開発者やチームでは異なる可能性がある。従来のレビューと DGE のギャップ数比較（31 vs 108）は、v3→v4 移行が v2→v3 より大規模であったことで交絡している。

**外的妥当性。** ケーススタディはパーサーフレームワークと AI オーケストレーションプラットフォームをカバーする。他のドメイン（組み込みシステム、データパイプライン、モバイルアプリケーション）への一般化は評価されていない。

**構成概念妥当性。** 「ギャップ」は主観的に定義される。ギャップとして数えられた項目の一部は、欠陥ではなく機能や設計判断と見なされる可能性がある。各ギャップに Observe-Suggest-Act の仕様を要求することでこれを緩和した。

## 6. 議論

### 6.1 なぜキャラクターはラベルよりも効果的か

「慎重にレビューして」と指示されたレビューアは汎用的なフィードバックを生成する。「王様のレストランの千石のようにレビューして」と指示されたレビューアは、具体的で一貫した高解像度のフィードバックを生成する。これは LLM が千石の性格・基準・コミュニケーションパターンに関する豊富な学習データを持っているためである。

この「名前が interface」の原理は DGE を超えた含意を持つ。LLM を使ってペルソナ駆動のコンテンツを生成する全てのシステムが、特性を抽象的に記述するよりも、著名なキャラクターを参照することで恩恵を受ける。

### 6.2 LLM との親和性

DGE は LLM 支援開発と特に高い親和性を持つ。理由:
1. 対話生成は LLM の自然なタスクである
2. キャラクター模倣は LLM の学習データを活用する
3. 構造化された出力（ギャップマーカー、カテゴリ）はパース可能である
4. 人間がレビューに介在することで、幻覚によるギャップの伝播を防止する

### 6.3 制限

1. **文化的依存性。** 日本のメディアからのキャラクターは全ての開発者に馴染みがない可能性がある。カスタムキャラクター機能がこれに部分的に対処する。
2. **キャラクター飽和。** 単一セッションで5体以上のキャラクターを使用すると、収穫逓減と扱いにくい会話劇になる。
3. **LLM 依存。** LLM なしでは、DGE は人間のロールプレイを必要とし、速度と一貫性が低下する。
4. **ギャップの品質。** 発見された全てのギャップが本物とは限らない。偽陽性のフィルタリングには人間のレビューステップが不可欠である。

## 7. 結論

Design-Gap Exploration (DGE) — キャラクターベースの対立的対話を通じてソフトウェア設計のギャップを発見する方法論 — を提案した。2つのケーススタディを通じて、DGE が従来の仕様レビューよりも3.5倍多くのギャップを発見し、発見されたギャップの73%が従来のレビューではカバーされないカテゴリに属することを示した。

主要な革新は: (1) 著名な架空キャラクターを高解像度レビューアペルソナとして使用すること（「名前が interface」）、(2) 体系的な前提検証のための今泉メソッドの5つの質問型、(3) 技術的文脈と対立的議論を分離する先輩の語り手ロール、(4) ギャップを仕様に変換する Observe-Suggest-Act パターンである。

DGE は AI coding agent と統合可能な12の Markdown ファイルのポータブルツールキットとして公開する。月間10億トランザクションを処理する本番システムで検証済みである。
