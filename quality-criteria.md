# DGE 出力品質基準

> Gap 2-013 への対応: 出力品質の最低基準が定義されていない

---

## 「良い DGE session」の定義

良い DGE session とは、**人間が読んで「これは確かに抜けていた」と思える Gap を 1 つ以上発見する session** である。

Gap の数は品質指標ではない。100 個の自明な Gap より、1 個の「気づかなかった致命的な穴」の方が価値がある。

### 良い session の 3 条件

1. **発見性**: 人間が自力では気づかなかった Gap が含まれている
2. **具体性**: 各 Gap が「誰が・いつ・何をしたとき・何が起きるか」で記述されている
3. **行動可能性**: Gap を読んだ人間が「次に何をすべきか」を判断できる

## 出力の最低基準

### 会話劇の最低基準

| 基準 | 合格 | 不合格 |
|------|------|--------|
| キャラクターの発言数 | 全キャラが 2 回以上発言している | 1 人のキャラだけが話し続けている |
| キャラクターの区別 | 各キャラの視点・口調が明確に異なる | 全員が同じトーンで同じ観点を繰り返す |
| 先輩の context 設定 | 先輩が技術的背景を neutral に設定してから議論が始まる | 背景説明なしにいきなり議論が始まる |
| 具体性 | 「ユーザーが〇〇したとき△△が起きる」という場面描写がある | 「セキュリティが重要です」のような抽象論のみ |
| 矛盾の存在 | キャラクター間で意見の対立がある | 全員が同意して終わる |
| Scene の多様性 | Happy path 以外（エッジケース、運用、セキュリティ等）を含む | Happy path だけで終わる |

### Gap の最低基準

| 基準 | 合格 | 不合格 |
|------|------|--------|
| Observe-Suggest-Act 構造 | 3 要素が揃っている | 「〇〇が問題」とだけ書いてある |
| 分類 | 6+5 カテゴリのいずれかに分類されている | 分類なし、または「その他」ばかり |
| 重複排除 | 同じ内容の Gap が統合されている | 表現を変えただけの同一 Gap が複数存在 |
| 粒度 | 1 つの Gap が 1 つの問題を指している | 1 つの Gap に 3 つ以上の問題が混在 |
| 行動可能性 | Act に具体的な次のステップ（UC, API, 調査項目等）がある | Act が「検討すべき」「考慮が必要」で終わる |

### Session 全体の最低基準

| 基準 | 合格 | 不合格 |
|------|------|--------|
| テーマとの関連性 | 発見された Gap の 80% 以上がテーマに直接関連している | テーマと無関係な一般論の Gap が半数以上 |
| 新規性 | 少なくとも 1 つの Gap が、session 前に認識されていなかった問題である | 全ての Gap が既知の問題の再確認にすぎない |
| 人間レビューの実施 | session 後に人間がレビューし、各 Gap の採否を判断している | 会話劇の出力をそのまま仕様に反映している |

## セルフチェックリスト（session 後に自分で検証）

session 完了後、以下のチェックリストで品質を自己検証する。

### A. 会話劇の品質

- [ ] 全てのキャラクターが意味のある発言をしているか（挨拶だけのキャラはいないか）
- [ ] キャラクター間で矛盾・対立が発生しているか
- [ ] 先輩が context を設定してから議論が始まっているか
- [ ] 抽象論ではなく具体的なユーザー場面が描写されているか
- [ ] Scene が 2 つ以上あるか（Happy path + それ以外）

### B. Gap の品質

- [ ] 各 Gap に Observe / Suggest / Act が揃っているか
- [ ] 各 Gap が 6+5 カテゴリのいずれかに分類されているか
- [ ] 重複 Gap を統合したか
- [ ] 「検討すべき」「考慮が必要」で終わる Gap を具体化したか
- [ ] 少なくとも 1 つの Gap が「これは気づかなかった」と思えるか

### C. Session プロセス

- [ ] 出力をそのまま仕様に反映していないか（人間レビューを挟んだか）
- [ ] 各 Gap の採否を判断したか（全採用・全不採用は警告サイン）
- [ ] Gap の severity を付与したか
- [ ] 次のアクション（何を・誰が・いつまでに）を決めたか

### D. 信頼性の確認

- [ ] 「LLM がそう言ったから正しい」と無批判に受け入れていないか
- [ ] Gap の中に事実と異なる前提に基づくものはないか
- [ ] 自分のドメイン知識で「これは的外れ」と判断できる Gap を除外したか

## アンチパターン（こういう session は失敗）

### 1. 全員合意パターン

```
症状: キャラクター全員が「その通りですね」と同意して終わる。
原因: LLM の合意バイアス。キャラクターの personality が十分に機能していない。
対策: キャラクターの組み合わせを見直す。
       対立が生まれやすいペアを入れる（千石 vs ヤン、ラインハルト vs 僕）。
       prompt で「必ず反論してください」と明示する。
```

### 2. Gap 量産パターン

```
症状: 50 個以上の Gap が出るが、どれも浅い。「〇〇が未定義」の羅列。
原因: 会話劇が表面的。具体的な場面描写がない。
対策: Scene 数を減らし、1 つの Scene を深掘りする。
       「ユーザーが実際にやること」を先輩の context に具体的に書く。
```

### 3. 一人芝居パターン

```
症状: 1 人のキャラクターが延々と話し、他のキャラは相槌だけ。
原因: テーマが特定キャラの専門領域に偏りすぎている。
対策: キャラ数を 3 人以上にする。
       先輩の context で複数の観点（技術・ビジネス・ユーザー）を提示する。
```

### 4. コピペ仕様パターン

```
症状: 会話劇の出力をレビューせずにそのまま仕様書に貼り付ける。
原因: DGE の核心（人間レビュー）を省略している。
対策: 「会話劇 → レビュー の往復が DGE の核心。レビューなしの会話劇は片手落ち。」
       Gap ごとに「採用 / 保留 / 却下」を明示的に判断する。
```

### 5. テーマ不在パターン

```
症状: 「とりあえず DGE して」で実行し、何を検証したいか不明確。
原因: 入力（テーマ、対象ドキュメント、検証したい観点）が曖昧。
対策: session 開始前に以下を明確にする:
       - 何の設計を検証するか
       - どの観点（セキュリティ、UX、ビジネス等）で検証するか
       - 既知の懸念事項は何か
```

### 6. 権威盲信パターン

```
症状: 「LLM がこう言ったから」を根拠に設計判断する。
原因: DGE の出力を「答え」と捉えている。DGE は「問い」を生成する手法。
対策: 全ての Gap を「仮説」として扱う。
       自分のドメイン知識と照合し、採否を人間が判断する。
       Gap の中に事実誤認がないか確認する。
```

### 7. 毎回同じキャラパターン

```
症状: 毎回同じキャラの組み合わせで実行し、同じ角度の Gap しか出ない。
原因: キャラクターの使い分けを理解していない。
対策: テーマに応じて catalog.md の推奨組み合わせを参照する。
       前回出なかった角度のキャラを意図的に追加する。
```

## LLM 差異への対処（モデルが変わっても品質を保つためのガイドライン）

DGE は LLM に会話劇を生成させる手法であるため、使用する LLM のモデル・バージョンによって出力品質が変わる。以下のガイドラインでモデル非依存の品質を確保する。

### 原則: 品質は LLM ではなくプロセスで担保する

DGE の品質を決めるのは以下の順序である:

```
1. 人間のレビュー品質（最重要）
2. 入力の品質（テーマの明確さ、context の具体性）
3. テンプレート・キャラの選択
4. LLM の能力（最も影響が小さい）
```

LLM は「問いの種」を生成するだけであり、その種が有用かどうかを判断するのは人間である。

### モデル変更時のチェックポイント

| チェック項目 | 確認方法 |
|-------------|---------|
| キャラクターの区別が維持されているか | 会話劇で各キャラの口調・視点が異なるか目視確認 |
| 合意バイアスが強まっていないか | キャラクター間の矛盾・対立が発生しているか確認 |
| Gap の具体性が維持されているか | Observe-Suggest-Act が揃っているか確認 |
| 日本語の品質が十分か | キャラクターの口調が自然か確認 |
| 先輩ロールが機能しているか | 先輩が narrator として背景を設定できているか確認 |

### モデル別の既知の傾向と対策

```
合意バイアスが強いモデル:
  → prompt に「必ずキャラクター間で意見を対立させてください」を追加
  → 対立が生まれやすいキャラペアを選ぶ（千石 vs ヤン、ラインハルト vs 僕）

Gap が浅くなるモデル:
  → Scene 数を減らし「1 つの Scene を 3 往復以上議論してください」と指示
  → 先輩の context をより具体的に書く

キャラの区別が曖昧なモデル:
  → キャラクター prompt をフルで渡す（catalog.md の prompt 全文）
  → キャラ数を 2-3 人に減らす

日本語が不自然なモデル:
  → 英語で実行し、Gap のみ日本語に翻訳する
  → または、英語のキャラクター（Red Team, Dr. ハウス, ソウル）を中心に使う
```

### 再現性の確保

DGE の出力は LLM の確率的生成に依存するため、完全な再現性は保証できない。ただし以下の方法で実用的な再現性を確保できる:

1. **入力を固定する**: テーマ、テンプレート、キャラクター、先輩の context を記録する
2. **session を保存する**: 会話劇の全文と抽出した Gap を保存する
3. **複数回実行する**: 同じ入力で 2-3 回実行し、共通して出現する Gap を信頼度が高いと判断する
4. **人間レビューを記録する**: 各 Gap の採否判断とその理由を記録する

完全な再現性よりも、**異なる実行でも同じクラスの Gap が発見されること**（テーマレベルの再現性）を目指す。

## 品質基準の適用レベル

全ての session に同じ厳密さを求める必要はない。目的に応じて適用レベルを選択する。

| レベル | 用途 | 最低基準の適用 |
|--------|------|--------------|
| 探索的 | 初期アイデアの検証、ブレスト的な利用 | 会話劇の最低基準のみ。Gap の Observe-Suggest-Act は省略可 |
| 標準 | 設計レビュー、機能企画の穴埋め | 全ての最低基準を適用。セルフチェックリストを実施 |
| 厳密 | 論文の方法論、重大な意思決定 | 全ての最低基準 + 複数回実行 + 複数人レビュー + 結果の保存 |
