# DGE ユーザーペルソナ

> Gap 2-021 への対応: ユーザーペルソナが定義されていない

---

## ペルソナ 1: 高橋亮太 — ソフトウェアエンジニア

### プロフィール

| 項目 | 内容 |
|------|------|
| 役割 | バックエンドエンジニア（5年目） |
| 経験レベル | 中級〜上級。設計レビューは受ける側が多く、リードする機会が増えてきた |
| 使用ツール | Claude Code を日常的に使用。VS Code, GitHub, Slack |

### DGE を使う動機（本人の言葉）

> 「設計レビューで毎回『他に抜けてない？』って聞かれるけど、自分で抜けを見つける方法がわからない。
> レビューに出す前に自分で穴を潰しておきたい。恥をかきたくない。」

### 期待する成果物

- API 設計ドキュメントの抜け漏れリスト
- レビュー前に自分で確認できる Gap 一覧
- 具体的な Use Case / Edge Case の洗い出し

### 前提知識

- REST API 設計の基本は理解している
- LLM をコード生成に使った経験あり
- DGE のような構造化された設計レビュー手法は初めて

### 導入の障壁

- 「会話劇」というフォーマットに最初は違和感がある
- キャラクターの元ネタを知らないと感情移入しにくい
- 自分の設計に対して辛辣な指摘を受けることへの心理的抵抗
- session の結果をどこまで信用していいかわからない

### 推奨する最初のテンプレート・キャラ組み合わせ

| テンプレート | キャラ | 理由 |
|-------------|--------|------|
| `api-design.md` | 今泉 + 千石 + 僕 | 今泉が前提を問い、千石が品質を守り、僕が scope を現実的に抑える。エンジニアの日常の設計レビューに最も近い構成 |

## ペルソナ 2: 山田美咲 — プロダクトマネージャー

### プロフィール

| 項目 | 内容 |
|------|------|
| 役割 | SaaS プロダクトの PM（3年目） |
| 経験レベル | 中級。機能企画はできるが、技術的な設計判断は苦手 |
| 使用ツール | Claude Code は使ったことがない。Notion, Figma, Jira |

### DGE を使う動機（本人の言葉）

> 「新機能の企画書を書いても、開発チームから『これ、こういうケース考えてますか？』と
> 毎回突っ込まれる。企画段階で穴を見つけられたら、開発との手戻りが減るはず。
> ユーザーが本当に困ることを先に洗い出したい。」

### 期待する成果物

- 機能企画の穴の一覧（ビジネス観点 + ユーザー観点）
- 「この機能、本当に必要？」への回答材料
- ステークホルダーへの説明に使える考慮事項リスト

### 前提知識

- ユーザーストーリーの書き方は知っている
- API や Data Model の詳細は書けない
- LLM は ChatGPT を雑談に使う程度

### 導入の障壁

- Claude Code のセットアップ自体がハードル
- `method.md` が技術者向けの記述で、PM には読みにくい
- 会話劇の結果を Spec（Use Case, API, Data Model）に落とすステップは自分ではできない
- 「Gap を見つけた後どうするか」が不明確

### 推奨する最初のテンプレート・キャラ組み合わせ

| テンプレート | キャラ | 理由 |
|-------------|--------|------|
| `feature-planning.md` | 今泉 + ヤン + 僕 | 今泉が「そもそもこの機能要るの？」を問い、ヤンが最もシンプルな解を提示し、僕が scope を縮小する。PM の企画段階に最適な構成 |

## ペルソナ 3: 田中誠一 — 研究者

### プロフィール

| 項目 | 内容 |
|------|------|
| 役割 | ソフトウェア工学の博士課程学生 / 若手研究者 |
| 経験レベル | 上級（研究能力）。ツール利用は中級 |
| 使用ツール | LaTeX, Git, Python。Claude Code は論文執筆補助で使用開始 |

### DGE を使う動機（本人の言葉）

> 「自分の研究のツール設計に DGE を使いたい。論文の方法論セクションで
> 『対話型 Gap 抽出手法を用いて設計を検証した』と書けたら説得力がある。
> unlaxer-parser の SLE 2026 採択事例があるので、再現可能な方法論として
> 自分の研究にも適用できるか試したい。」

### 期待する成果物

- 研究対象ソフトウェアの設計 Gap の体系的な一覧
- 論文に記載できる形式の方法論記述と結果
- Gap の分類・severity の定量的な分布データ
- 手法の再現性を示すための手順書

### 前提知識

- ソフトウェア工学の研究手法（形式手法、設計パターン等）に精通
- 既存のレビュー手法（STRIDE, ADR, Threat Modeling 等）を理解
- LLM の限界（hallucination, bias）を学術的に理解している

### 導入の障壁

- DGE の有効性を示す定量的エビデンスが不足（Gap 1-007, 1-008）
- 再現性の保証がない（LLM のバージョンや温度設定で結果が変わる）
- Gap の precision / recall が未計測で、論文に「信頼できる手法」と書きにくい
- 比較実験のデザインが存在しないため、自分で実験設計から始める必要がある

### 推奨する最初のテンプレート・キャラ組み合わせ

| テンプレート | キャラ | 理由 |
|-------------|--------|------|
| `api-design.md` | 今泉 + 千石 + Red Team | 今泉が前提を検証、千石が品質基準を維持、Red Team が adversarial な観点を提供。研究の厳密性を求める場面に適した構成。自分の研究対象に合わせてテンプレートを `dge-template-create.md` でカスタマイズすることを推奨 |

## ペルソナ 4: 佐藤ひなた — 初心者

### プロフィール

| 項目 | 内容 |
|------|------|
| 役割 | Web 開発者（1年目）/ 独学プログラマー |
| 経験レベル | 初級。チュートリアルを終えて、初めて自分のアプリを作っている |
| 使用ツール | Claude Code を使い始めたばかり。VS Code, GitHub |

### DGE を使う動機（本人の言葉）

> 「Claude Code でコード書いてもらってるけど、設計が合ってるかわからない。
> 誰かにレビューしてもらいたいけど、チームメンバーがいない。
> 一人でも設計の抜けを見つける方法がほしい。」

### 期待する成果物

- 自分のアプリ設計の「ここが危ない」リスト
- 何を直せばいいかの具体的な方向性
- 設計を考える力そのものの成長

### 前提知識

- プログラミングの基礎（変数、関数、HTTP）は理解
- 設計パターンや API 設計のベストプラクティスは未学習
- LLM は Claude Code でコード生成に使っている
- 「仕様書」を書いたことがない

### 導入の障壁

- `method.md` の用語（Use Case, Data Model, severity, precision）が難しい
- 3 ステップ（Generate → Extract → Specify）の Specify が自力でできない
- キャラクターが多すぎて誰を選べばいいかわからない
- Gap が大量に出たとき、どれから手をつけるべきか判断できない
- 「先輩」ロールに渡す技術的背景を自分で書けない

### 推奨する最初のテンプレート・キャラ組み合わせ

| テンプレート | キャラ | 理由 |
|-------------|--------|------|
| `feature-planning.md` | 今泉 + 僕 | 最小構成。今泉の「そもそも」質問で前提を確認し、僕の「小規模にしませんか」で scope を現実的に保つ。キャラ 2 人なら会話劇も短く、結果の理解もしやすい。慣れたら千石やヤンを追加する |

## ペルソナ横断: 共通の導入パス

```
Step 0: README.md を読む（全ペルソナ共通、5分）
Step 1: 推奨テンプレート + 推奨キャラで 1 回実行する
Step 2: 出力された Gap を人間がレビューする（★ここが最重要）
Step 3: 「これは確かに抜けてた」と思った Gap を 1 つ解決する
Step 4: 2 回目以降はキャラを追加し、テンプレートを変えて試す
```

## ペルソナ別の注意点

| ペルソナ | 最大のリスク | 対策 |
|---------|------------|------|
| 高橋（エンジニア） | Gap を全部直そうとして時間を使いすぎる | severity でフィルタし、Critical / High のみ対処 |
| 山田（PM） | Specify ステップが自力でできない | エンジニアとペアで Specify を実施する |
| 田中（研究者） | 再現性にこだわりすぎて session を始められない | まず 1 回実行し、その結果から実験デザインを考える |
| 佐藤（初心者） | Gap の量に圧倒されて挫折する | キャラ 2 人の最小構成から始め、Gap は上位 3 つだけ見る |
