# Gap の定義と管理基準

> 対応する Gap:
> - 1-015: Gap severity の判定基準（Critical / High / Medium / Low）が未定義
> - 1-016: Gap の粒度基準（1 gap の範囲の定義）が未定義
> - 3-005: gap 自体の信頼性検証メカニズムがない（偽陽性リスク）
> - 3-025: gap の優先度付け（severity / probability / impact）フレームワークが未定義

---

## 1. Gap の定義

**Gap とは、仕様書・設計文書において「未定義」「矛盾」「考慮漏れ」のいずれかに該当し、対処しなければユーザーまたはシステムに具体的な影響を及ぼす欠落のことである。**

### Gap でないもの

以下は Gap ではない。DGE session で出現しても除外する。

| 除外対象 | 理由 | 例 |
|----------|------|-----|
| 要望・改善希望 | 現状が壊れているわけではない | 「UI をもっとモダンにしたい」 |
| 主観的な感想 | 検証不能 | 「この設計は美しくない」 |
| 既知の仕様制限 | 意図的に定義済み | 「オフライン時は使えない」（仕様書に明記済み） |
| 技術的な好み | 正解が複数ある設計判断 | 「REST より GraphQL にすべき」 |
| LLM の知識不足に起因する指摘 | 事実に基づいていない | ドメイン固有の仕様を「未定義」と誤認 |

判断に迷ったら: **「この指摘を放置したとき、誰がどう困るか」を具体的に説明できなければ Gap ではない。**

## 2. Gap の粒度基準

### 基本原則

**1 Gap = 1 つの「未定義・矛盾・考慮漏れ」**

### 分割の原則

異なる担当者が独立に対処できるなら、別の Gap に分ける。

```
質問: 「この Gap を 2 人に分担できるか？」
  → Yes: 分割すべき
  → No: 1 つの Gap のまま
```

### 結合の原則

同じ根本原因から派生している複数の症状は、1 つの Gap にまとめる。

```
質問: 「根本原因を 1 つ修正したら、全部直るか？」
  → Yes: まとめるべき
  → No: 別の Gap
```

### 粒度の例

**良い例（適切な粒度）:**

| Gap | 理由 |
|-----|------|
| 「ログイン失敗時のリトライ上限が未定義」 | 1 つの未定義事項。1 人で対処可能 |
| 「管理者と一般ユーザーの権限境界が未定義」 | 1 つの未定義事項。権限設計者が対処 |
| 「API のレート制限とエラーレスポンスが矛盾」 | 1 つの矛盾。API 設計者が対処 |

**悪い例（粒度が不適切）:**

| Gap | 問題 | 修正 |
|-----|------|------|
| 「認証まわり全般が弱い」 | 粒度が大きすぎる。何を直せばいいかわからない | 「パスワードポリシー未定義」「セッションタイムアウト未定義」「MFA の有無が未決定」に分割 |
| 「API の /users エンドポイントで page パラメータが 0 始まりか 1 始まりか未定義」かつ「同エンドポイントの sort パラメータのデフォルト値が未定義」 | 同じ根本原因（/users のクエリパラメータ仕様が未整備）なのに分割されている | 「/users エンドポイントのクエリパラメータ仕様が未定義」にまとめる |

## 3. Severity 判定基準

| Severity | 基準 | 判定の目安 | 例 |
|----------|------|-----------|-----|
| **Critical** | 未対処だとシステムが動かない、セキュリティ事故、法的問題が発生する | 「リリースできない」レベル | 認証バイパスが可能、個人情報の暗号化方針が未定義、利用規約なしでサービス公開 |
| **High** | 主要ユースケースに影響し、ユーザー体験を大きく損なう | 「リリースできるが、主要機能でユーザーが困る」レベル | 検索結果が 100 件超のときのページネーション未定義、エラー時のリカバリ手順なし |
| **Medium** | エッジケースに影響する、または改善すべきだが致命的ではない | 「大半のユーザーは困らないが、特定条件で問題が起きる」レベル | タイムゾーン跨ぎの日時表示が未定義、低速回線でのタイムアウト値が未設定 |
| **Low** | nice-to-have、将来の改善候補 | 「なくても運用できるが、あった方が良い」レベル | エラーメッセージの文言統一、管理画面のソート順のデフォルト値 |

### Severity 判定のフローチャート

```
この Gap を放置したまま本番運用したら?
  ├─ システムが停止 or セキュリティ事故 or 法的問題 → Critical
  ├─ 主要ユースケースでユーザーが目的を達成できない → High
  ├─ 特定条件でのみ問題が起きる → Medium
  └─ 問題は起きないが改善の余地がある → Low
```

### 迷ったときのルール

- Critical か High か迷ったら → **High** にする。Critical の乱発はトリアージの意味を失わせる。
- Medium か Low か迷ったら → **Medium** にする。Low にすると永久に対処されない。

## 4. 優先度フレームワーク

### Priority スコア

Severity だけでは対処順序を決められない。以下の 3 軸で Priority を算出する。

| 軸 | 定義 | スコア |
|----|------|--------|
| **Impact（影響度）** | 発生した場合の被害の大きさ | 3: 重大 / 2: 中程度 / 1: 軽微 |
| **Probability（発生可能性）** | 実際に発生する確率 | 3: 高い / 2: 中程度 / 1: 低い |
| **Detection Difficulty（検知困難度）** | 発生後に気づけるまでの難しさ | 3: 気づきにくい / 2: やや気づきにくい / 1: すぐ気づく |

```
Priority Score = Impact × Probability × Detection Difficulty
  最大: 27（最優先）
  最小: 1（後回し可）
```

### 2x2 マトリクス（簡易版）

フルスコアリングが重い場合は、Impact × Probability の 2 軸で判断する。

```
                    Impact 高          Impact 低
                ┌──────────────────┬──────────────────┐
  Probability   │                  │                  │
     高         │   即座に対処     │   計画的に対処   │
                │  (Score 9)       │  (Score 3)       │
                ├──────────────────┼──────────────────┤
  Probability   │                  │                  │
     低         │   次スプリント   │   バックログ     │
                │  (Score 3)       │  (Score 1)       │
                └──────────────────┴──────────────────┘
```

### 運用上のガイドライン

- **DGE session 直後**: Severity だけ付与する（軽量に回す）
- **Spec に落とすとき**: Priority スコアを算出して対処順序を決める
- **3 軸フルスコアが不要なケース**: Gap が 10 件以下なら 2x2 マトリクスで十分
- **3 軸フルスコアが必要なケース**: Gap が多数あり、リソース制約でトリアージが必要なとき

## 5. 偽陽性の検証メカニズム

DGE は LLM が生成する会話劇から Gap を抽出するため、偽陽性（実際には問題ではないものを Gap と判定してしまうこと）のリスクが常にある。

### Gap Validation Checklist

Gap を Discovered から Validated に移行する際、以下の 4 項目を確認する。

- [ ] **再現性**: 別の LLM、別のプロンプト、別のキャラクター構成でも同様の Gap が出るか?
  - 1 回の session でしか出現しない Gap は信頼度が低い
  - ただし、特定キャラの専門視点からしか見えない Gap は再現しなくても有効な場合がある
- [ ] **仕様カバー確認**: 仕様書で既にカバーされていないか?
  - 仕様書の該当箇所を引用して「カバー済み」と示せるなら偽陽性
  - 仕様書に関連記述はあるが不十分な場合は、Gap の内容を修正して残す
- [ ] **影響の具体性**: 実際にユーザーに影響するか? 影響するユーザーを具体的に指名できるか?
  - 「理論上は問題だが実運用では発生しない」は優先度を下げる（ただし除外はしない）
- [ ] **根拠の引用**: 会話劇のキャラクターのセリフから、この Gap の根拠となる箇所を引用できるか?
  - セリフから根拠を引用できない Gap は、抽出プロセスに問題がある

### 4 項目中の判定基準

| 合格数 | 判定 |
|--------|------|
| 4/4 | 高信頼。そのまま Validated へ |
| 3/4 | 有効。不合格の 1 項目を注記して Validated へ |
| 2/4 | 要再調査。追加の DGE session か人間レビューで検証 |
| 1/4 以下 | 偽陽性の可能性が高い。理由を記録して除外 |

### 偽陽性の典型パターン

| パターン | 症状 | 対策 |
|----------|------|------|
| **LLM の hallucination** | 存在しない仕様上の問題を指摘する。「〇〇 API は△△を返さない」と言うが、実際には返す | 仕様書の該当箇所を必ず確認する |
| **過度な一般化** | あらゆるシステムに言える一般論を Gap として挙げる。「パフォーマンスを考慮すべき」「セキュリティを強化すべき」 | 「この仕様のどこが具体的に足りないか」を問い返す |
| **ドメイン知識の欠如** | ドメイン固有の慣行を「未定義」と誤認する。業界標準で自明なことを Gap と指摘する | ドメインエキスパートにレビューを依頼する |
| **過去バージョンへの指摘** | 既に修正済みの問題を Gap として挙げる（LLM の学習データが古い） | 最新の仕様書を context として渡す |
| **対立の過剰演出** | キャラクター間の対立を作るために、本来問題ではない点を問題化する | 「この対立は仕様上の実問題に基づいているか」を確認する |

## 6. Gap の Lifecycle

```
Discovered → Validated → Prioritized → Addressed → Verified
```

### 各ステージの定義

| ステージ | 定義 | 成果物 |
|----------|------|--------|
| **Discovered** | DGE session の会話劇から抽出された状態。まだ検証されていない | Gap ID、Observe-Suggest-Act、Category、Severity（暫定） |
| **Validated** | Validation Checklist を通過し、偽陽性でないことが確認された状態 | Checklist の結果、信頼度の判定 |
| **Prioritized** | Priority スコアが算出され、対処順序が決まった状態 | Priority Score、対処時期の割り当て |
| **Addressed** | Gap に対する対策（仕様追加、設計変更等）が実施された状態 | 変更された仕様書・コード・ドキュメントへのリンク |
| **Verified** | 対策が Gap を実際に解消していることが確認された状態 | レビュー記録、テスト結果 |

### ステージ移行の条件

```
Discovered → Validated:
  条件: Validation Checklist の 4 項目中 3 項目以上が合格
  担当: DGE session の実施者、またはドメインエキスパート

Validated → Prioritized:
  条件: Severity 確定 + Priority Score 算出（または 2x2 マトリクスでの位置づけ）
  担当: プロダクトオーナーまたはテックリード

Prioritized → Addressed:
  条件: Gap に対する具体的な対策が実施されている
  担当: 各 Gap の担当者

Addressed → Verified:
  条件: 対策が Gap を解消していることを第三者が確認
  担当: 実施者以外のチームメンバー
```

### 運用上の注意

- **全ての Gap が Verified まで到達する必要はない。** Low の Gap はバックログに残したまま放置してもよい。
- **Lifecycle の厳密さは session のレベルに合わせる。** 探索的 session では Discovered + Severity だけで十分。厳密な session では全ステージを通す。
- **Gap が却下される場合も記録する。** 「なぜ偽陽性と判断したか」の記録は、次の session の入力品質を上げる。
