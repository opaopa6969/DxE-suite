# DGE の限界と制約

> 対象 Gaps: 1-003, 1-006, 1-010, 3-022, 3-023, 3-029
> 作成日: 2026-03-28

DGE は強力だが魔法ではない。本文書は DGE の限界を正直に記述する。

---

## 1. LLM がキャラクターを演じる限界

### 1.1 根本的な制約

LLM はキャラクターの「知識」を持っているわけではない。学習データから抽出したパターンを再構成しているにすぎない。今泉が「そもそも」と問うとき、それは今泉というキャラクターの思考プロセスではなく、LLM が「今泉ならこう言うだろう」と推測したパターンマッチの結果である。

この区別は重要だ。パターンマッチは「よくある問い」を生成するのは得意だが、「誰も思いつかなかった問い」を生成する能力には本質的な限界がある。

### 1.2 専門ドメインでの浅さ

DGE のキャラクターは汎用的な思考パターン（前提を問う、品質を求める、シンプルさを追求する等）を持つが、特定の専門ドメインでは議論が浅くなるリスクがある。

| ドメイン | リスク | 具体例 |
|----------|--------|--------|
| 医療 | 臨床判断の微妙さを再現できない | 「患者の同意フロー」を議論しても、実際の IRB 要件の詳細まで踏み込めない |
| 法律 | 管轄固有の法規制を正確に反映できない | ソウルが「利用規約を書け」と言っても、特定国の個人情報保護法の要件は表面的 |
| 金融規制 | 規制の最新動向を反映できない | 鷲津が「IRR は？」と問っても、最新の IFRS 基準との整合性は検証できない |
| 医療機器 | IEC 62304 等の規格準拠の詳細が不十分 | 千石が品質を求めても、規格の具体的な要求事項との突合は困難 |
| セキュリティ | 最新の攻撃手法や CVE を網羅できない | Red Team の攻撃シナリオが学習データ時点の知識に制約される |

### 1.3 各キャラクターの「得意な議論」と「限界」

| キャラクター | 得意な議論 | 限界 |
|-------------|-----------|------|
| 今泉 | 前提の検証、暗黙の仮定の発見 | 技術的深堀りができない。専門用語の裏にある実務知識がない |
| 千石 | 品質基準の提示、UX の問題指摘 | ドメイン固有の品質基準（医療のGxP等）を知らない。MVP との折り合いが苦手 |
| ヤン | 不要な複雑さの発見、本質の抽出 | 「やらなくていい」が多すぎる。規制要件のように「やらなければならない」ものへの感度が低い |
| 僕 | スコープの縮小、リスクの直感的検知 | 大胆な提案ができない。議論を前に進める力が弱い |
| ラインハルト | ビジョンの明確化、大胆な方向転換 | 細部の検証が苦手。「攻めろ」だけでは実装に落ちない |
| 大和田 | ビジネスモデルの現実検証、収益の追求 | 技術的制約を理解しない。長期投資の価値を認めにくい |
| 鷲津 | 財務的な厳密さ、投資判断 | 具体的な業界の財務慣行に弱い。数字の裏付けが LLM の推測に依存 |
| リヴァイ | 実装品質の追求、over-engineering の排除 | 設計フェーズの価値を認めにくい。「動くもの」偏重 |
| 利根川 | ユーザーの支払い意思の検証 | 絶望的すぎて建設的な議論に発展しにくい |
| ハウス | 隠れた問題の発見、偽の安心感の破壊 | 「全員嘘をついている」が行きすぎると、正しい前提まで疑う |
| Red Team | 攻撃シナリオの提示、脆弱性の発見 | 学習データ時点の攻撃手法に制約。ゼロデイ的な新規攻撃は生成できない |
| ソウル | 法的リスクの指摘、利用規約の必要性 | 具体的な管轄の法律知識は表面的。法的助言の代替にはならない |

### 1.4 対策

1. **ドメイン情報のコンテキスト注入**: 先輩（ナレーター）の context に、ドメイン固有の規制要件・業界標準・制約条件を具体的に記述する。LLM が知らない情報は、人間が与える。
2. **専門家レビューの併用**: DGE の出力を専門家がレビューする。DGE は「問いの種」であり、その問いが的を射ているかの判断は専門家が行う。
3. **限界の明示**: session 開始時に「このドメインで DGE が弱い領域」を予め宣言し、その領域は別の手法（専門家インタビュー、規制チェックリスト等）で補完する。
4. **ドメイン固有キャラクターの作成**: catalog.md のカスタムキャラクター作成ガイドに従い、ドメイン専門家の思考パターンを反映したキャラクターを設計する。ただし、キャラクターの知識はコンテキストに注入した情報の範囲に限られることを認識する。

## 2. 合意バイアスと矛盾の弱体化

### 2.1 問題の構造

LLM は RLHF（Reinforcement Learning from Human Feedback）によって「調和的な出力」を好むように訓練されている。これは通常の会話では適切だが、DGE においては致命的な副作用を持つ。

DGE で最も価値があるのは**キャラクター間の矛盾**である（quality-criteria.md の「矛盾の存在」基準を参照）。しかし LLM の合意バイアスにより、この矛盾が自然に弱められる。

具体的な症状:

- 千石が品質問題を指摘した後、ヤンが「まあ、千石さんの言うことも分かりますが...」と同調してしまう
- Red Team が攻撃シナリオを提示した後、全員が「確かにそのリスクはありますね」と合意して終わる
- 今泉が「そもそも」と問うた後、他のキャラが答えを出して議論が収束してしまう

quality-criteria.md で定義した「全員合意パターン」アンチパターンが、LLM の本質的な傾向として発生しやすい。

### 2.2 なぜ矛盾が失われると問題なのか

矛盾は、設計における未解決のトレードオフを可視化する。矛盾が弱められると:

- 「品質 vs スピード」のトレードオフが曖昧なまま放置される
- 「セキュリティ vs UX」の緊張関係が表面化しない
- 「理想 vs 現実」の gap が「まあ、いい感じにやりましょう」で埋められる

これらは DGE が発見すべき最も重要な gap であり、合意バイアスによって最初に失われるものでもある。

### 2.3 対策

#### A. プロンプトでの明示的な対立要求

```
各キャラクターは自分の立場を最後まで譲らないでください。
妥協点を探す前に、なぜ自分の立場が正しいかを 2 回以上主張してください。
```

#### B. 強制的な反論制約

```
制約: キャラXはキャラYの意見に対して必ず反論しなければなりません。
反論の後に「ただし」で同調することは禁止します。
```

効果的なペア:
- 千石 vs ヤン（品質の絶対性 vs 簡潔さの追求）
- ラインハルト vs 僕（大胆な攻め vs 慎重な縮小）
- 鷲津 vs ヤン（数字の厳密さ vs 面倒なことの回避）
- リヴァイ vs 千石（実装の現実 vs 理想の品質）

#### C. 2 段階生成

1 つのプロンプトで全キャラを同時に生成すると、LLM の内部で矛盾を「解消」しようとする力が働く。これを防ぐために:

1. **まず 1 キャラの発言を生成する**（例: 千石が品質問題を指摘）
2. **その出力を入力として、別のキャラに反論させる**（例: ヤンが「それ本当に要る？」と返す）

2 回の生成に分けることで、各キャラの立場が独立に強化される。

#### D. temperature の調整

temperature を上げる（0.8-1.0）ことで、出力のバリエーションが増え、予定調和的な合意に陥りにくくなる。ただし、高すぎると発言の coherence が下がるため、0.7-0.9 の範囲を推奨する。

#### E. 合意バイアスの検出

session 後のセルフチェックで以下を確認する:

- [ ] キャラクター間で明確な意見の対立が 2 箇所以上あるか
- [ ] 対立が「確かにそうですね」で 1 往復で解消されていないか
- [ ] 最終的に未解決のまま残っている矛盾があるか（これが最も価値ある gap）

## 3. キャラクター間の gap 重複

### 3.1 現状

キャラクター間の gap 発見の重複率・ユニーク率は計測されていない（Gap 1-010）。これは DGE のキャラクター設計の効率性を評価するうえで重要な指標である。

### 3.2 仮説

同じアーキタイプのキャラクターは重複しやすい:

| 重複しやすいペア | 理由 |
|-----------------|------|
| 今泉 と ハウス | 両者とも「隠れた前提」を攻撃する。今泉は素朴な質問で、ハウスは診断的に |
| 千石 と リヴァイ | 両者とも品質を要求する。千石はユーザー視点で、リヴァイは実装視点で |
| 鷲津 と 大和田 | 両者ともビジネスの現実を突きつける。鷲津は投資家視点、大和田は経営者視点 |
| ヤン と 僕 | 両者ともスコープ縮小の方向。ヤンは合理性で、僕は不安で |

一方、重複しにくいペア:

| ユニークな組み合わせ | 理由 |
|---------------------|------|
| 今泉 と 鷲津 | 前提検証 vs 数字の追求。異なる角度 |
| Red Team と ソウル | 攻撃 vs 法的防御。攻守の違い |
| ラインハルト と 僕 | 拡大 vs 縮小。正反対のベクトル |

### 3.3 計測方法の提案

既存の session データ（unlaxer-parser: 108 gaps, AskOS: 85 gaps, Meta: 71 gaps）を使って計測する:

```
手順:
  1. 各 gap を「発見に寄与したキャラクター」に帰属させる
  2. 同じ gap を複数キャラが指摘した場合は「重複」とマークする
  3. 以下の指標を算出:
     - 重複率 = 重複 gap 数 / 全 gap 数
     - キャラ別ユニーク率 = そのキャラだけが発見した gap 数 / そのキャラの全 gap 数
     - ペア別重複率 = 特定ペアが共通して発見した gap 数 / 両者の gap 合計
```

### 3.4 重複に対する正しい態度

**重複自体は悪くない。**

同じ gap を複数のキャラクターが異なる角度から指摘する場合、それは gap の重要性の裏付けになる。千石が「品質として許せない」と言い、今泉が「そもそもそれで大丈夫なの？」と言い、ハウスが「その問題を隠してるだろ」と言う――3 人が同じ gap を指すなら、それは確実に対処すべき gap である。

重要なのは**ユニーク率**:

- ユニーク率が高い → そのキャラクターは他にない視点を提供している。価値がある。
- ユニーク率が低い → そのキャラクターは他と代替可能。キャラ選択を見直す余地がある。

目標値（仮説）:
- 3 キャラ構成でのユニーク率: 各キャラ 30% 以上
- 5 キャラ構成でのユニーク率: 各キャラ 20% 以上
- 全体重複率: 40% 以下（60% 以上の gap がユニーク）

## 4. 他手法との関係

### 4.1 比較マトリクス

DGE は既存の設計レビュー手法と競合するのではなく、カバー領域が異なる。

| 手法 | 対象 | DGE との関係 |
|------|------|-------------|
| **チェックリスト** | 「書いてあるべきこと」の確認 | 相補的。チェックリストは既知の項目を網羅。DGE は「書いてないこと」を発見 |
| **STRIDE** | セキュリティ脅威の体系的分析 | 領域特化 vs 汎用。STRIDE はセキュリティに深い。DGE は広い |
| **ADR** | アーキテクチャ決定の記録 | 時間軸が異なる。ADR は決定後の記録。DGE は決定前の探索 |
| **Threat Modeling** | 脅威の特定と対策 | 相補的。DGE でシナリオ発見 → Threat Modeling で深掘り |
| **形式手法** | 仕様の数学的検証 | 厳密性が異なる。形式手法は証明。DGE は発見 |
| **ペアレビュー** | コードの品質確認 | 粒度が異なる。ペアレビューは実装レベル。DGE は設計レベル |
| **ブレインストーミング** | アイデアの自由発想 | 構造が異なる。DGE はキャラクターの制約で方向を持つ |
| **FMEA** | 故障モードの体系的分析 | 分析手法 vs 発見手法。FMEA は既知のモードを評価。DGE は未知のモードを発見 |

### 4.2 詳細比較

#### DGE vs チェックリスト

```
チェックリスト:
  - 「認証は実装したか？」 → Yes/No
  - 「エラーハンドリングはあるか？」 → Yes/No
  - 既知の項目を漏れなく確認する

DGE:
  - 「認証が通った後、セッションが切れたらどうなる？」
  - 「エラーが連鎖したとき、ユーザーには何が見える？」
  - 書いてない状況を発見する

gap の重複:
  チェックリストで発見できる問題を DGE も発見することがある。
  これは重複であり、DGE 固有の価値ではない。
  DGE の価値はチェックリストに「ない」項目を発見すること。
```

#### DGE vs STRIDE

```
STRIDE:
  - Spoofing, Tampering, Repudiation, Information Disclosure,
    Denial of Service, Elevation of Privilege
  - セキュリティ脅威を体系的に分類・分析する
  - 深い。1 つの脅威を掘り下げる

DGE:
  - セキュリティに限らず、ビジネス・UX・運用・法的リスクも扱う
  - 広い。多角的な視点で gap を発見する
  - セキュリティの深さは STRIDE に劣る

推奨: セキュリティが重要なプロジェクトでは STRIDE + DGE の併用。
  DGE（Red Team + ハウス構成）でセキュリティ gap を発見 →
  STRIDE で各 gap を体系的に深掘り。
```

#### DGE vs ADR (Architecture Decision Records)

```
ADR:
  - 「なぜその決定をしたか」を記録する
  - 決定の背景、選択肢、トレードオフを文書化する
  - 時間軸: 決定後

DGE:
  - 「その決定の前に、何を見落としているか」を発見する
  - 選択肢の漏れ、考慮されていないトレードオフを洗い出す
  - 時間軸: 決定前

推奨: DGE → 決定 → ADR のフロー。
  DGE で選択肢と gap を洗い出す →
  人間が決定する →
  ADR に決定の記録と DGE で発見された gap への対応を記述する。
```

#### DGE vs Threat Modeling

```
Threat Modeling:
  - アーキテクチャ図から脅威を体系的に特定する
  - 攻撃者のモチベーション、能力、攻撃経路を分析する
  - 構造的・網羅的

DGE:
  - 会話劇でシナリオベースの脅威を発見する
  - 「こういう場面でこう使われたら」の具体的な状況を描写する
  - 発見的・探索的

推奨: 相補的に使う。
  DGE で「予想外のシナリオ」を発見 →
  Threat Modeling でそのシナリオを体系的に分析 →
  対策を設計する。
```

### 4.3 併用推奨マトリクス

| プロジェクトの性質 | DGE | チェックリスト | STRIDE | ADR | Threat Modeling | FMEA |
|-------------------|-----|-------------|--------|-----|----------------|------|
| 新規 API 設計 | **必須** | 推奨 | - | 推奨 | - | - |
| セキュリティ重要システム | **必須** | 推奨 | **必須** | 推奨 | **必須** | - |
| 医療・安全重要システム | **必須** | **必須** | 推奨 | **必須** | 推奨 | **必須** |
| 新機能企画（初期） | **必須** | - | - | - | - | - |
| アーキテクチャ変更 | **必須** | - | - | **必須** | 推奨 | - |
| SaaS 事業検討 | **必須** | - | - | - | - | - |
| 障害振り返り | 推奨 | **必須** | - | - | - | 推奨 |

DGE は「初期の探索」に最も効果を発揮する。発見した gap を他の手法で深掘りするのが最も効率的な使い方である。

## 5. データセキュリティガイドライン

### 5.1 DGE session に含まれる情報の種類

DGE session の入力と出力には、以下の種類の情報が含まれる:

| 情報の種類 | 入力での例 | 出力での例 |
|-----------|-----------|-----------|
| 設計ドキュメント | API 仕様書、アーキテクチャ図 | gap として「API の認証フローに穴がある」等 |
| ビジネス戦略 | 事業計画、競合分析 | gap として「pricing が検証されていない」等 |
| 技術詳細 | データベーススキーマ、インフラ構成 | gap として「SPOF がある」等 |
| ユーザー情報 | ペルソナ、利用データ | gap として「プライバシーリスクがある」等 |
| 財務情報 | 収益予測、コスト構造 | gap として「unit economics が成立しない」等 |
| セキュリティ情報 | 脅威モデル、脆弱性 | gap として「この攻撃ベクトルが未対策」等 |

重要な認識: **DGE の出力自体が機密情報を含む。** gap の一覧は「このシステムの弱点リスト」であり、攻撃者にとって最も価値のある情報になりうる。

### 5.2 LLM に送信されるデータの扱い

#### クラウド LLM（Claude, GPT-4, Gemini 等）の場合

```
リスク:
  - 入力データが LLM プロバイダーのサーバーに送信される
  - プロバイダーの利用規約に従ったデータ取扱い
  - 学習データへの利用可能性（プロバイダーの方針に依存）

確認事項:
  - プロバイダーの API 利用規約を確認する
  - 「API 経由の入力は学習に使用しない」を保証しているか
  - データの保持期間と削除ポリシーは何か
  - SOC 2, ISO 27001 等のセキュリティ認証を取得しているか

推奨:
  - API 経由で利用する（Web UI 経由は利用規約が異なる場合がある）
  - Enterprise プランが利用可能な場合は Enterprise プランを使う
  - 機密度の高い情報は抽象化してから送信する（後述）
```

#### ローカル LLM（Llama, Mistral 等）の場合

```
メリット:
  - データが外部に送信されない
  - 完全に自社管理下でのデータ処理

デメリット:
  - モデルの能力がクラウド LLM より低い場合がある
  - キャラクターの演じ分けや日本語品質が劣る可能性
  - 運用・メンテナンスのコスト

推奨:
  - 機密度が非常に高いプロジェクト（軍事、医療個人情報等）ではローカル LLM を使う
  - quality-criteria.md のセルフチェックリストで出力品質を検証する
  - 品質が不十分な場合、DGE の session 設計を簡素化する（キャラ数を減らす等）
```

### 5.3 session 成果物の公開範囲管理

DGE session の成果物（会話劇、gap リスト、spec）に対して、以下の 3 段階の公開範囲を定義する。

#### Level 1: Internal only（社内プロジェクト）

```
対象: 商用プロジェクト、社内システム、顧客案件
ルール:
  - session 成果物は社内リポジトリ/ドキュメント管理システムに保存
  - アクセス権はプロジェクトメンバーに限定
  - 外部への共有は禁止
  - gap リスト（= 弱点リスト）は特に厳重に管理
注意:
  - クラウド LLM に送信した時点で、プロバイダーの管理下にもデータが存在する
  - Internal only はあくまで「成果物の共有範囲」の制御
```

#### Level 2: Redacted（論文やブログ用に匿名化）

```
対象: 事例共有、学会発表、ブログ記事
ルール:
  - プロジェクト名、企業名、製品名を匿名化する
  - 具体的な数値（売上、ユーザー数等）を丸める or 削除する
  - セキュリティ gap の具体的な攻撃手法を一般化する
  - キャラクターの発言から固有名詞を除去する
匿名化チェックリスト:
  - [ ] プロジェクト名が含まれていないか
  - [ ] 企業名・製品名が含まれていないか
  - [ ] 具体的な金額・ユーザー数が含まれていないか
  - [ ] API endpoint やデータベーススキーマの実名が含まれていないか
  - [ ] セキュリティ脆弱性の具体的な悪用方法が含まれていないか
  - [ ] 個人名（開発者名、顧客名等）が含まれていないか
```

#### Level 3: Public（OSS プロジェクト）

```
対象: オープンソースプロジェクト、公開教材
ルール:
  - session 成果物をそのまま公開可能
  - ただし、以下の情報は公開前に確認:
    - コントリビューターの個人情報
    - まだ公開されていない脆弱性情報
    - 依存ライブラリのライセンス情報
```

### 5.4 機密プロジェクトでの推奨プラクティス

#### A. ローカル LLM の使用

機密度が高いプロジェクトでは、データが外部に送信されないローカル LLM の使用を推奨する。能力のトレードオフは section 5.2 を参照。

#### B. 具体的な数値・名称の抽象化

LLM に送信する前に、機密情報を抽象化する:

```
抽象化前:
  「ProjectX の月間売上 3,200 万円のうち、Enterprise プランが 70% を占める。
  主要顧客は A 社、B 社、C 社。API のレスポンスタイムは p99 で 230ms。」

抽象化後:
  「対象プロジェクトの月間売上は数千万円規模で、Enterprise プランが大半を占める。
  主要顧客は数社。API のレスポンスタイムは p99 で 200-300ms。」

DGE の議論に影響しない程度に抽象化する。
gap の発見には具体的な数値より「構造」が重要。
```

#### C. session 成果物へのアクセス制御

```
推奨:
  - gap リストへのアクセスを最小権限の原則で管理する
  - 特にセキュリティ gap は、対策完了まで閲覧者を制限する
  - session ログ（会話劇の全文）は、必要なメンバーのみアクセス可能にする

実装例（Git ベースの場合）:
  - .gitignore で session ログを除外する
  - 別の private リポジトリに session ログを保存する
  - gap リストのみをメインリポジトリに含め、詳細は参照リンクにする
```

#### D. プロジェクト終了後の成果物の扱い

```
検討すべき事項:
  - session ログの保持期間を定める（例: プロジェクト終了後 1 年）
  - 保持期間経過後の削除手順を定める
  - ただし、DGE の方法論改善に必要な統計データ（gap 数、カテゴリ分布等）は
    匿名化して保持する
  - クラウド LLM プロバイダー側のデータ削除も確認する（API 利用規約に従う）
```

### 5.5 免責事項

DGE は設計レビューを支援する手法であり、以下の代替にはならない:

- **セキュリティ監査**: DGE で発見されたセキュリティ gap は「候補」であり、専門のペネトレーションテストや脆弱性診断を代替しない
- **法的助言**: ソウルの法的リスク指摘は思考の出発点であり、弁護士の助言を代替しない
- **コンプライアンス認証**: DGE session の実施は、ISO 27001 や SOC 2 等の認証プロセスの一部にはならない
- **完全な gap 発見**: DGE は gap の発見を保証しない。「DGE を実施したから安全」という誤解を避けること（Gap 3-001, 3-002 参照）

---

## まとめ: DGE を正しく使うために

DGE は「設計の穴を対話で見つける」強力な手法だが、以下の限界を常に意識すべきである:

1. **LLM はパターンマッチであり、専門家ではない** — ドメイン知識は人間が注入する
2. **合意バイアスが最も価値ある矛盾を消す** — 意図的に対立を設計する
3. **キャラクター間の重複は計測し、最適化する** — ユニーク率を意識したキャラ選択
4. **DGE は万能ではない** — 他手法との併用で効果を最大化する
5. **DGE の出力自体が機密情報** — データセキュリティを設計段階から考慮する

これらの限界を理解したうえで DGE を使えば、「書いてないこと」の発見において他に代えがたい価値を発揮する。限界を知ることが、正しく使うための第一歩である。
