# Dialogue Grounding Engineering (DGE) — Revised Edition

## LLMに「会話劇」を書かせたら、仕様書の質が段違いに上がった話

---

### Abstract

ソフトウェア開発において、要件定義（spec）の品質はプロジェクトの成否を決定する。従来のウォーターフォール的アプローチでは「要件→設計→実装」のトップダウンが主流だが、LLM時代の開発ではこの順序が逆転しうる。本稿では、動作するプロトタイプを起点にLLMへ「会話劇（dialogue script）」を生成させ、そこからusecase・spec・architectureを逆算的に抽出する手法——**Dialogue Grounding Engineering（DGE）**——を提案する。

本手法は実プロジェクト（AskOS: AIエージェントオーケストレーションOS）で検証済みであり、v2→v7の5ラウンドのspecレビューで見つからなかった108のuse caseと97のspec implicationsを、会話劇10分で発見した。さらに、DGEワークフロー自体にDGEを適用する再帰的検証（DGE-on-DGE）を行い、13本の会話劇から32のワークフロー改善点を発見した。

**Keywords:** Requirements Engineering, LLM, Grounding, Dialogue Simulation, Reverse Specification

---

## 1. はじめに——仕様書が先か、動くものが先か

ソフトウェアエンジニアなら誰でも経験がある。完璧に見えた仕様書でも、実装が進むと「あれ、この画面遷移おかしくない？」「このエラーケース考えてなかった」という抜け漏れが見つかる。

従来の開発フローはこうだ：

```
要件定義 → usecase → spec → architecture → 実装 → テスト
```

上流から下流に水が流れるように、抽象から具体へ。しかし、この方向には構造的な弱点がある——**抽象的な段階で書かれたドキュメントは、具体的な使用状況を十分にシミュレーションしきれない**。

LLMの登場により、プロトタイプの構築コストは劇的に下がった。数時間で動くPOCを作れる時代に、なぜ我々はまだ白紙の状態から仕様書を書いているのか？

本稿が提案する **Dialogue Grounding Engineering（DGE）** は、このプロセスをひっくり返す。

```
POC（動くもの）→ 会話劇の生成 → usecase / spec / architecture の逆算
```

---

## 2. 実証: specレビューを何回しても見つからなかったことが、会話劇10分で出てきた

理論の前に、実際に起きたことを話す。

### 2.1 従来型specレビューの5ラウンド

AskOSというプロジェクトで、v2→v7まで5ラウンドのspecレビューを行った。LLMにspec文書を読ませて不整合を指摘させるアプローチだ。

```
v2→v3: agent.project_idのnullable問題を発見
v3→v4: InboxMessage transportの確認応答モデル追加
v4→v5: Commander triegeのタイムアウトフォールバック修正
v5→v6: portfolio.statusにpause追加
v6→v7: runtime-adapterのエラー回復手順補完
```

これらはすべて「既に書かれている仕様の不整合」だ。型の不一致、未定義のフォールバック、テーブル設計の漏れ。specレビューは仕様書の**中**を磨くのは得意だ。

### 2.2 会話劇アプローチに切り替えた

次に、同じプロジェクトに対してDGEを適用した。設定はこうだ：

> Operatorは10日間九州ツーリングに行っていた。スマホの電源も切っていた。プロジェクトのことは完全に忘れている。Docker volume pruneでデータが消えた。ゼロからの再スタート。

この状況で「AskOSを初めて開く」会話劇を書かせた。

```
僕: （ブラウザでlocalhost:3000を開く）
AskOS: Overview画面。Projects: 0, Questions: 0, Agents: 0
僕: ……何すればいいんだ？
    Create ボタン？ いや、agentに聞くか。
僕: このソフト、プロジェクト管理できるんだよね？
Agent: まず今やっているプロジェクトを全部出して。
僕: GitHubのURL全部あるよ。
Agent: 全部cloneして中身を読みます。
    ……unlaxerの中にサブプロジェクトが3つありますね。
    これは1つのPortfolioにまとめますか？
```

**10分で、以下のgapが見つかった：**

1. **Entry pointがformではなくconversation** — specの大前提が違った
2. **1 git repo ≠ 1 project** — モノレポ対応が未定義だった
3. **プロジェクト依存関係管理** — specに概念すら存在しなかった
4. **空状態のUX** — specではCRUDフォーム前提だった
5. **GitHub API統合** — clone以外のアプローチが未検討だった

### 2.3 最終結果

22本の会話劇から **108のuse case** と **97のspec implications** が発見された。5ラウンドのspecレビューでは1つも見つからなかった種類のgapだ。

| 手法 | ラウンド数 | 発見されたgapの種類 | 発見数 |
|------|-----------|-------------------|--------|
| specレビュー | 5ラウンド | 型不整合、未定義フォールバック、テーブル漏れ | 約20件 |
| DGE（会話劇） | 22ストーリー | 前提の転換、未知の概念、UXモデルの刷新 | 108 UC + 97 SI |

**specレビューは仕様書の「中」を磨く。会話劇は仕様書の「外」を見つける。**

## 3. DGEとは何か

### 3.1 手法の概要

DGEは3つのフェーズから構成される。

**Phase 1: 具体化（Instantiation）**
まず、ある程度動作するプロトタイプ（POC）を構築する。LLMの支援により、このフェーズは数時間〜数日で完了する。さらに重要なのは、**プロダクトのソースコード一式をLLMのコンテキストに投入する**こと。2025年時点で1Mトークン級のコンテキストが利用可能になり、中〜大規模プロジェクトの丸ごと入力が現実的になった。

**Phase 2: 会話劇の生成（Dialogue Script Generation）**
LLMに対し、ソースコード全体を理解させた上で、具体的なユーザーペルソナ同士の会話劇を生成させる。この「全体理解 → 特定機能フォーカス」の二段階が、LLMのattentionを適切に集中させ、幻覚の抑制と発見精度の向上を両立させる鍵となる。

**Phase 3: 逆算（Reverse Extraction）**
会話劇を素材として、LLMにusecase一覧、spec文書、architecture設計を抽出・生成させる。

### 3.2 ネーミングの由来

**Grounding（接地）** という語は意図的に選択している。認知科学において grounding とは「抽象的な記号を具体的な経験や知覚に結びつけること」を指す（Harnad, 1990）。DGEは、ソフトウェアの仕様という抽象的記号体系を、会話劇という具体的な使用場面に接地させることで、その品質を担保する手法である。

## 4. なぜDGEは効くのか——認知的メカニズム

### 4.1 人間とLLMに共通する「具体に降りる力」

熟練プログラマが配列のソートアルゴリズムで行き詰まった時にやることは決まっている——紙に `[3, 1, 4, 1, 5]` と書いて、ポインタの動きを手でなぞる。ParserCombinatorのような抽象度の高いコードを書いている時でも、行き詰まったら `"Hello, {name}!"` のような具体文字列を置いて、パースの過程を追う。

UI/UXも同じ構造だ。「ユーザビリティ要件」という抽象概念を直接記述するより、「田中さんがこのボタンを押して、次の画面で戻るボタンが見つからなくて困る」という具体的なシーンを想像した方が、問題が見える。

LLMにも同じ傾向がある。Transformerのself-attention機構は、入力されたトークン列の中で「離れた位置にある関連トークン同士を結びつける」仕組みである。そして**LLMの学習データにおいて「会話・対話」形式のテキストは圧倒的に多い**。技術仕様書のような形式的な文書よりも、対話形式のテキストの上でLLMのattentionはより高い精度で動作する。

### 4.2 対話形式が「不整合発見マシン」として機能する理由

会話のやり取りという形式が自然に「Aが起きたらBはどうなる？」「Bが失敗したら？」というエッジケース探索を強制する。箇条書きのusecase一覧からは見つけにくい問題が、会話劇の中では**対話の断絶や不自然さ**として顕在化する。

AskOSの実証で最も典型的だったのは「沈黙」パターンだ。ユーザーが当然あると思った機能にシステムが応答できない場面——この「沈黙」がspec上の欠落を可視化した。

## 5. DGE-on-DGE: 再帰的検証による手法の精緻化

### 5.1 DGEワークフロー自体にDGEを適用した

DGEをワークフローエンジンの定義として形式化した後、そのワークフロー定義自体に対してDGEを適用した。13本の会話劇を書き、32のgapを発見した。

これは「DGEの原理の再帰的証明」である——DGEのspecを直接レビューしても見つからなかったことが、会話劇で出てきた。

### 5.2 発見された主要なgap

**Hallucination問題（Story 11）:**
会話劇を生成するLLM自身が、未実装の「パスワードリセット機能」を自然に動作させてしまった。contract schema で構造的正確性は保証できるが、意味的正確性は保証できない。

→ **解決: `validate-dialogue` ステップの追加。** 会話劇内でシステムが応答した全機能を抽出し、verify-pocの実装済み機能リストと突合するcross-validationを導入した。

**アプリケーション種別問題（Story 7, 8, 9）:**
ParserCombinatorの会話劇を書こうとした時に「ユーザーとUIの対話」が不自然すぎて破綻した。CLI、ライブラリ、インフラ、それぞれ「会話」の意味が違う。

→ **解決: `app_type` 入力パラメータの追加。** Web/CLI/Library/Infrastructure/APIの5種別でプロンプトテンプレートを切り替え。

**収束判定問題（Story 12）:**
ECサイトの例で3回ループしてもgapが出続ける状況が発生。止め時がわからない。

→ **解決: 収束メトリクスの導入。** 各イテレーションの新規gap率を追跡し、閾値以下で「収束推奨」を通知。

### 5.3 v1 → v2 の主要な変更

| 変更 | 理由 | Source Story |
|------|------|-------------|
| validate-dialogue ステップ追加 | hallucination cross-check | Story 11 |
| app_type 入力追加 | Web/CLI/Library/Infraで会話劇スタイルを切り替え | Story 7, 8, 9 |
| 収束メトリクス追加 | 反復の止め時を可視化 | Story 12 |
| verify-poc skippable化 | シニアはプロジェクト理解済み | Story 3 |
| audience_level 追加 | ジュニア/PM向けガイダンス調整 | Story 4, 5 |
| scope_path 追加 | 巨大プロジェクトのスキャン制御 | Story 10 |
| scan_coverage 出力追加 | verify-poc の網羅率を可視化 | Story 10 |

## 6. 会話劇パターン分類（Dialogue Pattern Taxonomy）

DGE-on-DGEの過程で、会話劇には再利用可能なパターンがあることが判明した。

### 6.1 対比パターン（Comparison Patterns）

| Pattern | 説明 | 用途 |
|---------|------|------|
| **Before/After** | 同じプロジェクトで導入前/後を比較 | 手法の効果実証 |
| **Role Contrast** | 異なるユーザーロールで同じ操作を比較 | 権限・UX差異の発見 |
| **App-Type Variation** | 異なるアプリ種別で同じ手法を適用 | 手法の汎用性検証 |

### 6.2 探索パターン（Discovery Patterns）

| Pattern | 説明 | 用途 |
|---------|------|------|
| **Zero-State** | 空の状態から使い始める | 初期体験・オンボーディングの発見 |
| **Return-After-Absence** | 長期不在後に戻ってくる | 状態復帰・サマリーの発見 |
| **Escalation Chain** | 問題が段階的にエスカレートする | エラーハンドリングの発見 |
| **Cross-Persona Conflict** | 異なるペルソナが矛盾する期待を持つ | 権限・優先度の衝突発見 |

### 6.3 限界探索パターン（Stress Patterns）

| Pattern | 説明 | 用途 |
|---------|------|------|
| **Scale Break** | 大量データ/ユーザーで使う | パフォーマンス・スケーラビリティの発見 |
| **Hallucination Probe** | LLMが勝手に補完する場面を探す | spec正確性の検証 |
| **Convergence Test** | 反復してもspecが安定しない | 終了条件・収束判定の発見 |
| **Drift Detection** | 実装後にspecとの乖離を探す | メンテナンス・鮮度管理の発見 |

## 7. 既存手法との比較

| 手法 | 方向 | 生成主体 | 入力 | 目的 |
|------|------|----------|------|------|
| SBRE | spec → scenario | 人間 | 要件文書 | 要件の検証 |
| LLM-RE | prompt → spec | LLM | 抽象的な指示 | 要件の生成 |
| DiscoverLLM | dialogue → intent | LLM | ユーザーの曖昧な意図 | 対話システム改善 |
| **DGE** | **POC → dialogue → spec** | **LLM + 人間** | **動作するプロトタイプ** | **仕様の発見と精緻化** |

## 8. 実践ガイド

### 8.1 最小構成（今すぐ始められる）

1. 動くPOC（30-50%完成度）を用意する
2. LLMにプロダクトのソースコード一式を読み込ませる
3. 特定機能について会話劇を生成させる
4. 会話劇から不整合・抜け漏れを抽出する
5. usecase → spec → architecture を逆算する

### 8.2 会話劇パターンの選択

プロジェクトの状況に応じて、§6のパターンを組み合わせる：

- **新規プロジェクト**: Zero-State + Role Contrast
- **既存プロジェクトの拡張**: Before/After + Escalation Chain
- **リリース前の最終チェック**: Scale Break + Cross-Persona Conflict
- **手法の導入提案**: Before/After + App-Type Variation

### 8.3 反復と収束

```
POC v1 → 会話劇 v1 → spec v1 → POC v2 → 会話劇 v2 → spec v2 → ...
```

通常2-3回の反復で、新規gap率が30%以下に収束する。

## 9. 結論

Dialogue Grounding Engineering（DGE）は、「抽象で行き詰まったら具体に降りる」という人間の自然な認知戦略を、LLMを活用してソフトウェア開発プロセスに体系的に組み込んだ手法である。

実プロジェクトでの検証結果：
- specレビュー5ラウンドで見つからなかった108 use caseを会話劇で発見
- DGE自体にDGEを適用し、13本の会話劇から32のワークフロー改善点を発見
- 手法の再帰的適用が可能であること自体が、手法の有効性の証明

ソフトウェアエンジニアは昔から「紙に具体例を書いてデバッグする」ことをやってきた。DGEは、その直感を開発プロセス全体に拡張し、LLMという強力な会話劇生成エンジンで加速する手法にすぎない。

しかし、この「にすぎない」が、仕様品質の段違いの向上をもたらす。
