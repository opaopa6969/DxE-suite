# DGE で DGE を改善した話 ─ ツール自身に査読させたら v2 から v3 になった

> メソッドが自分自身を発見できるかどうかは、そのメソッドの本物の試金石だ。

## TL;DR

- DGE toolkit v2 を DGE toolkit 自身でレビューした
- 3 セッションで 33 件の Gap を発見
- Critical 3 件 / High 19 件 のほぼ全てが v3.0〜v3.3 に反映された
- ツールが自己改善できるなら、あなたのプロダクトにも使える

```bash
npm install @unlaxer/dge-toolkit
npx dge-install
```

---

## なぜツール自身をレビューしたのか

DGE toolkit v2.4.1 を作ったとき、手元の実績はあった。unlaxer-parser の査読劇で 108 件の Gap を発見し、volta-auth-proxy で 106 件を解決した。でも「このツール自体は本当に良いのか？」という問いには答えていなかった。

普通なら「使ってみれば分かる」で済ます。でもそれは素の AI と同じ ─ 書かれたものをチェックするだけ。DGE は「書かれていないこと」を見つけるツールだ。なら、書かれていないことが DGE の設計にあるかどうかも見つけられるはずだ。

**セルフ査読を 3 セッション実施した。**

---

## Session 1: UX 査読 ─ 初めてのユーザーを想像する

**キャラ**: 今泉（前提を問う）+ 深澤（UX 直感）+ 僕（不安）+ 金八（理解支援）

### 何が起きたか

**今泉**: 「この人は npm のプロジェクトを持ってるんですか？ Claude Code は入ってますか？ README に書いてありましたっけ？」

→ Gap: 前提条件（package.json の有無、Claude Code のインストール済み）が明示されていない

**深澤**: 「初めて見た人は `characters/` `templates/` `flows/` `specs/` `sessions/` って知らないフォルダがどっさり現れる。怖いよ。『これ何？ 消していいの？』って絶対思う。」

→ Gap: インストール直後に「何が入ったか」「gitignore すべきか」のガイダンスがない

**金八**: 「最初の 1 歩は『DGE して』と言うだけでいいんだ。でも、そのことがインストール完了後のメッセージに書いてあるかい？ **体験が先、理解が後。**」

→ Gap: サンプル設計ドキュメントがない。体験ファーストの導線がない

**深澤（再）**: 「なぜ locale 自動判定しないの？ ユーザーに `--lang en` を付けてねって頼むのは、設計者の怠慢をユーザーに押しつけてる。」

→ Gap: locale 自動判定が未実装

### 直後に実装したもの

- `dge/samples/auth-api.md` をインストール時に自動コピー（サンプル設計ドキュメント）
- install.sh に `$LANG` 自動判定を追加（C/POSIX/en* → --lang en）
- postinstall メッセージを「DGE して」と言えば起動に短縮

---

## Session 2: 査読劇 ─ 3 人の専門家がツール設計を独立評価

**Structure**: tribunal（査読劇）Phase 1 独立評価 → Phase 2 反論対話 → Phase 3 統合

**査読者**: メソッド設計専門家 + DevTools 専門家 + 利根川（マーケ視点）

### Phase 1: 独立評価（抜粋）

**🔬 メソッド設計専門家（Major Revision）**:

> Strengths: 先行手法との差別化明確、3-Phase 原理が理論的に正しい（Delphi 法・弁証法）
>
> Weaknesses: 再現性保証なし、偽陰性率不明、Gap 品質指標が主観的

→ Gap: 偽陰性率の測定方法とベンチマークが未設計

**🧰 DevTools 専門家（Major Revision）**:

> Strengths: 2 コマンドインストール、skill 自動発動、flow YAML カスタマイズ性
>
> Weaknesses: Claude Code ロックイン（MCP サーバー化が必要）、テストなし

→ Gap: install.sh / update.sh のテストが存在しない

**🎰 利根川（Major Revision）**:

> 「ターゲットが曖昧。設計ドキュメントがある人しか使えないとしたら、それは致命的な制限だ。9 PV の現実を直視しろ。差別化メッセージが弱すぎる。」

→ Gap: ターゲットペルソナ未定義、エレベーターピッチなし

### Phase 2: 反論（先輩・後輩）

**反論して気づいたこと**: 利根川の「ターゲット曖昧」は認めざるを得なかった。今まで「設計レビューツール」としか言っていなかったが、本当のターゲットは **「Claude Code ユーザーで設計レビュー相手がいない開発者」** だ。一人開発で設計を誰にも見せられない人。これをペルソナに明文化した。

DevTools 専門家の「Claude Code ロックイン」は一部賛成。ただし MCP サーバー化は中期目標（v4）として残した。今は Claude Code で最高体験を作る方が先。

### 反論して初めて見えた Gap

査読劇の構造が機能した瞬間 ─ **Phase 2 の反論をやったから気づいた**。

---

## Session 3: フィードバック分析 ─ 実用データを DGE にかける

volta-auth-proxy（認証認可 SaaS ミドルウェア）プロジェクトが DGE v3.0.0 を 8 セッション使い、106 Gap を解決した後にフィードバックをくれた。そのフィードバック自体を DGE にかけた。

**📊 ビーン**: 「DGE 案採用 13 件 vs LLM 案採用 11 件。ほぼ拮抗。でも内容が全く違う。DGE はアーキテクチャ判断（5→2 コンポーネント削減）、LLM はセキュリティ詳細（CSRF、email_verified）。**補完関係が数字で証明された**。」

→ Gap: この補完関係のルールが method.md に明文化されていない

**🎭 ソクラテス**: 「Round 1-2 の Gap が Round 3-4 で無効化された。アーキテクチャが変わると既存の Gap が意味をなさなくなる。**Gap にライフサイクルが必要だ。Active → Void → Archived。**」

→ Gap: アーキテクチャ変更時の Gap 棚卸しがない

**⚔ リヴァイ**: 「自動反復の収束条件が『C/H Gap = 0』だけだ。Gap が 0 でも成果物が揃ってない。DB スキーマ、API 一覧、エラーコード、環境変数。チェックリストにして、全部揃うまで『実装レディ』と言うな。」

→ Gap: 実装レディネスチェックリストがない

## 3 セッションの成果

| セッション | Gaps | Critical | High | v3 での対応 |
|-----------|------|----------|------|------------|
| UX 査読 | 10 | 1 | 5 | サンプル追加、locale 自動判定 |
| 査読劇 | 11 | 2 | 6 | ペルソナ明確化、エレベーターピッチ |
| フィードバック分析 | 12 | 1 | 7 | Gap ライフサイクル、レディネスチェック |
| **合計** | **33** | **4** | **18** | — |

### v3.0〜v3.3 で実装されたもの（メタ DGE 起源）

- **固定枠+可変枠キャラ構造** — 「ヤンは全ラウンドに入れるべき」の洞察から
- **Gap ライフサイクル管理**（Active/Void/Archived）— ソクラテスの発見
- **実装レディネスチェックリスト** — リヴァイの要求
- **専門家キャラ自動提案** — 「テーマ固有の専門家を提案せよ」
- **サンプル設計ドキュメント** — 金八の「体験が先」から
- **locale 自動判定** — 深澤の指摘
- **エレベーターピッチ** — 「書いてないことを見つける。素の AI は書いたことをチェックする。」
- **DGE/LLM 補完関係の明文化** — ビーンが数字で証明

## 自己適用が証明したこと

DGE が自分自身のことを発見できたのは偶然ではない。

**素の AI にレビュー頼むのと何が違うか**: 素の AI にも v2 の問題点は列挙できる。「README に前提条件を書くべき」「テストがない」「ロックイン問題がある」。でも素の AI は **書いてあること（README の文言）** をチェックする。DGE が見つけたのは **書いていなかったこと** ─ ターゲットペルソナが存在しないこと、Gap にライフサイクルが必要なこと、補完関係が method.md にないこと。

そして査読劇の構造 ─ Phase 1 で独立評価、Phase 2 で反論 ─ のおかげで「なぜ私はこの設計判断をしたのか」を初めて言語化できた。1 人でコード書いてると絶対やらない。

**メタ認知をツールが提供した。**

## 使い方（30 秒）

```bash
npm install @unlaxer/dge-toolkit
npx dge-install
```

自分のプロダクトや設計について Claude Code で:

```
DGE して
```

または査読劇なら:

```
dge/ のシステム全体を査読して
```
