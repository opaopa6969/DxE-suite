# 認証認可サービスに DGE を 8 回かけたら 106 件の Gap が出て、アーキテクチャが変わった

> DGE が見つけるのは仕様の穴。でも今回見つかったのは、アーキテクチャそのものの間違いだった。

## TL;DR

- 認証認可 SaaS ミドルウェア（volta-auth-proxy）の設計に DGE を 8 セッション適用
- 106 件の Gap、全件解決
- DGE 案採用 13 件 / LLM 案採用 11 件 — ほぼ拮抗、でも種類が全く違う
- アーキテクチャが 5 コンポーネント → 2 コンポーネントに削減された

---

## プロジェクト概要

**volta-auth-proxy**: マルチテナント SaaS 向けの認証認可ミドルウェア。Keycloak、Auth0、独自 IdP をバックエンドとして抽象化し、アプリケーションに統一 API を提供する。

設計フェーズで DGE toolkit v3.0.0 を導入。8 セッション、全セッションにわたって同じプロジェクトの設計を深掘りし続けた。

---

## 8 セッションの概観

| セッション | テーマ | 投入キャラ | 主な発見 |
|-----------|--------|-----------|---------|
| Round 1 | コンポーネント設計 | 今泉 + 千石 + ヤン | 5コンポーネント構成が過剰 |
| Round 2 | 認証フロー | 千石 + Red Team + 僕 | PKCE 未実装、トークンローテーション欠如 |
| Round 3 | マルチテナント | ヤン + ソクラテス + 鷲津 | テナント分離の前提が間違い |
| Round 4 | API 設計 | 今泉 + 利根川 + リヴァイ | アーキテクチャ変更で Round 1-2 の Gap が無効化 |
| Round 5 | エラー設計 | 僕 + ハウス + 右京 | エラーコード体系が未定義 |
| Round 6 | SaaS 課金連携 | 鷲津 + 大和田 + 深澤 | Auth と Billing の境界が曖昧 |
| Round 7 | 運用・障害対応 | Auth 専門家 + ビーン | 障害時のフォールバックが存在しない |
| Round 8 | 実装レディネス | 全員 | 106件整理、実装可能か最終判定 |

**合計 106 件 / 全件解決。**

---

## 最初の大発見: アーキテクチャが間違っていた

Round 1 で出た Gap は「コンポーネントが多すぎる」だった。

設計では 5 つのコンポーネントを独立マイクロサービスとして動かす予定だった。Token Service、Session Service、Tenant Service、Audit Service、Proxy Service。

**☕ ヤン**: 「Token Service と Session Service は本当に別要る？ JWT ならステートレスなんだから、Session Service いらないんじゃないの？ Audit は横断的関心事で別サービスにすべきでもない。*シンプルにしようよ。*」

**🎭 ソクラテス**: 「なぜ 5 サービスにしたのか。マイクロサービスはスケール要件がある場合の手段。今の要件でそれがあるか？ 前提を確認した？」

Round 1 の時点では「設計判断だから変えない」という立場だったが、Round 3 で別の問題が出た。マルチテナント分離の実装が Token Service と Tenant Service の両方に散在していて、整合性を保てない。

**これは Round 1 のヤンが正しかった証拠だ。**

Round 4 で正式にアーキテクチャを変更。5 コンポーネント → 2 コンポーネント（Auth Service + Proxy Service）に削減。

**同時に、Round 1-2 の Gap のうち 7 件が無効化された。** Session Service 前提の Gap が全部消えた。

---

## Gap ライフサイクルの発見

このプロジェクトで初めて気づいたこと: **Gap には寿命がある。**

アーキテクチャ変更後に Gap 一覧を見ると、前のラウンドで発見した「重大な問題」がいくつか「もう存在しない問題」になっていた。

| Gap | Round 1-2 での状態 | Round 4 以降 |
|-----|-------------------|-------------|
| Session Service のトークン管理ロジックが重複 | Active / High | Void（Service 自体を削除） |
| Token-Session 間の整合性保証 | Active / Critical | Void |
| Session Service のスケール戦略 | Active / High | Void |

**Void** にしなければ、解決済みの問題として実装チームが対応しようとしてしまう。

この経験から、DGE toolkit v3.1.0 に Gap ライフサイクル管理（Active / Void / Archived）が追加された。

---

## DGE vs LLM: 数字で分かった補完関係

8 セッション終了後、全 106 件の解決策を分類した。

| 解決策の出所 | 件数 | 主な内容 |
|------------|------|---------|
| DGE 案（DGEで発見・議論した方向性） | 13 件 | アーキテクチャ判断、設計の方向性変更 |
| LLM 案（素の LLM に出してもらった詳細）| 11 件 | CSRF トークン実装、email_verified 検証、JWT claim 設計 |
| 両方の組み合わせ | 82 件 | DGE が「何を決めるか」、LLM が「どう実装するか」 |

**DGE が決めたもの（例）:**
- 5 → 2 コンポーネント削減（Round 1 のヤンの指摘）
- PKCE 実装を必須化（Round 2 の Red Team の攻撃シナリオから）
- Auth と Billing の境界の明確化（Round 6 の鷲津の「IRR は？」から）

**LLM が補完したもの（例）:**
- CSRF トークン実装の具体的コード
- email_verified claim の検証ロジック
- JWT payload の詳細スキーマ

**DGE は「何を決めるか」を決め、LLM は「どう実装するか」を補完する。** 役割分担が明確だ。

## キャラクターの学習

8 セッションで分かった: **テーマ固有の専門家キャラが built-in キャラを上回る。**

| キャラ | 満足度 | コメント |
|-------|--------|---------|
| Auth専門家（カスタム） | ★★★★☆ | 認証プロトコルの細部を的確に突いた |
| SaaS専門家（カスタム） | ★★★★☆ | マルチテナント境界の問題を早期発見 |
| 今泉（built-in）| ★★★☆☆ | 前提を問うのは有効。でも認証ドメイン知識なし |
| 僕（built-in） | ★★☆☆☆ | 「全部大変そう...」の繰り返しに |

**教訓**: 専門性の高いプロジェクトには、専門家キャラを最初から作ってインストールする。

```
Auth専門家を追加して
```

Claude Code でこう言えば、プロジェクトの文脈から専門家キャラを自動生成して永続保存する。

## 106 件を全解決した後に残ったもの

全 Gap を解決した Round 8 終了後、ヤンが一言:

**☕ ヤン**: 「Gap ゼロになったけど、実装できる状態なの？ DB スキーマ、API 一覧、エラーコード定義、環境変数一覧、SDK API リファレンス。これ全部あるの？」

**Gap ゼロ ≠ 実装レディ。**

この発見から、DGE v3.1.0 に**実装レディネスチェックリスト**が追加された。

```
[ ] DB スキーマ（全テーブル・カラム定義済み）
[ ] API 一覧（全エンドポイント・メソッド・レスポンス定義済み）
[ ] エラーコード体系（全エラーコード・メッセージ定義済み）
[ ] 環境変数一覧（全変数・デフォルト値定義済み）
[ ] SDK API リファレンス（全メソッド・型定義済み）
[ ] 画面一覧（認証フロー画面の全遷移定義済み）
```

全部 ✅ になって初めて「実装レディ」。

## 8 セッション振り返り

**DGE でやってよかったこと:**
- アーキテクチャの根拠を問い直せた（Round 1 のヤン）
- 攻撃者視点の Gap が早期に出た（Round 2 の Red Team）
- Gap が無効化されるのを管理できた（ライフサイクル）

**素の LLM に任せてよかったこと:**
- セキュリティ実装の詳細（CSRF、JWT claim、email_verified）
- エラーコードの網羅的なリスト
- API スキーマの具体的な型定義

**両方やって初めて「設計完了」と言えた。**
