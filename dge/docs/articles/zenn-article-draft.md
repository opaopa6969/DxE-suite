# 設計レビューが退屈すぎるので、キャラクターに殴り合いさせることにした

> 仕様書のレビューは「書いてあることの検証」。DGE は「書いてないことの発見」。

## TL;DR

- プロジェクトルートで `npm install @unlaxer/dge-toolkit && npx dge-install`
- Claude Code に「DGE して」と言う
- キャラクターが設計について議論して、仕様書の穴を見つけてくれる
- 2 分で 3 件の Gap。しかも設計の前提を覆すやつ。

## 設計レビュー、楽しいですか？

正直に言いましょう。楽しくない。

Pull Request のレビューで「ここの型が...」「このエラーハンドリングが...」と指摘する日々。spec レビューで「この API のレスポンスコードが...」と会議室で眠くなる午後。

**DGE（Dialogue-driven Gap Extraction）** は、設計レビューを「会話劇」に変えるツールです。

キャラクターたちが設計について議論する。品質の鬼が「それはユーザーへの侮辱です」と怒り、怠惰な天才が「要らなくない？ 紅茶ください」と返す。その対話の中から、仕様書に書かれていない「穴」が浮かび上がってくる。

楽しい。そして実用的。

## セットアップ: 30 秒

プロジェクトのルートで:

```bash
npm install @unlaxer/dge-toolkit
npx dge-install
```

終わり。`dge/` フォルダと `.claude/skills/` にスキルファイルが入ります。

Claude Code で「DGE して」と言えば起動。

## 実際にやってみた: 認証 API の設計レビュー

とあるプロジェクトの認証 API を DGE にかけてみます。

```
Human: 認証 API の設計を DGE して
```

DGE toolkit v3.3.0

Phase 0: コンテキスト収集中...
構造: ⚙ design-review を自動選択
固定枠: 今泉 + ヤン / 可変枠: 千石 + 僕 を推奨します。変更しますか？

```
Human: それで
```

```
Human: Red Team も入れて
```

### Scene 1: Happy Path — 「基本フロー」

> **先輩（ナレーション）**: 認証 API は JWT ベースのトークン認証。ログイン → トークン発行 → リクエストごとにトークン検証。リフレッシュトークンは 30 日有効。

**👤 今泉**: 「あの... 基本的なことなんですけど、リフレッシュトークンの 30 日って、誰が決めたんですか？ ユーザーに聞きました？ 30 日間ログインしっぱなしって、セキュリティ的にどうなんですか？ そもそもリフレッシュトークンって何ですか？」

→ Gap 発見: リフレッシュトークンの有効期限の根拠が未定義

**🎩 千石**: 「30 日？ 7 日でしょう。いえ、問題はそこではありません。リフレッシュトークンを発行した後、古いトークンはどうなるのですか？ ローテーションしないのですか？ 同時に複数のリフレッシュトークンが有効になる設計は、お客様への侮辱です。」

→ Gap 発見: リフレッシュトークンのローテーションポリシーが未定義

**😰 僕**: 「...あの、ログアウトって、どうするんですか...？ JWT はステートレスだから、サーバー側でトークンを無効化できないんじゃ...。ブラックリストを管理するんですか...？ Redis とか要りますか...？ すべてがダメになる予感がするんですけど...。」

→ Gap 発見: ログアウト（トークン無効化）の仕組みが未設計

**今泉がいきなり本質を突いた。** 「リフレッシュトークンって何ですか？」— この質問、専門家は聞けない。知ってるから。でも今泉は知らないから聞ける。そして「30 日の根拠は？」は、実は誰も答えられない。

**僕の不安は予言だった。** JWT のログアウト問題は定番のハマりポイント。僕みたいなキャラがいなかったら「実装してから気づく」やつ。

---

### Scene 2: エッジケース — 「攻撃者が来る」

> **先輩**: ここで Red Team が参戦する。攻撃者の視点で認証 API を見る。

**😈 Red Team**: 「リフレッシュトークンが漏洩した場合のシナリオ。攻撃者がトークンを盗んで、正規ユーザーより先にリフレッシュする。正規ユーザーのトークンは無効化される。攻撃者がアカウントを乗っ取り、正規ユーザーはログインできなくなる。——で、この API にレート制限はあるのか？」

→ Gap 発見: トークンリフレッシュのレート制限がない

**👤 今泉**: 「ちょっと待ってください。トークンが盗まれたって、どうやって盗むんですか？ localStorage に入ってるんですか？ Cookie ですか？ httpOnly ですか？ そもそも聞いたんですか？ フロントエンドの人に。」

→ Gap 発見: トークンの保存場所（localStorage vs httpOnly Cookie）が未決定

**😈 Red Team**: 「localStorage なら XSS で一発。httpOnly Cookie なら CSRF。どっちを選んでも攻撃ベクトルがある。で、どっちも書いてない。——セキュリティレビューの前にセキュリティの設計がない。」

**😰 僕**: 「...やっぱりすべてがダメになる予感がします...。小規模にしませんか...？ セッションベース認証でいいんじゃ...？ JWT 使う理由ってあるんですか...？ ...でも、やるしかないですけどね...」

→ Gap 発見: JWT を選択した技術的理由が未記載（セッションベースで十分な可能性）

**落ちがついた。** 「そもそも JWT いる？」— 僕が最後にぼそっと言った一言が一番重い。認証方式の選定理由が仕様書にない。ありがちだけど、普通のレビューでは「もう JWT で作り始めてるし...」で流される。DGE では流されない。僕が止める。

### Gap 一覧

たった 2 シーンで 6 個の Gap が出た。

| # | Gap | Severity |
|---|-----|----------|
| 1 | リフレッシュトークン有効期限の根拠がない | High |
| 2 | トークンローテーションポリシー未定義 | High |
| 3 | ログアウト（トークン無効化）が未設計 | Critical |
| 4 | レート制限がない | High |
| 5 | トークン保存場所が未決定 | High |
| 6 | JWT を選んだ理由が未記載 | Medium |

**Critical 1 件、High 4 件。** 30 分の spec レビュー会議より多い。

## 「普通に AI にレビュー頼むのと何が違うの？」

一番聞かれる質問なので、実験した。

同じ認証 API を、DGE と素の AI（完全に独立した別セッション）の両方でレビュー。

| | DGE | 素の AI |
|---|-----|--------|
| 問題点の数 | **9 件** | **28 件** |
| Critical | 2 件 | 2 件 |
| High | **6 件** | 9 件 |

**数は素の AI の圧勝。** 3 倍。でも中身が違う。

**素の AI**: 「CSRF 対策がない」「HTTPS 強制が未記載」「パスワードリセットがない」— **ベストプラクティスのチェックリスト**。正しいけど、設計ドキュメントを読まなくても出せる。

**DGE だけが出したもの**: 「JWT を選んだ根拠は？」「トークン有効期限がアプリの性質に合ってない」「OAuth 追加時にスキーマが壊れる」— **設計の前提そのものを問う指摘**。

素の AI は設計を受け入れてから「足りないもの」を列挙する。DGE は設計自体を疑う。

**面白い発見:** 同じ AI に DGE → 素の順で両方やらせると、素の結果は 15 件。別セッションでやると 28 件。AI が無意識に DGE の結果を避けてしまう。だから DGE toolkit は自動で別セッションで実行してマージする機能がある。

**結論: 両方やるのが最強。** 素の AI は広く浅いチェックリスト、DGE は深く鋭い洞察。「JWT いる？」は素の AI には出せない。空気を読むから。今泉は読まない。

## 19 人のキャラクター

DGE には 19 人の built-in キャラクターがいます（日本語版 + 英語版）。

```
前提が怪しい    → 👤 今泉   「そもそも聞いたんですか？」
品質が低い      → 🎩 千石   「お客様への侮辱です」
全部複雑        → ☕ ヤン   「要らなくない？ 紅茶ください」
前に進みすぎ    → 😰 僕     「小規模にしませんか...？」
大胆さが足りない → 👑 ラインハルト 「攻めろ」
数字が甘い      → 🦅 鷲津   「IRR は？」
攻撃への耐性    → 😈 Red Team「競合がこうしたら？」
収益の現実      → 🦈 大和田  「いくら稼げるんだ？」
実装の不足      → ⚔ リヴァイ 「汚い。作れ。」
ユーザーの本音   → 🎰 利根川  「ユーザーの言葉で語れ」
隠れた問題      → 🏥 ハウス  「全員嘘をついている」
法的リスク      → ⚖ ソウル  「利用規約は書いたか？」
翻訳者         → 🪄 林     「つまりこういうことです」
理解を助ける    → 🧑‍🏫 金八   「一緒に考えよう」
UX の直感      → 🎨 深澤   「それ、必然ですか？」
データで語る    → 📊 ビーン  「データは何と言ってる？」
混乱を整理      → 🤝 後輩   「建設的にいきましょう」
矛盾を見抜く    → 🕵 右京   「細かいことが気になるんです」
```

テーマで組み合わせが変わる。API 設計なら今泉+千石+僕。セキュリティなら千石+Red Team+ハウス。好きなキャラを追加することもできる（「ガッツを追加して」で永続保存）。

## 中身と機能

- **3 モード**: ⚡ Quick（即座に会話劇）/ 🔍 Design Review（テンプレート+パターンで構造的に）/ 💡 Brainstorm（アイデア出し）
- **20 の会話パターン**: `zero-state`（データ 0 で起動）、`scale-break`（N=100K で動かす）、`security-adversary`（攻撃者視点）等
- **自動反復**: 「実装できるまで回して」で Gap が出なくなるまでパターンを変えて繰り返す
- **Gap → Spec 変換**: Use Case / Tech Spec / ADR を自動生成
- **カスタムキャラ**: 好きなキャラを追加。wizard でオリジナルも作れる
- **マルチツール対応**: Claude Code、Codex、Gemini CLI、Cursor で動作
- **フローカスタマイズ**: YAML でフロー定義を変更可能

詳しくは [INTERNALS.md](https://github.com/opaopa6969/DGE-toolkit/blob/main/kit/INTERNALS.md)（mermaid フロー図あり）

## DGE が見つけるもの、見つけないもの

ここまで読んで「便利そう」と思ってくれた方に、正直に書いておく。

**DGE が得意なこと:**
- 仕様書に書かれていない前提を炙り出す（「なぜ JWT？」「30 日の根拠は？」）
- 複数の機能が組み合わさったときに生まれる問題を発見する（Path Traversal + API 公開 → データ漏洩）
- 設計判断の根拠を問い直す — 普通の AI レビューが「空気を読んで」スキップするところ

**DGE が苦手なこと:**
- ベストプラクティスの網羅的チェック — これは素の AI の方が得意（28 件 vs 9 件）
- 実装レベルのバグ検出 — DGE は設計レビューツール。コードレビューは別で
- 1 人で使い続けると「キャラに慣れる」— たまにキャラを変えるか、カスタムキャラを追加するのが効く

だから **DGE だけで完結するとは言わない**。素の AI のチェックリストも、人間のレビューも必要。DGE が埋めるのは「その間にある、誰もカバーしていなかった穴」。

そしてその穴が、一番ヤバいやつだったりする。

## DGE が生まれたきっかけ

パーサーフレームワーク [unlaxer-parser](https://github.com/opaopa6969/unlaxer-parser) の開発中、Claude にコーディングとドキュメンテーションをやらせていた。ある日、Claude がコードの奥の方に**回文パーサ**を発見した。そこから unlaxer の PropagationStopper 等の独自性に気づき、「これは論文にすべきだ」と Claude が主張した。

「じゃあ書いてみれば」と書かせてみた。結構よかった。

でも査読してもらう人が周りにいない。じゃあ**架空の学会で査読者に査読してもらおう**。3 人の査読者が独立に論文とソースコードを評価し、先輩と後輩がその評価に必死で反論する「査読劇」を生成した。

**5 回の査読劇で 108 個の Gap を発見。** この手法に惚れ込んで「誰でも使えるようにしよう」と npm パッケージ化したのが DGE toolkit。

- 査読劇の全文: [unlaxer-parser DGE papers](https://github.com/opaopa6969/unlaxer-parser/blob/master/paper/INDEX.md)
- DGE toolkit: [GitHub](https://github.com/opaopa6969/DGE-toolkit)
