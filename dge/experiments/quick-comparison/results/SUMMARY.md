## 主要な発見

### DGE が強いこと
1. **前提を疑う**: 「JWT いる？」「GraphQL いる？」「そもそもリファクタ必要？」— 素は設計を受け入れてから改善案を出す
2. **ビジネス視点**: 大和田「コスト構造は？」利根川「ユーザーの本音は？」— 素は技術面しか見ない
3. **運用視点**: 僕「テーブル膨張する...」「監視は？」— 素は実装面に偏る
4. **代替戦略**: ヤン「REST + OpenAPI で十分」「Strangler Fig で」— 素は改善案は出すが代替は出さない
5. **詭弁検知**: ソクラテスが 5 回検知。仕様書の暗黙の前提を壊す

### 素が強いこと
1. **網羅性**: Medium/Low の実装詳細を漏れなく列挙
2. **チェックリスト的**: 「CORS は？」「2FA は？」「税金は？」— 一般的な機能リストの確認
3. **速度**: 構造化せず全部出すので出力が速い

## 結論

```
素の LLM = 幅広いチェックリスト（Medium/Low が多い）
DGE      = 深い洞察（Critical/High が多い、前提を疑う）
最強     = 両方やる
```

### 推奨ワークフロー
1. **DGE で回す** → Critical/High の「設計の穴」を見つける
2. **素の LLM にもレビューさせる** → Medium/Low の「実装の漏れ」を拾う
3. **人間がマージ** → 両方の Gap を統合して Spec 化

### 注意事項
- この実験は同一の LLM（Claude）が両方を実行している。厳密な比較ではない
- DGE の出力はキャラ選択に依存する。異なるキャラなら異なる結果になる
- N=3（テーマ 3 つ）は統計的有意性にはほど遠い。傾向の確認程度
