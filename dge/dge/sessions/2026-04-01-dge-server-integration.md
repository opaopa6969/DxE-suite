# DGE Session: dge-server 統合設計
- Date: 2026-04-01
- Flow: quick ⚡
- Theme: dge-server をスキルからオプションで呼ぶ設計
- Characters: ヤン、今泉、Saul、Reinhard
- Rounds: 2

## Scene 1: そもそも論

**今泉：** 確認させてください。dge-server がないと何が困るんですか？今もファイルベースで動いていますよね。

**ヤン：** そう。`dge/characters/index.md` を読めばキャラ一覧は取れる。`POST /api/characters/recommend` を呼ぶより速いかもしれない。サーバーを立てるコストに見合うメリットを先に定義しないと。

**Saul：** 私が気になるのはそこじゃなくて、「サーバーが落ちているときの挙動」です。ファイルモードにフォールバックするとして、その切り替えを誰がいつやるのか。セッション中にサーバーが落ちたら？

**Reinhard：** 設計の一貫性の問題ですね。「オプション扱い」と言うのは簡単ですが、サーバーモードとファイルモードで挙動が変わるなら、テスト可能性が半分になる。

→ Gap 発見: セッション中のサーバーダウン時、中途切替の挙動が未定義 🔴 Critical

## Scene 2: 推奨エンジンの価値

**今泉：** `POST /api/characters/recommend` の推奨ロジックはスキル内のロジックと何が違うんですか？

**Reinhard：** つまり同じロジックが2箇所に存在する。スキルとサーバーで推奨結果が変わるなら、どちらが正典か？

**Saul：** ユーザーから見ると「サーバーあり/なしで推奨キャラが変わる」という状況が起きうる。再現性が失われます。

**ヤン：** サーバーに移すなら、スキル内のロジックは完全に削除すべきです。中途半端に残すと永遠に両方メンテする羽目になる。

→ Gap 発見: 推奨ロジックがスキルとサーバーに二重存在、移行後の削除計画なし 🟠 High

## Scene 3: session 保存の価値

**Saul：** セッションをDBに保存する価値は何ですか？

**ヤン：** DB に入れると「プロジェクト横断の集計」ができる。

**今泉：** でも今それを使う人がいますか？集計・可視化の UI がなければ意味がない。

**Reinhard：** 「ファイルが正典」と「DBが正典」の境界を最初に決めるべきです。

→ Gap 発見: ファイルとDBの正典定義がない 🟡 Medium

## Scene 4: メリット・デメリット整理

**今泉：** サーバーを入れることで何が「できるようになる」のか、一言で言えますか？

**ヤン：** 「カスタムキャラの永続管理」じゃないですか。

**Saul：** DBはローカルの SQLite ですよね。チームで使う場合、DBが各自のマシンにバラバラに存在する。カスタムキャラを共有したければ結局ファイルをコピーしないといけない。

→ Gap 発見: カスタムキャラDBはローカル単体でチーム共有に使えない 🟡 Medium

**ヤン：** 推奨が「もっと賢くなる必要がある」という前提があって初めてサーバー化の価値が出る。

→ Gap 発見: 前提（推奨品質への不満・チーム共有要件）が曖昧なまま設計を進めている 🟠 High

## メリット・デメリット まとめ

| 機能 | メリット | デメリット |
|---|---|---|
| キャラ推奨 API | ロジックを外部化・差し替えやすい | スキル内ロジックと二重になる |
| セッション記録 DB | 将来の横断集計に備えられる | UI なしでは使えない・書き込みコストだけ増える |
| カスタムキャラ管理 | ファイルより手軽 | ローカル単体・チーム共有できない |
| オプション起動 | サーバーなしでも動く | 挙動が2パターンになりテストが複雑 |
