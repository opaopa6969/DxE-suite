# DGE Session: 初めてのユーザー体験レビュー

- **日時**: 2026-03-31
- **テーマ**: 初めての人が npm install してから DGE を使えるまでのユーザー体験
- **Flow**: ⚡ quick
- **キャラ**: 👤 今泉 + 🎨 深澤 + 😰 僕 + 🧑‍🏫 金八

---

## Scene 1: インストール — 「最初の 30 秒」

> **先輩（ナレーション）**: 初めてのユーザーが dev.to の記事を見て DGE を試そうとしている。README か記事の指示通り `npm install @unlaxer/dge-toolkit && npx dge-install` を実行する。プロジェクトに `dge/` フォルダと `.claude/skills/` が生成される。

**👤 今泉**: 「あの... 基本的なことなんですけど、この人は npm のプロジェクトを持ってるんですか？ `package.json` がない状態で `npm install` したらどうなるんですか？ そもそも、この人は Claude Code を持ってるんですか？ 記事に書いてありましたっけ？」

→ Gap 発見: 前提条件（package.json の有無、Claude Code のインストール済み）が明示されていない

**🎨 深澤**: 「`dge/` に 20 個くらいファイルがコピーされるわけでしょう。初めて見た人はどう感じる？ 自分のプロジェクトに突然 `characters/` `templates/` `flows/` `specs/` `sessions/` って知らないフォルダがどっさり現れる。怖いよ。『これ何？ 消していいの？ .gitignore に入れるの？』って絶対思う。」

→ Gap 発見: インストール直後に「何が入ったか」「何を gitignore すべきか」のガイダンスがない

**😰 僕**: 「...あの、postinstall のメッセージが `Run: npx dge-install` って出るんですけど...、npm install の出力って大量のログに埋もれるんじゃ...？ 見逃しますよ、絶対...。しかも `npx dge-install` って 2 段階あるのが...分かりにくいんじゃ...。`npm install` だけで終わると思う人、いますよね...？」

→ Gap 発見: 2 段階インストール（npm install → npx dge-install）が直感的でない。postinstall メッセージが埋もれるリスク

**🧑‍🏫 金八**: 「ちょっと待ちなさい。この人は "DGE" が何か分かってるのかい？ 記事を読んで来たとしても、`dge/method.md` は 300 行以上ある。最初に開くファイルが 300 行のメソッド解説だったら、そこで閉じるよ。最初の 1 歩は『DGE して』と言うだけでいいんだ。でも、そのことがインストール完了後のメッセージに書いてあるかい？」

→ Gap 発見: インストール後の「最初にやること」が明確でない

---

## Scene 2: 最初の「DGE して」 — 「何が起きるの？」

> **先輩**: ユーザーが Claude Code で「DGE して」と入力した。skill が発動してテンプレートとキャラを自動選択する。しかし、このユーザーは自分のプロジェクトの「何を」DGE したいか明確でないかもしれない。

**👤 今泉**: 「そもそも、『DGE して』だけで何がレビューされるんですか？ ユーザーは設計ドキュメントを持ってない場合もありますよね？ README しかないプロジェクトで『DGE して』って言ったら何が起きるんですか？ サンプルのドキュメントとか、ないんですか？」

→ Gap 発見: 設計ドキュメントがないプロジェクトでの初回体験が未定義。サンプル入力がない

**🎨 深澤**: 「最初の体験が `テンプレート: api-design（自動選択）/ キャラクター: 今泉 + 千石 + 僕 を推奨` って、初めての人には意味不明だよ。今泉って誰？ 千石って誰？ テンプレートって何のこと？ いきなり選択を迫られる感じ。最初の 1 回だけでいいから『おまかせで回しますね、見ててください』って全自動で走らせてほしい。体験してから理解する順番じゃないと。」

→ Gap 発見: 初回起動時にキャラ説明やテンプレート説明なしで選択を提示している

**😰 僕**: 「...あの、会話劇が出力された後の話なんですけど...。Gap 一覧が出て、番号付き選択肢が出ますよね。『1. DGE を回す / 2. 実装できるまで回す / 3. 詳しくやる / 4. 素の LLM でマージ / 5. 後で』...。初めてで、どれ選べばいいか分かりませんよね...？ 『4. 素の LLM でマージ』って何ですか...？ ...僕だったら、怖くて 5 の『後で』を押します...。」

→ Gap 発見: 初回ユーザーに高度な選択肢を提示するのは認知負荷が高い

**🧑‍🏫 金八**: 「深澤の言う通りだよ。初めての授業は、まず先生がやって見せるものだ。サンプルの設計ドキュメント 1 枚入れておいて、『まず試してみたい人はこれで DGE して、と言ってください』でいい。体験が先、理解が後。」

→ Gap 発見: 「体験ファースト」の導線がない

## Scene 3: 英語ユーザー — 「--lang en の落とし穴」

> **先輩**: dev.to の記事を読んだ英語ユーザーが `npx dge-install --lang en` を実行した。英語版のキャラクターとテンプレートがインストールされる。

**👤 今泉**: 「ところで、`--lang en` を付け忘れた人はどうなるんですか？ 日本語ファイルがインストールされますよね？ 英語圏の人が `npx dge-install` って打ったら、日本語のキャラクターが入る。それ、README に書いてありますか？ 『デフォルトは日本語』って。」

→ Gap 発見: デフォルト日本語の明示なし。--lang en 忘れのリカバリなし

**🎨 深澤**: 「そもそも、なぜ自動判定しないの？ 環境の `LANG` 変数か、シェルの locale を見ればいい話でしょ。ユーザーに『--lang en を付けてね』って頼むのは、設計者の怠慢をユーザーに押しつけてる。」

→ Gap 発見: locale 自動判定が未実装

**😰 僕**: 「...あの、もっと怖い話があるんですけど...。`--lang en` で入れた後に、`npx dge-update` を実行したら...。update.sh は `.lang` ファイルを読むって書いてあるけど...、今日作ったばかりですよね...？ テスト...してますか...？ `.lang` ファイルが消えてた場合...日本語で上書きされちゃうんじゃ...？」

→ Gap 発見: .lang ファイル消失時に英語環境が日本語で上書きされる

## Round 2: 角度変更 — 「つまずいたとき」と「2回目以降」

キャラ: ☕ ヤン + ⚔ リヴァイ + 🎰 利根川 + 🕵 右京

---

### Scene 1: エラーと挫折 — 「壊れたとき何が起きる」

> **先輩（ナレーション）**: ユーザーが DGE を使おうとしてエラーに遭遇するケースを考える。npm install の失敗、Claude Code が skill を認識しない、会話劇の生成が途中で止まる、など。

**☕ ヤン**: 「まず整理しよう。このツールの依存関係は何だ？ npm、Node.js、Claude Code、そして Claude Code が skill を読むこと。4 つの前提が全部揃わないと動かない。で、そのうち 1 つでもコケたとき、ユーザーに何が表示される？ ……何も表示されないだろ？ Claude Code が skill を読まなかったら、ただ『DGE して』に普通の返事が返るだけ。エラーじゃないから気づけない。最悪のパターンだ。」

→ Gap 発見: skill が読み込まれなかった場合のフィードバックがゼロ。ユーザーはDGEが「動いてない」ことに気づけない

**⚔ リヴァイ**: 「install.sh のエラーハンドリング。`cp "${SRC}/characters/en/"*.md` — en ディレクトリにファイルが 1 つもなかったら？ glob が展開されずにそのまま渡されて cp がコケる。`set -euo pipefail` で即死。ユーザーには `cp: cannot stat` のシェルエラーが出るだけ。……汚い。ユーザー向けのエラーメッセージを書け。」

→ Gap 発見: install.sh / update.sh のエラーがシェル生エラーのまま

**🎰 利根川**: 「お前ら技術の話ばかりだが、ユーザーの本音を言ってやる。『npm install したけど動かない』。この時点でユーザーは何をする？ GitHub の Issues を見る。README の Troubleshooting を見る。……あるか？ ないだろう。Stack Overflow で検索する？ ヒットするか？ 新しいツールだぞ。ユーザーは詰む。詰んだらアンインストールする。終わり。」

→ Gap 発見: トラブルシューティングガイドが存在しない

**🕵 右京**: 「細かいことが気になるんですよ。README には `npx dge-install` と書いてある。dev.to の記事には `npx dge-install --lang en` と書いてある。npm の postinstall メッセージには `Run: npx dge-install` と書いてある。3 箇所で微妙に違うコマンドを案内してますね。……あと、`npx dge-install --help` を叩いたら何が出ますか？」

→ Gap 発見: ドキュメント間でコマンド不統一。--help 未実装

### Scene 2: 2 回目以降 — 「リピーターにならない問題」

> **先輩**: 初回の DGE を体験したユーザーが、翌日もう一度使おうとしている。

**☕ ヤン**: 「昨日の session を見たい。`dge/sessions/` にファイルがあるはず。ファイル名は誰が決めた？ LLM が勝手に決めた。一覧を出すコマンドは？ `dge-tool` で出せるのか？ 出せないなら `ls dge/sessions/` を打てと？」

→ Gap 発見: 過去セッションの一覧・検索手段が不足

**⚔ リヴァイ**: 「キャラを変えたいユーザー。19 人の一覧はどこだ？ 会話の中で『キャラ一覧見せて』と言えば出るのか？ dge-session.md にそのトリガーがあるか。」

→ Gap 発見: 会話中のキャラ一覧確認の導線が不明確

**🎰 利根川**: 「2 回目に使うかは、1 回目の Gap が『刺さったか』で決まる。Gap が全部『知ってたこと』だったら 2 回目はない。デフォルトキャラ（今泉+千石+僕）でプロジェクト固有の深い洞察が出るか？ context を知らないキャラが出す Gap は汎用的になりがちだぞ。」

→ Gap 発見: デフォルトキャラが汎用的すぎて「刺さる Gap」が出にくい

**🕵 右京**: 「前回どのパターンを使ったかは session ファイルに書いてありますか？ 『前回 zero-state だから今回は scale-break で』という判断の導線がありますか？」

→ Gap 発見: 前回パターンの記録と次回選択の導線がない

## Round 3: パターン `role-contrast` — 「ユーザーの種類で体験が違う」

キャラ: 🧑‍🏫 金八 + 🎰 利根川 + ☕ ヤン + 🎭 ソクラテス

### Scene 1: ソロ開発者 vs チームリーダー

> **先輩**: DGE を使うユーザーには少なくとも 3 タイプいる。① 個人開発者、② チームリーダー、③ ジュニア。

**🎰 利根川**: 「チームリーダーの立場で言う。メンバーに『DGE 入れろ』と指示する。`dge/` がプロジェクトに入る。——で、これ git に入れるのか？ `sessions/` はローカルだろ？ `.gitignore` のテンプレートが必要だ。チーム導入のガイドがない。」

→ Gap 発見: チーム導入ガイドがない。git / .gitignore の方針が不明

**🧑‍🏫 金八**: 「ジュニアの子は Gap が出ても次のアクションが分からない。チケットに起こす？ Spec 化する？ DGE の出力をワークフローに繋げる導線がスキルレベルに依存しすぎてる。」

→ Gap 発見: DGE 出力から次のワークフローへの接続がスキルレベル依存

**☕ ヤン**: 「もっと根本的な話。チームの全員が Claude Code を使ってるか？ Copilot や Cursor のメンバーは？ skill は Claude Code 専用だ。**Claude Code に完全ロックイン**。チーム導入のハードルがめちゃくちゃ高い。」

→ Gap 発見: Claude Code ロックイン

**🎭 ソクラテス**: 「method.md には『任意の LLM に貼り付ける』プロンプトがある。この存在を知っているユーザーが何人いると思う？」

→ Gap 発見: Claude Code 以外での使い方の案内が埋もれている
