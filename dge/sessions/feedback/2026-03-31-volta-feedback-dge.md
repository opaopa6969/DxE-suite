## Scene 1: 「何がうまくいって、何が足りなかったか」

> **先輩（ナレーション）**: volta-auth-proxy で DGE v3.0.0 を本格使用した。8 セッション、106 Gap、全て解決。フィードバックには 7 つの改善提案がある。

**📊 ビーン**: 「DGE 案採用 13 件 vs LLM 案採用 11 件。ほぼ拮抗。でも内容が全く違う。DGE はアーキテクチャ判断（5→2 コンポーネント削減）、LLM はセキュリティ詳細（CSRF、email_verified）。**補完関係が数字で証明された**。問題は、この補完関係のルールがどこにも明文化されてないこと。」

→ Gap 発見: DGE と LLM の補完関係（方針 vs 詳細）が method.md に明文化されていない

**☕ ヤン**: 「『DGE の解決策が方針レベルで止まりがち。LLM は DB スキーマ・コード例まで出す。DGE は「こうすべき」で止まる。』DGE の根本的な弱点。**DGE のゴールは「判断」であって「実装」ではない**。これを明確にすべき。」

→ Gap 発見: DGE のスコープ（判断まで / 実装は LLM）が明確に定義されていない

**🎭 ソクラテス**: 「『Round 1-2 の Gap が Round 3-4 で無効化された』。アーキテクチャが変わると既存の Gap が意味をなさなくなる。**Gap にライフサイクルが必要だ。Active → Void → Archived。**」

→ Gap 発見: アーキテクチャ変更時の Gap 棚卸し（無効化判定）がない

**⚔ リヴァイ**: 「自動反復の収束条件が『C/H Gap = 0』だけだ。Gap が 0 でも成果物が揃ってない。DB スキーマ、API 一覧、エラーコード、環境変数、SDK API、画面一覧。チェックリストにして、全部揃うまで『実装レディ』と言うな。」

→ Gap 発見: 自動反復の収束条件が「C/H = 0」のみ。実装レディネス判定がない

## Scene 2: 「キャラクター構成の教訓」

**📊 ビーン**: 「専門家キャラ（ad-hoc）が built-in を上回っている。Auth専門家★4、SaaS専門家★4 > 今泉★3、僕★2。テーマ固有の専門家を自動提案する仕組みが必要。」

→ Gap 発見: テーマ固有の専門家キャラ自動提案がない

**☕ ヤン**: 「『ヤンは全ラウンドに入れるべき』。固定枠（常に入る汎用キャラ）+ 可変枠（テーマで変わる専門家）の 2 層構造。」

→ Gap 発見: 固定枠 + 可変枠の 2 層キャラ構造がない

**🎭 ソクラテス**: 「深澤の投入が遅すぎた。推奨アルゴリズムに『不足領域の警告』が要る。」

→ Gap 発見: キャラ推奨に「不足領域の警告」がない

**⚔ リヴァイ**: 「新パターン 3 件（delegation-matrix, phase-minimization, protocol-design）。patterns.md と templates/ に追加しろ。」

→ Gap 発見: 新パターン 3 件が未追加

## Scene 3: 「フィードバックループの仕組み」

**📊 ビーン**: 「フィードバックの構造をテンプレート化。プロジェクト横断で比較・集約できるようにする。」

→ Gap 発見: フィードバックテンプレートがない

**☕ ヤン**: 「フィードバック → 改善のパイプライン: 使う → 記録 → DGE で分析 → method/patterns に反映 → 次で使う。問題はセッション終了時の自動収集がないこと。」

→ Gap 発見: セッション終了時のフィードバック自動収集がない

**🎭 ソクラテス**: 「n=1 で method を変えていいのか？ 複数プロジェクトで確認されたものだけ反映すべきでは。」

→ Gap 発見: method 反映の基準が未定義

**⚔ リヴァイ**: 「議論してる暇があるなら作れ。すぐやれるもの: パターン追加、テンプレート作成、method に補完関係セクション。」

→ Gap 発見: 7 提案の実装優先順位が未整理
