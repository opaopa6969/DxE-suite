## Scene 1: 初回体験

先輩 (ナレーション):
DGE toolkit は README.md でクイックスタートを提供している。
手順は 3 ステップ: (1) プロジェクトに dge/ フォルダをコピー、
(2) Claude Code に「DGE して」と言う、(3) Gap を Spec に変換。
method.md は 209 行、characters/catalog.md は 155 行。
テンプレートは 5 種類用意されている。
初めてこの toolkit を見た開発者が、最初の 5 分間で何を体験するかを議論する。

利根川: いいか、現実を教えてやる。開発者がこの README を開いた瞬間に何が起きるか。「DGE は 2 つの意味を持つ」「Design-Gap Exploration」「Dialogue-driven Gap Extraction」......読む側の脳が処理するのは最初の 3 行だ。そこで「これは何をしてくれるのか」が分からなければ、タブを閉じる。お前たちの README は「定義」から始まっている。ユーザーが知りたいのは定義じゃない。「俺の仕事が楽になるのか？」だ。

→ Gap 発見: README の冒頭がユーザーの課題ではなく方法論の定義から始まっている

僕: ...確かに、僕が最初に見たとき、「DGE は 2 つの意味を持つ」って言われても、どっちの意味で使えばいいのか分からなくて...。それで method.md を開いたんですけど、209 行あって...。「原理」「3 ステップ」「会話劇の書き方」「先輩の役割」...全部読まないと使えないんですか？ 全部読まないと使えるのかどうかすら分からないのが問題で...。

→ Gap 発見: method.md に「読まなくていい部分」と「最低限読むべき部分」の区別がない

千石: 問題はもっと根本的です。クイックスタートの Step 2 を見てください。「Claude Code に『DGE して』と言う」。これは手順ではありません。これは祈りです。ユーザーが入力する言葉が「DGE して」の 4 文字で、出力が「会話劇 → gap 抽出」。その間のプロセスが完全にブラックボックスになっている。品質管理の観点からすれば、再現不可能な手順を「クイックスタート」と呼ぶのはユーザーへの侮辱です。

→ Gap 発見: クイックスタートの Step 2 が LLM の挙動に依存しており再現性がない

利根川: 千石の言う通りだ。だがもう一つ致命的なことがある。「Claude Code に『DGE して』と言う」——Claude Code を持っていない奴はどうするんだ？ この toolkit は Claude Code 専用なのか？ ChatGPT のユーザーは？ Cursor のユーザーは？ 対象を明記していない時点で、「誰のためのツールか」が未定義だ。

→ Gap 発見: 対応する LLM / ツールの要件が明記されていない

僕: ...あの、もっと基本的なことなんですけど...。「プロジェクトに dge/ フォルダをコピー」って、自分のプロジェクトの中に入れるんですよね？ でも、DGE って設計レビューの手法ですよね？ なんでソースコードと一緒に管理するんですか...？ .claude/skills/ にコピーするのと dge/ をコピーするのと、両方やるんですか？ 片方だけでいいんですか...？

→ Gap 発見: dge/ フォルダと .claude/skills/ の関係性・それぞれの役割が説明されていない

千石: さらに言えば、クイックスタートには前提条件が書かれていない。Node.js のバージョンは？ Claude Code のバージョンは？ API キーの設定は？ 「コピーする」だけで動くのであれば、それは素晴らしいことです。しかし実際には .claude/skills/ というディレクトリ構造に依存しているはずです。その前提を書かないのは、プロの仕事ではありません。

→ Gap 発見: 前提条件（Prerequisites）セクションが存在しない
→ Spec implication: README に Prerequisites セクションを追加し、必要な環境を明記する

利根川: まとめてやる。初めての開発者が 5 分で体験するのは「混乱」だ。定義を読み、method.md を開き、長さに怯み、テンプレートを眺め、何をすればいいか分からず去っていく。「DGE して」と言える人間は、既に設計力がある人間だ。本当に設計に困っている奴は「DGE」という単語すら知らない。お前たちのターゲットユーザーは「既に強い奴をもっと強くする」のか「弱い奴を助ける」のか、どっちだ？

→ Gap 発見: ターゲットユーザーの設計力レベルが未定義（上級者向け強化 vs 初心者向け支援）

## Scene 2: テンプレート選択

先輩 (ナレーション):
DGE toolkit は 5 つのテンプレートを提供している:
api-design.md, feature-planning.md, go-nogo.md,
incident-review.md, security-review.md。
それぞれに推奨キャラクター組み合わせがある。
ユーザーは自分の課題に合ったテンプレートを選ぶ必要がある。
しかし、テンプレート間の違いや選択基準は明示されていない。

僕: ...テンプレート 5 種類あるんですけど...。僕がやりたいのは「新しい画面の設計レビュー」なんです。API じゃないし、セキュリティでもないし、障害でもない...。feature-planning が近いのかな？ でも「企画」と「設計レビュー」って違いますよね...？ どれを選んでいいか分からないので、全部開いて読み比べるしかない...。5 つ全部読むのは...つらい...。

→ Gap 発見: テンプレート選択のためのディシジョンツリーまたはフローチャートが存在しない

利根川: テンプレートが 5 つしかないのに迷うのは、テンプレートの名前が悪いんじゃない。ユーザーが自分の課題を「DGE のカテゴリ」にマッピングできないんだ。ユーザーの頭の中にあるのは「認証の設計が不安」「この機能、本当に作るべきか迷ってる」「先週の障害を繰り返したくない」——こういう自然言語だ。「api-design.md」「go-nogo.md」——これは作り手の言葉であってユーザーの言葉じゃない。

→ Gap 発見: テンプレート名がユーザーの課題ではなくツール側の分類になっている
→ Spec implication: テンプレート選択画面に「こんなときに使う」ユースケース説明を追加

千石: テンプレートの問題はもう一段深いところにあります。5 つのテンプレートで全てのケースをカバーしていると暗黙に主張していますが、実際にはカバーしていない。データモデル設計は？ パフォーマンスレビューは？ マイグレーション戦略は？ 5 つに収まらないケースが来たとき、ユーザーは何をすればいいのですか。custom-guide.md でオリジナルキャラクターは作れるようですが、オリジナルテンプレートの作り方は？

僕: ...あ、skills/ に dge-template-create.md ってありますね...。でもこれ、skills の中にあるから、まず skills が何かを理解しないと...。テンプレートを作る前にテンプレートの構造を理解しないといけなくて、テンプレートの構造を理解するには method.md を読まないといけなくて...。入れ子になってて出口が見えない...。

→ Gap 発見: テンプレートのカバー範囲とカバー外のケースへの誘導が未定義

利根川: もう一つ。テンプレートと推奨キャラクターの組み合わせ表が catalog.md にあるが、なぜその組み合わせなのかの理由が書いていない。「API 設計: 今泉 + 千石 + 僕」——なぜ利根川がいない？ なぜヤンがいない？ 「この組み合わせがベスト」と言うなら根拠を示せ。根拠なしの推奨は権威主義だ。ユーザーは自分で判断できない。

→ Gap 発見: キャラクター推奨組み合わせの選定理由が未記載

千石: 加えて、キャラクターが 12 人もいる。12 人の中から 3-4 人を選ぶ組み合わせは数百通りです。初心者に「自由に選んでください」と言うのは、自由ではなく放棄です。テンプレートごとに「固定のキャラクター」か「選択式のキャラクター」か、どちらかに決めるべきです。あるいは、最初は選ばせず、慣れてから選ばせる段階的開放が必要です。

→ Spec implication: 初回ユーザー向けにキャラクター選択を省略できるデフォルト構成を用意する

僕: ...段階的開放、いいですね...。最初は「おまかせモード」みたいなのがあると...。でも、それって DGE の思想と矛盾しませんか？ キャラクターの組み合わせで角度が変わるのが DGE の強みだって書いてあるのに、選択肢を隠したら強みが消えるんじゃ...。

利根川: 矛盾してない。初心者が「おまかせ」で 1 回成功体験を得て、「もっと深くやりたい」と思ったときに初めてキャラクター選択を開放する。ゲームのチュートリアルと同じだ。最初から全スキルを解放するゲームは、チュートリアルを放棄したゲームだ。

→ Gap 発見: 初心者向けの段階的オンボーディングパスが設計されていない

## Scene 3: 出力品質と再現性

先輩 (ナレーション):
DGE の出力は LLM が生成する会話劇である。
同じテンプレート、同じキャラクター、同じテーマで実行しても、
LLM のモデル、温度設定、コンテキスト長によって結果が変わる。
method.md は「レビューフロー」を定義しているが、
出力の品質基準や最低要件は定義されていない。
また、実績として「unlaxer-parser で 108 gaps」「AskOS で 16 gaps」と
記載されているが、これらの数字の意味や比較基準も未定義である。

千石: ここが本丸です。DGE は「方法論」を名乗っています。方法論であるなら、再現性がなければなりません。同じ入力に対して、同じ品質の出力が得られること。しかし実態はどうですか。Claude Opus で実行した場合と GPT-4 で実行した場合で、発見される Gap の数も質も変わります。モデルのバージョンアップで結果が変わります。温度パラメータで結果が変わります。これは方法論ですか？ それとも占いですか？

→ Gap 発見: 出力品質の最低基準（Gap 数、カテゴリカバー率等）が定義されていない

利根川: 千石の言い方はキツいが、核心を突いている。さらに言えば、「108 gaps 発見」「16 gaps 発見」——この数字に意味はあるのか？ 108 gaps のうち、実際に仕様に反映されたのは何件だ？ 「Gap を発見した」と「有用な Gap を発見した」は全く違う。ノイズだらけの 108 件と、精鋭の 5 件と、どちらが価値がある？ 効果測定の指標がない状態で実績を語るな。

→ Gap 発見: Gap の有用性・採用率（precision）の測定指標がない

僕: ...そもそも、僕みたいな初心者が DGE を実行して、出てきた Gap が「正しい」のかどうか、どうやって判断するんですか...？ method.md には「人間がレビューする（5-10分）」って書いてあるんですけど、レビューの観点が書いてないんです...。何を見て「この Gap は重要」「この Gap はノイズ」って判断するのか...。レビューの品質がレビューアーに依存してたら、結局 DGE の再現性はレビューアーの能力に帰着するだけじゃ...。

→ Gap 発見: レビューフローにおける判断基準・レビュー観点が未定義
→ Spec implication: Gap レビューのためのチェックリストまたはスコアリング基準を定義する

千石: さらに深刻な問題があります。DGE の出力品質は、入力するテーマの記述精度に大きく依存するはずです。「認証 API の設計を DGE して」と言ったとき、認証 API の仕様書がどの程度詳細に書かれているかで、Gap の深さが変わる。しかし、入力の品質要件がどこにも書かれていない。「テーマを決める」とだけ書いてある。テーマの粒度は？ 事前に必要なドキュメントは？ コンテキストの与え方は？

→ Gap 発見: DGE 実行時の入力品質要件（テーマの粒度、事前ドキュメント、コンテキスト量）が未定義

利根川: もう一つ、実績の話に戻る。unlaxer-parser は学術論文のパーサー設計、AskOS はプロダクト設計。全く異なるドメインだ。「DGE はテーマを問わない」と主張しているが、その根拠がたった 2 つのケースだ。2 例で「汎用性」を語るのは統計学的に無意味だ。しかも両方とも同じ作者が実行している。作者バイアスが排除されていない。

→ Gap 発見: 汎用性の主張を裏付ける事例数と実行者の多様性が不足している

僕: ...あの、もう一つ気になることがあって...。LLM って学習データが更新されるじゃないですか。今日うまくいったプロンプトが来月もうまくいく保証がない...。キャラクターの prompt が catalog.md に書いてあるんですけど、この prompt が特定のモデルに最適化されてるとしたら、モデルが変わったらキャラクターの性格も変わりますよね...。千石が急に妥協し始めたり、利根川が優しくなったり...。

千石: それは由々しき問題です。キャラクターの一貫性はDGE の品質の根幹です。もし LLM のバージョンアップでキャラクターの振る舞いが変わるなら、それは DGE の出力品質が外部要因に支配されていることを意味します。キャラクターの振る舞いに対するテスト、つまり「千石はこの入力に対してこう反応すべき」というアサーションが必要です。

→ Gap 発見: キャラクター prompt のバージョン固有性テスト・品質アサーションが存在しない
→ Spec implication: 各キャラクターに対する振る舞いテストケースを定義する

## Scene 4: ターゲットユーザー

先輩 (ナレーション):
DGE toolkit は「設計の穴を発見する方法論」として位置づけられている。
適用範囲は「パーサー文法設計からビジネス戦略まで」と広く定義されている。
しかし、具体的にどのようなユーザーが、どのような場面で、
どのような動機でこの toolkit を使うのかは明文化されていない。
README には「Claude Code に『DGE して』と言う」と書かれており、
Claude Code ユーザーが暗黙のターゲットとなっている。

利根川: はっきり言う。この toolkit のターゲットユーザーは「DGE の作者本人」だ。自分が使う方法を自分で整理して、それを公開した。それ自体は悪くない。だが「toolkit」と名乗った瞬間、他人が使うことを前提にしたはずだ。他人が使う前提なら、「なぜこれを使うのか」「何が嬉しいのか」を他人の言葉で語れ。「書いてないことの発見」——これは作者の言葉だ。ユーザーの言葉ではない。

→ Gap 発見: ユーザーが DGE を使う動機（ペインポイント）がユーザーの言葉で記述されていない

僕: ...利根川さんの言う通りで、僕が知りたかったのは「これを使うと何が防げるのか」なんです...。仕様漏れでリリース後に障害が起きるのを防げる？ レビューの時間を短縮できる？ 具体的な Before / After が分からないと、導入を上に提案できない...。「108 gaps 発見」って言われても、上司は「で、それでバグが何件減ったの？」って聞いてくるんですよ...。

→ Gap 発見: DGE 導入前後の具体的な効果比較（Before / After）が存在しない

千石: ユーザーの話に戻りましょう。この toolkit を使うユーザーを想定してみます。(1) 個人開発者が自分の設計をレビューしたい、(2) チームリーダーが設計レビュープロセスに組み込みたい、(3) 学術研究者が論文の手法として使いたい。この 3 者では、求めるものが全く違います。個人開発者は手軽さ、チームリーダーは再現性と組織展開、研究者は厳密性と引用可能性。今の toolkit はこの区別がない。

→ Gap 発見: ユーザーペルソナが定義されておらず、異なるユーザー層への対応が未分化

利根川: ペルソナの話が出たから追加する。この toolkit は無料で公開されている。無料のツールに対してユーザーが持つ期待値は低い。「使えなくても文句は言えない」。だが同時に、無料だからこそ「5 分で価値が分からなければ去る」。有料ツールなら「金を払ったから頑張って使おう」とするが、無料は容赦なく捨てられる。初回の 5 分で「おっ」と思わせる体験設計が全くない。

僕: ...5 分で体験ですか...。今の状態だと、5 分では README を読み終わることすらできないですよね...。小規模にしたいんですけど...。例えば、README に「30 秒で試す」セクションを作って、コピペ 1 回で動く最小の DGE を体験させるとか...。

→ Spec implication: README に「30 秒で試す」セクションを追加し、最小の DGE 体験を提供する

千石: 僕くんの提案は正しい方向です。しかし、もう一つ見落としている問題があります。DGE は「Claude Code に『DGE して』と言う」ことで発動しますが、Claude Code を持っていないユーザーは排除されます。ChatGPT や Gemini のユーザーが DGE を使いたい場合、method.md を読んで自分でプロンプトを組み立てるしかない。しかし method.md は 209 行です。これは「Claude Code 専用ツール」なのか「汎用方法論」なのか。

利根川: そこだ。「汎用方法論」を名乗るなら、特定のツールに依存してはいけない。「Claude Code 専用ツール」を名乗るなら、それを明記しろ。どっちつかずが一番ユーザーを混乱させる。

→ Gap 発見: Claude Code 依存と汎用方法論の間のポジショニングが曖昧

僕: ...あと、これは言いにくいんですけど...。DGE のキャラクター、全部日本の漫画やアニメのキャラクターなんですよね...。利根川さんはカイジ、千石さんは王様のレストラン...。海外のユーザーがこれを見たら、キャラクターの背景が分からなくて、性格の意味が理解できないんじゃ...。「利根川って誰？」ってなりますよね...。

千石: これは深刻な指摘です。キャラクターの強みは「既知の人格」を借りることで prompt の精度を上げることにあります。しかし、そのキャラクターを知らないユーザーにとっては、prompt の personality description だけが頼りになる。現在の catalog.md の prompt は十分に詳細でしょうか？ 「現実を見ろ。世間はお前の母親ではない。」——利根川を知らない人がこれを読んで、適切に使えるとは思えません。

→ Gap 発見: キャラクターの personality が原作知識に暗黙に依存しており、知らないユーザーへの説明が不足

利根川: 最後にもう一つ。この toolkit は「meta session」で自分自身を DGE できると言っている。それは面白い。だが、meta session の結果として発見された Gap が、実際に toolkit の改善に反映されるプロセスが定義されていない。Gap を見つけて終わりか？ 誰が直すのか？ いつ直すのか？ この session で見つかった Gap は、いつ反映されるんだ？

→ Gap 発見: meta session で発見された Gap の反映プロセス（誰が・いつ・どう反映するか）が未定義
→ Spec implication: meta session の Gap を issue 化し tracker で管理するワークフローを定義する
