# DGE Session: 「DGEは本当に必要か？」— 方法論の根本検証

> DGE 自身を DGE で検証する。方法論が自己言及に耐えられなければ、他の何にも使えない。

---

## Scene 1: 基本検証 — 「そもそも会話劇じゃなきゃダメなんですか？」

先輩（ナレーション）:
DGE は「キャラクターが議論する会話劇を生成し、仕様書にない前提・暗黙の制約・考慮漏れを表面化させる」手法である。unlaxer-parser では 5 sessions で 108 gaps を発見し、AskOS では 11+ sessions で 14,978 行の設計ドキュメントを生成した。しかし、そもそもこのアプローチ自体が最善である保証はどこにもない。DGE の存在意義を根本から問う。

今泉: 「そもそも、なんで会話劇なんですか？ 普通にチェックリストを作って上から順に確認していけば、同じ gap が見つかるんじゃないですか？」

→ Gap 発見: DGE がチェックリストベースのレビューより優れている根拠が文書化されていない

ヤン: 「まあ、チェックリストで済むならそのほうがいいですよね。会話劇を書く 10〜30 分より、チェックリストを 5 分で回すほうが効率的です。紅茶ください。」

ハウス: 「お前たちは嘘をついている。チェックリストが有効なのは『何を確認すべきか事前に分かっている場合』だけだ。DGE が見つけると主張しているのは『書いてないこと』だ。チェックリストに書いてないことを、チェックリストでどうやって見つけるんだ？」

今泉: 「あ、でもそれって、要するにブレインストーミングと同じじゃないですか？ 3 人で集まって自由に議論すれば、キャラクターなんて要らないんじゃ......」

→ Gap 発見: DGE とブレインストーミング/フリーディスカッションの差分が未定義

ヤン: 「ブレインストーミングとの違い...... ひとつ言えるのは、ブレインストーミングは参加者の専門性に依存するということですかね。今泉さんが 3 人集めてセキュリティの議論をしても、3 人ともセキュリティの素人なら何も出てこない。キャラクターは専門性をシミュレートできる。」

ハウス: 「だがそれは LLM がシミュレートした『偽の専門性』だ。本物のセキュリティ専門家を 1 人連れてくるほうが、LLM が演じる Red Team より 100 倍マシだろう。」

→ Gap 発見: LLM がキャラクターを演じる際の専門性の限界が未定義（どこまで信頼できるか）

今泉: 「誰が困るんですか、これ？ DGE を使う人は、本物の専門家を連れてこれない人ですよね。1 人で設計してる個人開発者とか、チームが小さいスタートアップとか。そういう人にとっては、偽の専門性でもないよりマシ......ってことですか？」

ヤン: 「つまり DGE のポジショニングは『本物の専門家レビューの代替品』なのか、それとも『専門家レビューの補完ツール』なのか。そこが曖昧ですね。紅茶もう一杯。」

→ Spec implication: DGE のポジショニング（代替 vs 補完）を明確に定義する必要がある

ハウス: 「もうひとつ。method.md には『正解を出さない — キャラクターは問題を提起するだけ。解決は human。』と書いてある。つまり DGE は gap を『発見する』だけで『解決しない』。だが gap の発見と解決の間には深い谷がある。108 gaps 見つけて、何個解決された？ その数字がないのは、全員が見て見ぬふりをしているからだ。」

→ Gap 発見: DGE で発見された gap の解決率（resolution rate）が追跡されていない

今泉: 「前もそうだったっけ...... 他のレビュー手法で『発見した問題の何パーセントが実際に修正されたか』って追跡してましたか？」

ハウス: 「してないだろう。だからこそ DGE が『108 gaps 発見しました』と誇るなら、その後どうなったかも追跡すべきだ。発見だけして放置するなら、それは gap のゴミ屋敷だ。Vicodin くれ。」

## Scene 2: 代替手法との比較 — 「他にないの？」

先輩（ナレーション）:
仕様の gap を発見する手法は DGE だけではない。形式手法（TLA+, Alloy）、Design Review、Threat Modeling（STRIDE）、ユーザビリティテスト、Dog fooding、ADR (Architecture Decision Records) など、多くの既存手法がある。DGE がこれらと比較してどのような位置づけにあるのかを検証する。

ヤン: 「形式手法は数学的に正しさを証明できる。STRIDE は脅威を体系的に分類できる。ADR は意思決定の記録を残せる。DGE は...... キャラクターが喋る。正直、学術的な信頼性では勝てないですよね。」

今泉: 「そもそも、これらの手法と DGE は競合するんですか？ それとも補完するんですか？ method.md にはそのマッピングがないですよね。」

→ Gap 発見: DGE と既存のレビュー手法（形式手法、STRIDE、ADR 等）の関係性マッピングが未定義

ハウス: 「問題はもっと深い。形式手法は『数学的に検証可能な性質』を扱う。DGE は『人間の直感が引っかかるもの』を扱う。この 2 つのカバレッジは重なるのか？ 重ならないのか？ 誰も検証していない。」

ヤン: 「形式手法で見つかる gap と DGE で見つかる gap の Venn diagram があれば一発で分かるんですが。unlaxer-parser の 108 gaps のうち、形式手法でも見つかったはずのものは何個ですか？」

→ Gap 発見: DGE の発見 gap と他手法の発見 gap の重複分析が行われていない

今泉: 「他にないの？ って聞きたいんですけど、DGE の代わりに使えるもっと簡単な手法って本当にないんですか？ 例えば...... プロンプトエンジニアリングでストレートに『この仕様の穴を 10 個見つけて』って LLM に聞いたら、会話劇なしでも gap は出てくるんじゃないですか？」

→ Gap 発見: 「キャラクターなし単純プロンプト」と「DGE 会話劇」の出力品質比較が未検証

ヤン: 「実は僕もそれ思ってました。Claude に『この API 仕様のレビューをして。抜け漏れを列挙して。』と頼むだけで、キャラクターを使わなくても相当数の gap が出てくる可能性はある。会話劇のオーバーヘッドを正当化できるだけの差分がないと、DGE はただの過剰な演出ですよ。」

ハウス: 「だが単純なプロンプトには perspective の固定という問題がある。LLM は 1 つのペルソナで回答する。ペルソナが変わると注目するポイントが変わる。セキュリティの視点で見るのと、UX の視点で見るのでは、同じ仕様から異なる gap が出てくるはずだ。...... はずだ、と言ったのは、これも検証されていないからだ。」

→ Spec implication: 「キャラクターによる視点の強制切り替え」が gap 発見数/質に与える影響の定量評価が必要

今泉: 「誰が困るんですか？ DGE を使うか単純プロンプトを使うか迷っている人は、判断基準がないと選べないですよね。『こういう場面では DGE が優位、こういう場面では単純プロンプトで十分』というガイドラインがほしいです。」

ハウス: 「本音を言おうか。DGE が本当に優れているかどうかは、比較実験をしない限り永遠に分からない。同じ仕様を（A）チェックリスト、（B）単純プロンプト、（C）DGE で検証して、発見された gap の数・質・ユニーク率を比較する。これをやらない限り、DGE の優位性は信仰だ。Vicodin くれ。」

→ Gap 発見: DGE の有効性を示す比較実験（controlled experiment）のデザインが存在しない

## Scene 3: 複雑さの検証 — 「キャラ 12 人にテンプレ 5 種、本当に要る？」

先輩（ナレーション）:
DGE toolkit は 12 キャラクター、5 テンプレート、先輩ロール、6+5 の gap カテゴリ、Observe → Suggest → Act パターン、今泉メソッド 5 Types、4 幕構成を含む。これは方法論として「必要十分」なのか「過剰に複雑」なのかを検証する。

ヤン: 「12 キャラクターですか。正直に言っていいですか？ 僕なら 3 人で十分です。『前提を問う人』『シンプルにする人』『隠れた問題を見つける人』。つまり今泉、僕、ハウス。この 3 人でほとんどのケースをカバーできる。紅茶ください。」

今泉: 「そもそも、12 キャラ全員が実際に使われているんですか？ 利用統計ってあるんですか？ もしかして、よく使うキャラは 3〜4 人で、残りはほとんど使われてないんじゃ......」

→ Gap 発見: キャラクターごとの使用頻度データが収集されていない

ハウス: 「お前たちはまだ表面しか見ていない。問題は人数じゃない。各キャラクターの発見する gap に有意な差があるかどうかだ。今泉が見つける gap とヤンが見つける gap が 80% 重複しているなら、片方は要らない。逆に、各キャラクターがユニークな gap を安定して発見できるなら、12 人でも足りないかもしれない。」

→ Gap 発見: キャラクター間の gap 発見の重複率・ユニーク率が未計測

ヤン: 「テンプレートも 5 種類ありますが...... api-design, feature-planning, go-nogo, incident-review, security-review。これ、テンプレートの差分って scene のタイトルが違うだけじゃないですか？ 構造は全部『先輩が背景説明 → キャラが議論 → gap 発見』の 4 幕構成。テンプレートなしで『テーマ: 認証 API のセキュリティ。キャラ: 千石, Red Team, ハウス』と指定すれば、同じ結果が出るのでは？」

→ Gap 発見: テンプレートの付加価値（テンプレートなしの自由形式との品質差）が未検証

今泉: 「要するに、DGE toolkit の複雑さは『使いやすさ』のためなんですか、それとも『正確さ』のためなんですか？ テンプレートがあると初心者が始めやすいけど、上級者には邪魔になる...... みたいな話ですか？」

ハウス: 「違う。本当の問題は、この toolkit が『方法論の説明』と『ツールの提供』を混同していることだ。method.md は方法論を説明する。templates/ はツールを提供する。characters/ はデータを提供する。これらのレイヤーが分離されていない。ユーザーは method.md を読めば DGE を理解できるのか？ それとも templates/ も読まないと使えないのか？ 学習パスが不明確だ。」

→ Gap 発見: DGE toolkit のユーザー学習パス（onboarding journey）が設計されていない

今泉: 「前もそうだったっけ...... 複雑な方法論を作って、結局ドキュメントが複雑すぎて誰も読まなかった、みたいな経験ないですか？」

ヤン: 「あと、gap カテゴリが 11 個あるのも気になります。Missing logic, Spec-impl mismatch, Type/coercion gap, Error quality, Integration gap, Test coverage, Business gap, Safety gap, Ops gap, Message gap, Legal gap。11 カテゴリ。これ、session 中にリアルタイムでカテゴリ分けなんてできます？ 僕なら High / Medium / Low の 3 分類で十分ですよ。」

→ Spec implication: Gap カテゴリの粒度を「session 中のリアルタイム分類」に適したレベルに再設計すべき

ハウス: 「さらに言えば、Observe → Suggest → Act パターンも本当に毎回必要か疑問だ。session 中は Observe（発見）だけで十分で、Suggest と Act は session 後にまとめてやればいい。session 中に 3 段階を全部やらせると、議論の流れが途切れる。」

今泉: 「誰が困るんですか？ session を書く AI が困る？ それとも、session を読んでレビューする人間が困る？」

ハウス: 「両方だ。書く側は形式に縛られて自然な議論ができない。読む側は形式的なマーキングが多すぎて本質を見失う。DGE は『形式の奴隷』になるリスクがある。Vicodin くれ。」

→ Gap 発見: Session 中の gap マーキング量が議論の質に与える負の影響が考慮されていない

## Scene 4: 実証の検証 — 「108 gaps は本物か？」

先輩（ナレーション）:
DGE の実績として「unlaxer-parser で 108 gaps」「AskOS で 16 gaps」が挙げられている。この数字の信頼性、gap の質、そして DGE がなければ発見できなかったかどうかを検証する。これは DGE の存在意義に直結する最も重要な問いである。

ハウス: 「108 gaps。印象的な数字だ。だが数は質を保証しない。108 のうち、severity が Critical だったのは何個だ？ trivial な gap を 100 個見つけるより、Critical な gap を 3 個見つけるほうが価値がある。severity の分布が公開されていない。」

→ Gap 発見: 過去の DGE session で発見された gap の severity 分布が分析・公開されていない

今泉: 「そもそも、severity はどうやって決めるんですか？ method.md には severity の定義がないですよね。Critical, High, Medium, Low の基準が書いてない。」

→ Gap 発見: Gap severity の定義（Critical / High / Medium / Low の判定基準）が未定義

ヤン: 「あと、108 gaps という数字自体が怪しいですね。Gap の粒度がバラバラだったら、1 つの gap を 3 つに分割して水増しすることもできる。逆に、5 つの関連する gap を 1 つにまとめて過少報告することもできる。gap の粒度の基準がない。」

→ Gap 発見: Gap の粒度基準（1 つの gap の範囲の定義）が未定義

今泉: 「他にないの？ gap を数える方法。例えば、2 人の独立したレビュアーが同じ session を読んで gap を数えたら、同じ数になりますか？ Inter-rater reliability みたいな......」

ハウス: 「ならないだろう。Gap の識別は主観的だ。ある人が『これは gap だ』と思うものを、別の人は『これは仕様の意図的な省略だ』と解釈するかもしれない。DGE の gap 数は再現性がない。再現性のない数字に基づいて方法論の有効性を主張するのは、科学ではなく宗教だ。」

→ Spec implication: Gap 識別の inter-rater reliability を測定するプロトコルが必要

ヤン: 「もっと根本的な問いがあります。DGE がなくても、同じ gap が通常の開発プロセスで発見されていた可能性はないですか？ テストを書く段階で気づく、コードレビューで指摘される、QA で発見される...... DGE で前倒しして見つけたのか、どうせ後で見つかったのかが分からない。」

今泉: 「要するに、DGE の本当の価値は『gap を見つけること』じゃなくて『gap を早く見つけること』なんですか？」

ヤン: 「そう。もしそうなら、DGE の価値は『発見の前倒し効果』で測るべきです。設計段階で見つけた gap を、もし実装後に見つけていたらどれだけコストがかかったか。その差分が DGE の ROI です。」

→ Gap 発見: DGE の ROI 測定フレームワーク（発見の前倒し効果の定量化）が存在しない

ハウス: 「最後に、最も不都合な真実を言おう。DGE は LLM に会話劇を生成させる。LLM は学習データに含まれる既知のパターンに基づいて gap を『発見する』。つまり DGE が見つける gap は、原理的に『誰かが過去に経験した gap』の再構成でしかない。本当に novel な gap — 誰も経験したことのない、このプロジェクト固有の gap — を DGE は発見できるのか？」

今泉: 「それって...... そもそも、他のどんなレビュー手法でも同じじゃないですか？ 人間のレビュアーだって過去の経験に基づいてレビューするんだから。」

ハウス: 「違う。人間は文脈を持っている。このプロジェクトのドメイン知識、チームのスキルセット、ユーザーの特性。LLM はそれを session ごとに context として渡されるだけだ。渡し忘れた context に関する gap は、原理的に発見できない。DGE の限界は context の完全性に依存する。だが context が完全なら、DGE なしでも gap は見えているはずだ。矛盾だ。Vicodin くれ。」

→ Gap 発見: DGE の理論的限界（LLM の context 依存性と novel gap 発見能力の限界）が文書化されていない

### Observe → Suggest → Act

**G-META-001** DGE vs チェックリストの根拠不在
- Observe: DGE がチェックリストより優れている理由が「書いてないことの発見」という主張のみで、具体的な根拠やデータがない
- Suggest: method.md に「なぜチェックリストでは不十分か」のセクションを追加し、具体例で説明する
- Act: method.md に「## DGE vs チェックリスト」セクションを追加。チェックリストが有効な領域（既知の確認項目）と DGE が有効な領域（未知の探索）を明示する

**G-META-002** ブレインストーミングとの差分未定義
- Observe: DGE とフリーディスカッションの違いが明確でない。キャラクターによる視点の強制切り替えが本質的な差分である可能性はあるが、未検証
- Suggest: 「DGE はフリーディスカッションと何が違うのか」を FAQ として method.md に追加する
- Act: method.md に「## FAQ」セクションを追加。「キャラクターが視点を固定することで、フリーディスカッションでは到達しにくい角度から gap を発見できる」を仮説として明示し、検証計画を記載する

**G-META-003** LLM キャラクターの専門性限界
- Observe: LLM がセキュリティ専門家や法務専門家をシミュレートする際の信頼性上限が不明
- Suggest: DGE の README に「DGE は専門家レビューの代替ではなく、補完である」と明記する
- Act: README.md の冒頭に「## 注意事項」を追加。「DGE は仕様の穴の早期発見を支援するが、セキュリティ監査・法務レビュー等の専門家レビューを代替するものではない」を明記する

**G-META-004** Gap 解決率の未追跡
- Observe: 108 gaps を発見したが、何個が実際に解決されたかの追跡がない
