# DGE 演劇技法カタログ — Theatrical Techniques for Dialogue-driven Gap Extraction

> **目的**: 演劇・脚本術から技法を体系的に棚卸しし、DGE の会話劇品質向上と Gap 発見力強化の両方に使えるものを選定する。  
> **handoff 先**: Claude Code（method.md / patterns.md / キャラファイルへの反映）  
> **日付**: 2026-04-07

---

## 0. 現状の課題

今の DGE 会話劇は「討論会フォーマット」に固定されている:

- キャラが axis に沿って指摘 → 別キャラが応答 → Gap マーカー
- シーン間の変化が乏しい（同じ会議室で同じトーンが続く）
- キャラの人格は prompt で定義されるが、**関係性・過去・内面**が会話に表出しない
- 「読みたくなる力」（＝spec レビュー以外の動機で読まれる力）が弱い

**仮説**: 演劇技法の導入は「面白さ」だけでなく、**従来のパターンでは出てこない角度の Gap** を発見する力にもなる。なぜなら、シチュエーション・時間軸・人間関係を変えることで、同じシステムの異なる断面が露出するから。

## 1. 構造系技法（Structural Techniques）

### 1.1 三幕構造 / エピソード構造

| 技法 | 演劇での意味 | DGE への適用 | Gap 発見への寄与 |
|---|---|---|---|
| **三幕構造** | 提示→展開→解決 | 1 Session を「問題提起 Scene → 深掘り Scene → 収束 Scene」に構造化 | 収束フェーズで「結局どうする？」が強制され、曖昧な結論が Gap として浮上 |
| **エピソード構造**（Brecht） | 各シーンが独立して成立 | 各 Scene を独立した状況設定で構成（今と同じだが意図的に） | シーン間の断絶が「このシーンの結論はあっちのシーンと矛盾してないか？」を問う |
| **フレーム物語**（劇中劇） | 物語の中に物語がある | キャラが「こういうユーザーがいたとして」とミニシナリオを演じる | ユーザー視点のウォークスルーが自然に発生し、UX Gap が出る |
| **時間軸操作** | フラッシュバック/フラッシュフォワード | 「半年後、このコードを引き継いだ新人は…」「導入前の運用はこうだった」 | 保守性・移行パス・オンボーディングの Gap が時間差で露出 |

### 1.2 シーン設計の原則

| 原則 | 説明 | DGE への適用 |
|---|---|---|
| **各シーンに dramaturgical purpose** | plot 前進 / character 開示 / 方向転換のいずれか | 各 Scene が「新しい Gap を出す」「既出 Gap を深掘る」「前提を覆す」のいずれかを担う |
| **Late arrival, early exit** | シーンの途中から始まり、結論の前に切る | 議論が煮詰まった状態から Scene を開始。読者を引き込みつつ、前提説明を省略 |
| **因果律 > 時系列** | 「だから次にこうなった」が「次にこれが起きた」より重要 | Scene 間の接続を因果で設計。「Scene 1 で JWT の議論をしたから Scene 2 でセッション管理の矛盾が出る」 |

## 2. キャラクター系技法（Character & Dialogue Techniques）

### 2.1 キャラクターの奥行き

| 技法 | 演劇での意味 | DGE への適用 | 実装案 |
|---|---|---|---|
| **バックストーリー開示** | 対話を通じて過去が自然に漏れる | キャラが指摘の中で過去のプロジェクト経験を匂わせる | キャラファイルに `backstory:` フィールド追加（2-3 行のエピソード） |
| **キャラ間の共有史** | 登場人物同士に既知の関係がある | 「ヤンさん、あなたまた『削れ』から入るんですか」的なやりとり | キャラファイルに `history_with:` フィールド追加 |
| **弱点の発動** | weakness が実際にシーン内で表出する | 深澤の理想論暴走を林が「比喩じゃなくて妄想です」と切る | method.md に「weakness は 1 Session 1 回以上発動させる」ルール |
| **内的葛藤 × 外的障害** | 外の問題が内面の弱さと衝突 | ヤンが「シンプルに保て」と主張するが、要件の複雑さに自分の原則が揺らぐ | axis と theme の衝突を意図的に設計 |
| **Character arc（変化）** | セッション内でキャラの立場が変わる | 最初は反対だったキャラが議論を通じて賛成に転じる（またはその逆） | 「少なくとも 1 キャラは Session 内で立場が変わる」ルール |

### 2.2 対話技法

| 技法 | 演劇での意味 | DGE への適用 | Gap 発見への寄与 |
|---|---|---|---|
| **サブテキスト** | 言葉の裏に本音がある | 「いいんじゃないですか、それで」（＝全然よくない）→ 別キャラが見抜く | 表面的合意の裏にある懸念が Gap として顕在化 |
| **On-the-nose の回避** | キャラが感情や意図を直接言わない | 「この設計はダメです」→「この設計で、夜中に呼ばれるのは誰ですか？」 | 間接的な問いかけが、直接指摘では出ない角度の Gap を引き出す |
| **割り込み・遮り** | 文が完結しない | 「でもそれって——」「待って、その前に」 | 議論の衝突点が明確になり、優先順位の Gap が浮上 |
| **沈黙・間（Beat）** | 意味のある沈黙 | `（……全員が黙る）` → 直前の指摘の重大さを強調 | 読者（spec レビュアー）の注意を最重要 Gap に誘導 |
| **コールバック（伏線回収）** | 序盤の何気ない一言が終盤で回収 | Scene 1 の雑談が Scene 4 の Gap 発見トリガーになる | 一見無関係な論点間の依存関係が発見される |
| **反復と変奏（Leitmotif）** | 同じフレーズが文脈で意味を変える | 「シンプルに保とう」が最初は前向き → 終盤で思考停止の言い訳に反転 | 設計原則の適用限界が自然に炙り出される |
| **独白・傍白（Aside）** | 他キャラに聞こえない心内語 | `（今泉、心の中で：本当にこれでいいのか…？）` | 集団合意への違和感が可視化され、同調圧力による Gap 見逃しを防ぐ |

### 2.3 キャラクターの声（Voice）

| 技法 | 説明 | DGE への適用 |
|---|---|---|
| **Character fingerprint** | 名前を隠しても誰の台詞かわかる | キャラファイルに `speech_pattern:` を追加。口癖、語尾、比喩の傾向 |
| **レジスター変化** | 相手や状況で話し方が変わる | 同じキャラが上司向け vs 同僚向けで説明を変える → 説明の一貫性 Gap |
| **専門用語 vs 平易語** | 知識レベルの差が対話に出る | 林が専門用語を比喩に変換する過程で、概念の曖昧さが露出 |

## 3. ブレヒト系技法（Epic Theatre / Verfremdungseffekt）

DGE にとって特に価値が高い。**「考えさせる」が DGE の本質目的と完全に一致**するから。

| 技法 | ブレヒトでの意味 | DGE への適用 | Gap 発見への寄与 |
|---|---|---|---|
| **異化効果（V-effekt）** | 見慣れたものを異化して批判的に見せる | 「当たり前」とされている設計判断を異化。「なぜ REST？ なぜ JSON？」 | 暗黙の前提が強制的に問い直される |
| **第四の壁を破る** | キャラが観客に直接語りかける | キャラが読者（＝開発者）に直接問いかける。「あなたならこのエラーをどう直しますか？」 | 読者を巻き込み、受動的な spec 消費を防ぐ |
| **ゲストゥス（Gestus）** | 身振りが社会的態度を表す | ト書きでキャラの態度を社会的文脈で描写。「*腕を組み、マネージャーの顔をする*」 | 技術判断の裏にある組織的・政治的動機が可視化 |
| **プラカード・ナレーション** | シーン前に要約を掲示 | Scene 冒頭に「このシーンでは認証設計の前提が崩壊する」と予告 | 予告と実際の差分が「予想外の Gap」として際立つ |
| **歴史化（Historicisation）** | 出来事を歴史的文脈に置く | 「2020 年にこのアーキテクチャで炎上した事例がある」とキャラが言及 | 同じ設計の過去の失敗パターンが参照され、リスク評価精度が上がる |
| **エピソード独立性** | 各シーンが独立して機能 | 既に patterns.md の構造に近い。さらに意図的に |  |

## 4. シチュエーション系技法（Setting & Situation）

| 技法 | 説明 | DGE への適用 | Gap 発見への寄与 |
|---|---|---|---|
| **場面転換** | 時間・場所・状況を変える | 「—— 翌日、コードレビューの席で ——」「—— 障害対応 Slack、深夜 2:37 ——」 | 同じ論点が異なるコンテキストで検討され、環境依存の Gap が露出 |
| **圧力状況（Pressure Cooker）** | 締め切り・危機的状況 | 「本番デプロイ 30 分前」というシチュエーションで議論 | 時間圧力下での判断品質・ロールバック手順の Gap |
| **異文化接触** | 異なる背景の人が出会う | フロントエンド開発者とインフラエンジニアの対話 | 専門領域間の認識ギャップ、用語の不一致 |
| **権力勾配** | 上下関係が会話に影響 | CTO vs ジュニアエンジニアの会話。ジュニアが言いたいことを言えない | 組織内の情報非対称、心理的安全性の欠如による意思決定品質低下 |
| **傍観者の導入** | 議論を横で聞いている第三者 | 営業が技術議論を横で聞いて「お客さんにはなんて説明すればいいの？」 | ステークホルダー間の説明 Gap、ドキュメント不足 |

## 5. メタ演劇系技法（Meta-theatrical）

| 技法 | 説明 | DGE への適用 | 備考 |
|---|---|---|---|
| **劇中劇（Play within a Play）** | 劇の中でキャラが別の劇を演じる | 今泉がエンドユーザーを演じて操作手順を辿る | 既存パターン B1（zero-state）の拡張。ロールプレイ型 |
| **メタ言及** | 劇が自分自身について言及する | 「この DGE セッション自体が、仕様の曖昧さを証明してますね」 | DGE プロセス自体のフィードバックループ |
| **信頼できない語り手** | ナレーターの信頼性が疑わしい | 先輩ナレーションが「順調に見えたが、実は…」と後から覆す | 楽観バイアスの排除。最初の合意が覆る構造 |
| **多視点語り** | 同じ出来事を複数の視点で語る | 同じ API 設計を「呼ぶ側」「呼ばれる側」「運用する側」で順に語る | 既存パターン A2（role-contrast）の強化版 |

## 6. 出力フォーマット拡張

現在の DGE 出力は「会話劇」一択だが、script/台本系のフォーマットを選択可能にする。

### 6.1 フォーマット候補

| フォーマット | 特徴 | 向いているケース |
|---|---|---|
| **座談会（現行）** | ナレーション + キャラ対話 + Gap マーカー | 汎用。設計レビュー全般 |
| **台本（Screenplay）** | シーン見出し + ト書き + 台詞。場面転換を明示 | シチュエーション変化が重要なセッション |
| **ずんだもん/霊夢型** | 短いやりとりの連続。ボケとツッコミ。「〜なのだ」「ゆっくりしていってね」 | カジュアルな解説。新人オンボーディング。SNS 共有向け |
| **法廷劇型** | 証人尋問形式。質問→回答→異議→裁定 | 設計判断の正当性を厳密に検証するケース |
| **インシデント報告型** | タイムライン + 証言 + 根本原因分析 | postmortem / investigation 構造との統合 |
| **書簡/チャットログ型** | メール往復や Slack ログとして描写 | 非同期コミュニケーションの Gap 発見 |

### 6.2 フォーマット切り替えのトリガー

```
output_format:
  default: roundtable       # 現行
  screenplay: true          # 「台本形式で」
  zundamon: true            # 「ずんだもん形式で」「ゆっくり形式で」
  courtroom: true           # 「法廷形式で」「尋問して」
  incident: true            # investigation 構造で自動選択
  epistolary: true          # 「メール形式で」「Slack 形式で」
```

## 7. 技法 × 既存パターンのマッピング

既存の 23 パターン（A1-C8）と演劇技法の組み合わせで、Gap 発見力が増幅されるペアリング:

| 既存パターン | 相性の良い演劇技法 | 理由 |
|---|---|---|
| A1 `before-after` | 時間軸操作（フラッシュバック） | before/after を「回想」として演出 |
| A2 `role-contrast` | 多視点語り + 権力勾配 | ロール間の認識差を権力関係で増幅 |
| A4 `expertise-contrast` | ずんだもん/霊夢型 + レジスター変化 | 初心者キャラの素朴な質問が本質を突く |
| B1 `zero-state` | 劇中劇（ユーザー RP） | キャラが実際に新規ユーザーを演じる |
| B2 `return-after-absence` | 場面転換（時間跳躍） | 「3 ヶ月後に戻ってきた」をシーンとして描写 |
| B3 `escalation-chain` | 圧力状況 + 台本形式 | 障害発生→エスカ→解決の時系列を台本で |
| B4 `cross-persona-conflict` | サブテキスト + 内的葛藤 | 表面的合意の裏の不満が爆発 |
| B6 `multi-tenant` | 異化効果 | 「なぜテナント分離が必要か」を根本から問い直す |
| C1 `scale-break` | 圧力状況 + 傍観者 | 大量リクエスト下で営業が「お客さん怒ってるんだけど」 |
| C5 `security-adversary` | 法廷劇型 + ゲストゥス | 攻撃者と防御者の立場を法廷形式で |
| C7 `disaster-recovery` | インシデント報告型 + 歴史化 | 過去の障害事例を参照した復旧訓練 |

## 8. 実装計画 — どこに何を入れるか

### 8.1 method.md への追加

```markdown
## 演出指示（Dramatic Direction）

1 Session につき以下の演出技法を**最低 1 つ**使うこと:
- シチュエーション変化（場面転換・時間跳躍・圧力状況）
- キャラクター深化（バックストーリー開示・弱点発動・立場変化）
- 対話技法（サブテキスト・コールバック・沈黙）
- メタ技法（劇中劇・第四の壁・異化効果）

### 演出技法と Gap の関係
演出技法は「面白さ」のためだけに使わない。
各技法は特定の種類の Gap を露出させる道具である。
技法選択は theme と pattern の組み合わせから自動推奨する。
```

### 8.2 patterns.md への追加 — Category D: 演出パターン

```markdown
### Category D: 演出パターン（6）
シーンの構成・対話の質を変えて、通常の議論では出ない Gap を引き出す。

| ID | パターン | 発見するもの |
|---|---|---|
| D1 | `backstory-reveal` バックストーリー開示 | 過去の類似判断の失敗、経験則の限界 |
| D2 | `setting-change` 場面転換 | 環境依存の前提、運用コンテキストの欠如 |
| D3 | `play-within-play` 劇中劇 | UX ウォークスルーの欠落、ユーザー視点の不在 |
| D4 | `temporal-shift` 時間跳躍 | 保守性、移行パス、技術的負債の蓄積 |
| D5 | `subtext-conflict` サブテキスト衝突 | 表面的合意の裏の未解決リスク |
| D6 | `pressure-cooker` 圧力状況 | 障害時の判断手順、ロールバック、SLO 違反 |
```

### 8.3 キャラファイルへの追加フィールド

```yaml
# 既存フィールドに追加
backstory: |
  前職で JWT の有効期限を 1 年に設定し、トークン漏洩で 3 日間の障害対応を経験。
  以来「セキュリティは最初から」が口癖になった。
history_with:
  imaizumi: 「去年のプロジェクトで今泉の『そもそも』に 2 週間止められた」
  fukasawa: 「深澤とは UX vs 削減でいつも対立するが、飲みに行く仲」
speech_pattern: |
  語尾: 「〜だろう」「〜じゃないか」
  口癖: 「削れ」「本当に要るのか」
  比喩傾向: 軍事・戦略系（「退路を断つ」「前線が伸びすぎだ」）
```

### 8.4 dge-session.md Step 5 への追加

```markdown
### 演出の適用
- Scene ごとにシチュエーション（時間・場所・状況）を変えてよい
- Scene 冒頭のナレーションで場面設定を行う（ト書き）
- キャラの weakness は 1 Session に最低 1 回発動させる
- 最低 1 キャラは Session 内で立場（賛成/反対）が変わる
- output_format が指定されていれば、そのフォーマットに従う
```

  output_format: roundtable # roundtable | screenplay | zundamon | courtroom | incident | epistolary
  weakness_activation: true

## 9. 評価基準 — 「面白くなったか」をどう測るか

| 指標 | 測定方法 | 目標 |
|---|---|---|
| Gap 発見数 | 同一テーマで旧方式 vs 新方式を比較 | +20% 以上 |
| Gap の多様性 | Category (A/B/C) の分布 | 偏りが減る |
| 可読性 | 「途中で読むのをやめたくなったか」の主観評価 | 最後まで読める |
| キャラ識別性 | 名前を隠して誰の発言か当てられるか | 80% 以上正答 |
| 再利用性 | Session ファイルが他の開発者にも読まれるか | チーム内共有率 |

## 10. 優先順位と着手順序

Phase 1（即効性が高い・実装が軽い）:
1. **method.md に演出指示セクション追加** — ルールを 1 つ足すだけ
2. **キャラファイルに `backstory:` / `history_with:` 追加** — 2-3 キャラから
3. **patterns.md に Category D 追加** — D2（場面転換）と D3（劇中劇）から

Phase 2（フォーマット拡張）:
4. **output_format 選択機能** — flow YAML + dge-session.md の対応
5. **ずんだもん/霊夢型テンプレート作成** — 軽量解説向け
6. **台本（Screenplay）型テンプレート作成** — ト書き・場面見出し付き

Phase 3（高度な技法）:
7. **ブレヒト系技法の統合** — 異化効果・第四の壁
8. **キャラ間関係グラフの設計** — DVE データモデルとの統合
9. **自動演出推奨エンジン** — theme × pattern → 推奨技法の自動選択
