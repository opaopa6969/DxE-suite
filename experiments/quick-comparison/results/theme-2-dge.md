# Theme 2: 課金モデル — DGE レビュー

- **キャラクター**: 今泉 + 大和田 + 利根川 + 僕 + ソクラテス
- **パターン**: monetization (role-contrast, scale-break, cross-persona-conflict)
- **テンプレート**: go-nogo

---

## Scene 1: Happy Path — 「誰が払うのか」

**先輩**: B2B SaaS の課金モデル。Free / Pro ($10/user/月) / Enterprise（要問合せ）の 3 プラン。Stripe 使用。

**🦈 大和田**: 「$10/user/月か。10 人のチームで $100/月。で、この $100 のうち Stripe の手数料が 2.9% + 30¢。実質 $96.80。サーバー代、開発コスト、サポートコストを引いて利益はいくら残る？ LTV は計算したか？ CAC は？ — ちなみに Free ユーザーのコストは誰が払ってるんだ？ 1GB のストレージ × Free ユーザー数。Free が 1 万人いたら 10TB。AWS の S3 料金知ってるか？」

→ Gap 発見: Free プランのコスト構造が未検討。Free ユーザーが増えるほど赤字になる仕組みになっている

**🎰 利根川**: 「現実を見ろ。"Pro: 全機能" と書いてある。全機能？ じゃあ Enterprise を買う理由は何だ？ SSO と監査ログだけで "要問合せ" の価格を払うか？ — ユーザーの言葉で語れ。CTO が CFO に "この SaaS に月 $5000 払いたい、理由は SSO" と言って通ると思うか？ Enterprise の価値提案が弱すぎる。」

→ Gap 発見: Pro と Enterprise の差別化が SSO + 監査ログだけで、Enterprise の価値提案が不明確

**👤 今泉**: 「そもそも、$10/user/月の根拠は何ですか？ 競合はいくらですか？ Asana は $10.99、Monday.com は $9。同じ価格帯にいるのは偶然ですか意図的ですか？ 価格の根拠が "競合と同じくらい" だけだと、競合が値下げしたら追随するしかなくなりません？」

→ Gap 発見: 価格設定の根拠（コストベース？ 価値ベース？ 競合ベース？）が未記載

## Scene 2: エッジケース — 「金の問題」

**😰 僕**: 「...あの、"Pro: メンバー無制限" って怖くないですか...？ 1 チームが 10,000 人招待したらどうするんですか...？ $100,000/月 の請求書が届くんですか...？ ユーザーが "間違えて全社員を招待してしまった" って問い合わせてきたら...？ 返金するんですか...？ しないんですか...？」

→ Gap 発見: 大量招待の事故防止策がない。返金ポリシーが未定義

**🎭 ソクラテス**: 「"Free: 3 人まで" — なぜ 3 人か？ 2 人ではダメか？ 5 人ではダメか？ "3 人" の根拠は何だ？ もし 4 人目のメンバーを追加したい起業家がいたら、月 $40（4 × $10）払うのか？ 3 人 → 4 人の段差が大きすぎないか？ $0 → $40 のジャンプ。これは 16 番（全か無か）の設計だ。中間プランがない。」

→ Gap 発見: Free と Pro の間の価格断崖。$0 → $40/月のジャンプが大きすぎる。中間プランの不在（16 番検知）

**🦈 大和田**: 「解約の話をしよう。解約後のデータ保持。30 日猶予とする。その 30 日間のストレージコストは誰が持つ？ 解約したユーザーのデータが 50GB あったら、30 日で約 $0.70。1000 人が解約したら $700/月の "死んだコスト" だ。それより重要なのは — 解約しようとしたユーザーに何を見せる？ "本当に解約しますか？" のリテンション施策は？ 解約理由のアンケートは？ ダウングレードの提案は？ — 解約 API だけ作って終わりか？」

→ Gap 発見: 解約時のリテンション施策（ダウングレード提案、理由収集、猶予期間の案内）が未設計

## Scene 3: 運用とリスク

**🎰 利根川**: 「支払い失敗。クレジットカードの期限切れ。これ、SaaS の解約理由の 20-30% は "involuntary churn"（意図しない解約）だ。Stripe の自動リトライだけで十分か？ ユーザーに "カードが失敗しました" のメールを送るタイミングは？ 1 回目の失敗？ 3 回目？ 最終警告？ — ユーザーの言葉で語れ。"突然使えなくなった" が最悪の体験だ。」

→ Gap 発見: 支払い失敗時のユーザーコミュニケーション（dunning email）が未設計。involuntary churn 対策がない

**🎭 ソクラテス**: 「最後に。subscriptions テーブルに `plan` が VARCHAR で入っている。"Free", "Pro", "Enterprise"。将来プランが変わったら？ "Pro Plus" が追加されたら？ 過去のレコードの "Pro" は新しい Pro と同じか？ — そもそもプラン定義がコードにハードコードされているのか DB に入っているのか。Stripe 側のプランとの同期は？ Stripe のプラン ID と DB の plan カラムがズレたら？ 6 番（関係ない話）に聞こえるかもしれないが、課金のデータ整合性は命だ。」

→ Gap 発見: プラン定義の管理場所（コード vs DB vs Stripe）が未定義。プラン変更時の既存レコードとの整合性

**👤 今泉**: 「誰が困るんですか、具体的に。経理の人です。月末に請求書を突き合わせて、Stripe の金額と自社 DB の金額が 1 円でもズレたら問題。invoice テーブルに amount_cents があるけど、Stripe 側の金額と同期するタイミングは？ webhook で更新？ webhook が欠損したら？ 経理の人が「この請求書の金額が Stripe と違う」って問い合わせてきた時のオペレーション手順は？」

→ Gap 発見: Stripe と自社 DB の金額整合性の保証メカニズムがない。webhook 欠損時のリカバリ手順がない
