# Theme 1: 認証 API — DGE レビュー

- **キャラクター**: 今泉 + 千石 + Red Team + 僕 + ソクラテス
- **パターン**: feature-extension + security-adversary
- **テンプレート**: api-design

---

## Scene 1: Happy Path

**先輩**: JWT ベースの認証 API。ログイン → トークン発行 → リフレッシュ → ログアウト。

**👤 今泉**: 「そもそも、このリフレッシュトークンの 30 日って誰が決めたんですか？ ユーザーに聞きました？ 30 日間ログインしっぱなしは嬉しいのか怖いのか。銀行アプリなら 30 日は長すぎる。SNS なら短すぎるかもしれない。アプリの性質によって違いませんか？」

→ Gap 発見: リフレッシュトークンの有効期限がアプリケーションの性質を考慮していない

**🎩 千石**: 「POST /api/auth/login のレスポンス形式が未定義です。トークンをどう返すのですか？ レスポンスボディ？ Set-Cookie ヘッダー？ 両方？ そしてエラー時 — メールアドレスが間違っているのか、パスワードが間違っているのか、区別して返すのですか？ 区別して返したらアカウントの存在が漏洩します。区別しなければユーザーが困ります。これは品質の問題です。」

→ Gap 発見: ログインのレスポンス形式が未定義。エラー時の情報漏洩 vs UX のトレードオフが未検討

**😰 僕**: 「...あの、refresh_tokens テーブルなんですけど、expires_at しかないですよね。ユーザーが 10 台のデバイスで 30 日間ログインしたら、1 ユーザーで 10 レコード。100 万ユーザーなら 1000 万レコード...。期限切れトークンの掃除はいつやるんですか...？ バッチジョブ...？ そのバッチが止まったら...？」

→ Gap 発見: 期限切れリフレッシュトークンのクリーンアップ戦略がない。テーブルが無限に膨張する

## Scene 2: エッジケース

**😈 Red Team**: 「リフレッシュトークンを盗む。手順: (1) XSS でクライアントのストレージを読む。(2) リフレッシュトークンを取得。(3) 自分のデバイスでリフレッシュ。(4) 正規ユーザーより先に新しいアクセストークンを取得。——ところで、このフローでリフレッシュトークンはローテーションされるのか？ されないなら、1 回盗むだけで 30 日間やりたい放題だ。」

→ Gap 発見: リフレッシュトークンのローテーション未実装。盗まれたら 30 日間有効

**😈 Red Team**: 「もう 1 つ。POST /api/auth/login にレート制限がない。1 秒に 1000 回ログイン試行できる。パスワードリストアタックで 10 分あればよくあるパスワードは全部試せる。IP ベースの制限？ プロキシ通せば回避できる。アカウントベースの制限が必要だ。——ちなみにアカウントロックの仕組みも書いてない。」

→ Gap 発見: ログイン試行のレート制限とアカウントロック機構がない

**🎭 ソクラテス**: 「少し待て。全員が JWT を前提に議論しているが、5 番（根拠なき一般論）に気をつけろ。"JWT はモダンな認証の標準" — 誰がそう言った？ この API はステートレスである必要があるのか？ サーバーが 1 台なら session でいい。水平スケールする予定はあるのか？ ないなら JWT は不要な複雑さだ。」

→ Gap 発見: JWT を選択した技術的根拠が未記載。セッションベースで十分な可能性（ソクラテスが 5 番詭弁を検知）

**👤 今泉**: 「前もそうだったっけ。JWT で困った事例ってありませんか？ ログアウトできない問題とか、トークンが大きすぎて Cookie に入らない問題とか。前もそうだったなら、同じ轍を踏む前に考えた方がいいですよね？」

→ Gap 発見: JWT の既知の問題（ログアウト、トークンサイズ、クロスドメイン）の考慮が未記載

## Scene 3: 運用

**🎩 千石**: 「本番でトークンの秘密鍵をローテーションする手順は？ 秘密鍵が漏洩した場合、全ユーザーの全トークンが無効になります。その時の対応手順は？ ユーザーに再ログインを強制する通知は？ これが未設計のまま本番に行くのは、お客様への侮辱です。」

→ Gap 発見: JWT 署名鍵のローテーション手順と漏洩時の対応が未設計

**😰 僕**: 「...ログのこと考えてますか？ ログイン成功・失敗のログ。不正アクセスの検知。"このアカウントに 3 カ国から同時にログインしてる" みたいな異常検知。...全部ないですよね。監視の仕組みもない。怖い...。」

→ Gap 発見: 認証イベントのログ・監視・異常検知が未設計

**🎭 ソクラテス**: 「最後に 1 つ。users テーブルに email と password_hash しかない。名前もない。プロフィールもない。"認証" と "ユーザー管理" は同じテーブルでいいのか？ 将来 OAuth（Google ログイン等）を追加する時、password_hash が NOT NULL だと困らないか？ 今の設計は "パスワード認証しか存在しない" という暗黙の前提を持っている。15 番（新概念がすべて正しいとミスリード）の逆 — "古い概念（パスワードのみ）がすべて正しい" という思い込みだ。」

→ Gap 発見: 認証方式の拡張性がない。OAuth/SSO 追加時にスキーマが破綻する（ソクラテスが 15 番を応用）
