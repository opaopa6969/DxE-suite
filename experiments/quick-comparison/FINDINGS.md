# DGE vs 素の LLM — 実験結果と解釈

## 実験概要

DGE（キャラクター対話による Gap 抽出）が素の LLM レビューと比べて何が違うかを、**isolated subagent**（DGE の context を持たない独立プロセス）を使って検証した。

### 実験条件
- **モデル**: Claude Opus 4.6（両条件とも同一）
- **テーマ**: 認証 API の設計（JWT ベース）
- **DGE**: 今泉 + 千石 + Red Team + 僕 + ソクラテス（5 キャラ）
- **素の LLM**: `isolation: "worktree"` で起動した subagent。DGE の結果を一切知らない
- **入力**: 同一の設計ドキュメント（42 行）

## 結果

| 指標 | DGE | 素の LLM（isolated） |
|------|-----|---------------------|
| **Gap 総数** | 9 | 28 |
| **Critical** | 2 | 2 |
| **High** | 6 | 9 |
| **Medium** | 1 | 11 |
| **Low** | 0 | 6 |

### 人間による重複判定

自動マッチング（dge-tool compare）では表記の違いで重複検出率が低かったため、人間が意味的に判定:

- **両方が発見**: 6 件
- **DGE のみ**: 3 件
- **素のみ**: 22 件

## DGE のみの 3 件 — 最も重要な発見

| Gap | Severity | 特徴 |
|-----|----------|------|
| リフレッシュトークン有効期限の根拠がアプリ性質を無視 | High | 「30 日」という数字の妥当性を、アプリの種類（銀行？ SNS？）で問う |
| JWT を選択した技術的根拠がない | High | ソクラテスが詭弁 5 番（根拠なき一般論）で検知。「JWT はモダンだから」は理由ではない |
| 認証方式の拡張性がない（OAuth/SSO で DB が破綻） | High | password_hash が NOT NULL だと Google ログインが追加できない。将来の拡張を問う |

**共通点: 3 件とも「設計の前提を疑う」タイプ。** 素の LLM は設計を受け入れてから改善案を出す。DGE は「そもそもこの設計でいいのか」を問う。

## 素のみの 22 件 — 網羅的チェックリスト

素の LLM が得意なのは「一般的な認証 API に必要な機能・セキュリティ要件のリスト」:

- パスワードハッシュアルゴリズム、CSRF、HTTPS、MFA、パスワードリセット、メール検証、パスワードポリシー、アカウントロック、JWT claim 定義、レスポンス形式...

これらは「認証 API のベストプラクティス」から導かれるもので、設計ドキュメントの内容に関わらず同じ指摘が出る。重要だが自明。

## Severity の評価差 — DGE は運用を重く見る

同じ Gap でも DGE と素で Severity が異なった:

| Gap | DGE | 素 |
|-----|-----|-----|
| トークンローテーション | **Critical** | Medium |
| 署名鍵ローテーション | **High** | Medium |
| 認証ログ・監視 | **High** | Medium |

DGE が運用系 Gap を重く評価する理由: **「僕」の不安ベースの思考**。「テーブルが膨張する...」「監視がない...怖い...」— この不安が Severity を押し上げる。素の LLM は冷静に「ないより あった方がいい」と Medium を付ける。

どちらが正しいかはアプリ次第だが、**運用上の問題は「後で気づくと手遅れ」なことが多い**ので、DGE の方が実践的とも言える。

## バイアス除去の効果

### 同一 context vs isolated の比較

| 指標 | 素（同一 context） | 素（isolated） |
|------|-------------------|---------------|
| Gap 総数 | 15 | **28** (+87%) |
| Critical | 1 | **2** |
| High | 4 | **9** |

**同じ LLM が DGE の結果を「知っている」と、Gap が 87% 少なくなった。**

解釈: DGE の Gap を知っている LLM は、無意識に「もう指摘されていること」を避けて出力する。これは「補完バイアス」— DGE が見つけなかったものだけを出そうとする傾向。isolated にすることで、LLM は完全に独立した視点でレビューし、本来出すべき Gap を全て出す。

**これは DGE vs 素の比較実験において isolation が必須であることの実証。**

## 結論

### DGE の強み
1. **前提を疑う**: 「JWT いる？」「30 日の根拠は？」「OAuth 追加できる？」
2. **運用を重く見る**: 僕の不安ベースで Severity が上がる
3. **詭弁を検知する**: ソクラテスが暗黙の前提を壊す
4. **再現性がある**: 同じキャラ構成なら似た角度で毎回攻める

### 素の LLM の強み
1. **網羅的**: 28 件。認証のベストプラクティスを一通りカバー
2. **速い**: 構造化なしで一気に出す
3. **実装詳細に強い**: ハッシュアルゴリズム、CSRF、JWT claim — 実装者が直接使える

### 最適な使い方

```
1. DGE で回す → 設計判断の穴を見つける（9 件、High 中心）
2. 素の LLM でも回す（isolated）→ 実装チェックリストを得る（28 件、Medium 中心）
3. マージ → 37 件の統合 Gap リスト（重複除外で ~31 件）
4. 人間がレビュー → 優先度を最終決定
```

**DGE は素の LLM の「代替」ではなく「補完」。両方やるのが最強。**

## 実験の限界

1. **N=1**: テーマ 1 つだけの結果。統計的有意性はない
2. **同一モデル**: DGE も素も Claude Opus 4.6。異なるモデルでの比較は未実施
3. **キャラ選択の影響**: 別のキャラ構成なら DGE の結果は変わる
4. **自動マッチングの精度**: dge-tool compare の word-based matching は日本語で精度が低い。LLM ベースのマッチングが望ましい
5. **DGE の context 汚染**: DGE session を実行した後に比較を分析しているので、分析自体に DGE バイアスがかかっている可能性
