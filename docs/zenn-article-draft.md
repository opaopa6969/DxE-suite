# 設計レビューが退屈すぎるので、キャラクターに殴り合いさせることにした

> 仕様書のレビューは「書いてあることの検証」。DGE は「書いてないことの発見」。
> ――ただし会話劇で。

## TL;DR

- プロジェクトルートで `npm install @unlaxer/dge-toolkit && npx dge-install`
- Claude Code に「DGE して」と言う
- キャラクターが勝手に喧嘩を始めて、仕様書の穴を見つけてくれる
- 仕事が楽しくなる（生産性が上がるかは知らない）

---

## 設計レビュー、楽しいですか？

正直に言いましょう。楽しくない。

Pull Request のレビューで「ここの型が...」「このエラーハンドリングが...」と指摘する日々。spec レビューで「この API のレスポンスコードが...」と会議室で眠くなる午後。

**DGE（Dialogue-driven Gap Extraction）** は、設計レビューを「会話劇」に変えるツールです。

キャラクターたちが設計について議論する。品質の鬼が「それはユーザーへの侮辱です」と怒り、怠惰な天才が「要らなくない？ 紅茶ください」と返す。その対話の中から、仕様書に書かれていない「穴」が浮かび上がってくる。

レビューが、ちょっと楽しくなります。

## セットアップ: 30 秒

プロジェクトのルートで:

```bash
npm install @unlaxer/dge-toolkit
npx dge-install
```

終わり。`dge/` フォルダと `.claude/skills/` にスキルファイルが入ります。

Claude Code で「DGE して」と言えば起動。

## 実際にやってみた: 認証 API の設計レビュー

とあるプロジェクトの認証 API を DGE にかけてみます。

```
Human: 認証 API の設計を DGE して
```

DGE toolkit v1.5.0

テンプレート: api-design（自動選択）
パターン: 🔧 feature-extension を推奨します。変更しますか？

```
Human: それで
```

キャラクター: 今泉 + 千石 + 僕 を推奨します。変更しますか？

```
Human: Red Team も入れて
```

---

### Scene 1: Happy Path — 「基本フロー」

> **先輩（ナレーション）**: 認証 API は JWT ベースのトークン認証。ログイン → トークン発行 → リクエストごとにトークン検証。リフレッシュトークンは 30 日有効。

**👤 今泉**: 「あの... 基本的なことなんですけど、リフレッシュトークンの 30 日って、誰が決めたんですか？ ユーザーに聞きました？ 30 日間ログインしっぱなしって、セキュリティ的にどうなんですか？ そもそもリフレッシュトークンって何ですか？」

→ Gap 発見: リフレッシュトークンの有効期限の根拠が未定義

**🎩 千石**: 「30 日？ 7 日でしょう。いえ、問題はそこではありません。リフレッシュトークンを発行した後、古いトークンはどうなるのですか？ ローテーションしないのですか？ 同時に複数のリフレッシュトークンが有効になる設計は、お客様への侮辱です。」

→ Gap 発見: リフレッシュトークンのローテーションポリシーが未定義

**😰 僕**: 「...あの、ログアウトって、どうするんですか...？ JWT はステートレスだから、サーバー側でトークンを無効化できないんじゃ...。ブラックリストを管理するんですか...？ Redis とか要りますか...？ すべてがダメになる予感がするんですけど...。」

→ Gap 発見: ログアウト（トークン無効化）の仕組みが未設計

**今泉がいきなり本質を突いた。** 「リフレッシュトークンって何ですか？」— この質問、専門家は聞けない。知ってるから。でも今泉は知らないから聞ける。そして「30 日の根拠は？」は、実は誰も答えられない。

**僕の不安は予言だった。** JWT のログアウト問題は定番のハマりポイント。僕みたいなキャラがいなかったら「実装してから気づく」やつ。

---

### Scene 2: エッジケース — 「攻撃者が来る」

> **先輩**: ここで Red Team が参戦する。攻撃者の視点で認証 API を見る。

**😈 Red Team**: 「リフレッシュトークンが漏洩した場合のシナリオ。攻撃者がトークンを盗んで、正規ユーザーより先にリフレッシュする。正規ユーザーのトークンは無効化される。攻撃者がアカウントを乗っ取り、正規ユーザーはログインできなくなる。——で、この API にレート制限はあるのか？」

→ Gap 発見: トークンリフレッシュのレート制限がない

**👤 今泉**: 「ちょっと待ってください。トークンが盗まれたって、どうやって盗むんですか？ localStorage に入ってるんですか？ Cookie ですか？ httpOnly ですか？ そもそも聞いたんですか？ フロントエンドの人に。」

→ Gap 発見: トークンの保存場所（localStorage vs httpOnly Cookie）が未決定

**😈 Red Team**: 「localStorage なら XSS で一発。httpOnly Cookie なら CSRF。どっちを選んでも攻撃ベクトルがある。で、どっちも書いてない。——セキュリティレビューの前にセキュリティの設計がない。」

**😰 僕**: 「...やっぱりすべてがダメになる予感がします...。小規模にしませんか...？ セッションベース認証でいいんじゃ...？ JWT 使う理由ってあるんですか...？ ...でも、やるしかないですけどね...」

→ Gap 発見: JWT を選択した技術的理由が未記載（セッションベースで十分な可能性）

**落ちがついた。** 「そもそも JWT いる？」— 僕が最後にぼそっと言った一言が一番重い。認証方式の選定理由が仕様書にない。ありがちだけど、普通のレビューでは「もう JWT で作り始めてるし...」で流される。DGE では流されない。僕が止める。

---

### Gap 一覧

たった 2 シーンで 6 個の Gap が出た。

| # | Gap | Severity |
|---|-----|----------|
| 1 | リフレッシュトークン有効期限の根拠がない | High |
| 2 | トークンローテーションポリシー未定義 | High |
| 3 | ログアウト（トークン無効化）が未設計 | Critical |
| 4 | レート制限がない | High |
| 5 | トークン保存場所が未決定 | High |
| 6 | JWT を選んだ理由が未記載 | Medium |

**Critical 1 件、High 4 件。** 30 分の spec レビュー会議より多い。

## なぜ「楽しい」のか

DGE が楽しいのは、キャラクターに「人格」があるからです。

今泉は本当に分かっていない。だからこそ「そもそもリフレッシュトークンって何？」と聞ける。この質問、チームの中で一番ジュニアなエンジニアが聞きたかったけど聞けなかった質問かもしれない。

千石は品質に妥協しない。「お客様への侮辱です」— このフレーズ、レビューで使いたくなりません？（使うと怒られるかもしれないけど。）

僕は不安の塊。「すべてがダメになる予感がする」— でもその予感、だいたい当たる。

Red Team は容赦ない。「で、レート制限は？」— 攻撃者は空気を読まない。

**これは「ツール」じゃなくて「演劇」に近い。** 設計の穴を見つけるという目的は同じだけど、チェックリストを消化する作業ではなく、キャラクターの対話を楽しむ体験。

## 「普通に AI にレビュー頼むのと何が違うの？」

はい、一番聞かれる質問。

Claude に素で「認証 API をレビューして」と頼んでも、それなりの指摘は返ってくる。でも違いがある。

**素の AI は空気を読む。DGE のキャラは読まない。**

素の AI: 「JWT の有効期限は適切に設定することをお勧めします」（やんわり）
今泉: 「そもそもリフレッシュトークンって何ですか？」（ド直球）
僕: 「...JWT いります？ セッションでよくないですか...？」（全否定）

素の AI は「もう実装始めてるのに今さら JWT やめろとは言いづらい」という自己検閲が働く。今泉は知らないから聞ける。僕は不安だから止められる。**キャラクターに人格があるから、AI の遠慮が外れる。**

**素の AI は 1 人で全部言う。DGE は殴り合う。**

素のレビュー結果:
```
1. リフレッシュトークンの有効期限を検討してください
2. ログアウト機能を実装してください
3. レート制限を追加してください
```

全部正しい。でも全部同じトーンで並んでるから、どれが本当にヤバいか分からない。

DGE だと千石が「品質に妥協するな」と言い、ヤンが「全部は要らない」と返し、僕が「...そもそも JWT が...」とつぶやく。**キャラ間の対立から優先度が浮かぶ。** 千石とヤンが両方「これは必要」と言った Gap は本物。

**素の AI は網羅を保証しない。DGE は構造で強制する。**

素の AI は思いついた順に指摘する。DGE は Scene 構造（Happy Path → Edge Case → 運用 → セキュリティ）とパターン（20 種）で角度を強制的に変える。「セキュリティ考えました？」と聞き忘れることがない。

**とはいえ正直に言うと:**

DGE を使わなくても「多角的にレビューして、品質の鬼と怠惰な天才とセキュリティの攻撃者の 3 つの視点で」と超具体的に頼めば、似た結果は出る。たぶん。

**でもそのプロンプトを毎回書きますか？**

DGE は「DGE して」の 4 文字。キャラもパターンも自動で選ばれる。再現性がある。そして何より、千石が怒る姿を見るのが楽しい。

## 12 人のキャラクター

DGE には 12 人の built-in キャラクターがいます。全員、日本のフィクション作品の思考パターンを参考にしたオリジナルです。

```
前提が怪しい    → 👤 今泉   「そもそも聞いたんですか？」
品質が低い      → 🎩 千石   「お客様への侮辱です」
全部複雑        → ☕ ヤン   「要らなくない？ 紅茶ください」
前に進みすぎ    → 😰 僕     「小規模にしませんか...？」
大胆さが足りない → 👑 ラインハルト 「攻めろ」
数字が甘い      → 🦅 鷲津   「IRR は？」
攻撃への耐性    → 😈 Red Team「競合がこうしたら？」
収益の現実      → 🦈 大和田  「いくら稼げるんだ？」
実装の不足      → ⚔ リヴァイ 「汚い。作れ。」
ユーザーの本音   → 🎰 利根川  「ユーザーの言葉で語れ」
隠れた問題      → 🏥 ハウス  「全員嘘をついている」
法的リスク      → ⚖ ソウル  「利用規約は書いたか？」
```

テーマによって組み合わせが変わります。API 設計なら今泉+千石+僕。セキュリティレビューなら千石+Red Team+ハウス。VC ピッチの準備なら鷲津+ラインハルト+今泉+僕。

### 好きなキャラを追加できる

12 人で足りない？ 追加しましょう。

```
Human: ベルセルクのガッツを追加して
```

LLM がガッツの性格を分析して、axes（判断速度、リスク許容度、品質へのこだわり等）をベクトルで推定。名言やトラウマまで抽出して、次回から使えるキャラとして永続保存されます。

wizard モードなら、対話形式でオリジナルキャラも作れます。「判断は早い？ 慎重？」「口癖は？」「怒るとどうなる？」— 根掘り葉掘り聞かれて、あなただけのレビュアーが誕生します。

## 20 の会話劇パターン

「いつも同じ角度のレビューになる」問題を解決するのがパターンです。

| パターン | 何をするか | 見つかるもの |
|---------|-----------|-------------|
| `zero-state` | データ 0、設定 0 で起動 | オンボーディングの穴 |
| `role-contrast` | 管理者 vs 一般ユーザーで同じ操作 | 権限の抜け |
| `before-after` | 導入前後を比較 | 本当に改善されたか |
| `scale-break` | N=1 → N=100,000 で動かす | パフォーマンスの壁 |
| `security-adversary` | 攻撃者になりきる | セキュリティの穴 |
| `disaster-recovery` | DB が飛んだ後に復旧 | バックアップの穴 |

全 20 パターン、5 つのプリセット付き。「新規プロジェクト」「リリース前チェック」「社内提案用」等、状況に合わせて選べます。

## 自動反復モード: 「実装できるまで回して」

一番クールな機能。

```
Human: 実装できるまで回して
```

DGE が自動的にパターンを変えながら繰り返し、新しい Gap が出なくなったら止まる。毎回ファイルに保存されて、最後に累計の Gap 一覧が出る。

寝てる間に 5 回回って「もう新しい穴は見つかりません」って言ってくれたら最高じゃないですか。（実際には寝てる間にはできないけど、気分的にはそう。）

## 実験: DGE vs 素の AI、本当に違うのか？

「キャラクターに喧嘩させなくても、普通に AI にレビュー頼めばよくない？」

一番聞かれる質問なので、実験しました。

### 実験方法

同じ設計ドキュメント（認証 API）を 2 つの方法でレビュー:

- **DGE**: 今泉 + 千石 + Red Team + 僕 + ソクラテスの 5 人で会話劇
- **素の AI**: 「この設計をレビューして。問題点を全部挙げて」とシンプルに依頼

**ミソ: 素の AI は「別人格」で回した。** DGE の結果を知らない、完全に独立した AI に頼んだ。同じ AI に両方やらせると「さっき DGE で出なかったやつを補完しよう」というバイアスがかかるから。

### 結果

| | DGE | 素の AI |
|---|-----|--------|
| 問題点の数 | **9 件** | **28 件** |
| ヤバい問題（Critical） | 2 件 | 2 件 |
| 重要な問題（High） | **6 件** | 9 件 |

**数は素の AI の圧勝。** 28 対 9。3 倍。

でも中身を見ると景色が変わる。

### 素の AI が出したもの（抜粋）

```
- パスワードハッシュアルゴリズムが未指定
- CSRF 対策がない
- HTTPS の強制が記載されていない
- パスワードリセット機能がない
- 多要素認証の考慮がない
...
```

正しい。全部正しい。でも **これ、「認証 API のベストプラクティス一覧」と同じ**。設計ドキュメントを読まなくても出せる。

### DGE が出したもの（素の AI には出せなかったやつ）

**ソクラテス**: 「全員が JWT を前提に議論しているが、待て。"JWT はモダンな認証の標準" — 誰がそう言った？ サーバーが 1 台なら session でいい。JWT を選んだ根拠は？」

**今泉**: 「リフレッシュトークンの 30 日って、このアプリの性質に合ってますか？ 銀行なら長すぎる。SNS なら短いかも。」

**ソクラテス**: 「users テーブルに password_hash しかない。将来 Google ログインを追加するとき、password_hash が NOT NULL だと困りませんか？」

3 件。全部 **「そもそもこの設計でいいのか」** を問う指摘。

素の AI は設計を受け入れてから「ここが足りない」を列挙する。DGE は設計自体を疑う。

### 面白い発見: 「知ってる」だけで結果が変わる

実は最初、同じ AI に DGE → 素の順番で両方やらせた。そのときの素の結果は **15 件**。

次に、DGE の結果を知らない別の AI にやらせた。結果は **28 件**。

**同じ AI でも、DGE の結果を知ってると 87% 少なくなった。** 無意識に「もう指摘されたこと」を避けてしまう。

だから比較するなら **完全に別の AI** にやらせないとダメ。（DGE toolkit v2 はこれを自動でやる機能がある。）

### で、どっちがいいの？

```
素の AI = 広く浅いチェックリスト（28 件、Medium 多め）
DGE    = 深く鋭い洞察（9 件、Critical/High 中心）
最強   = 両方やる
```

DGE toolkit v2 には「素の AI でも回してマージ」ボタンがある。DGE の会話劇を楽しんだ後に、ボタン 1 つで素のチェックリストも手に入る。

**「JWT いる？」は素の AI には一生出せない。** 素の AI は空気を読むから。ソクラテスは読まない。

## 内部構造: 会話劇のエンジン

DGE の中身は意外とシンプルです。

```
テーマ選択 → パターン選択 → キャラ選択
    ↓
会話劇生成（先輩ナレーション → キャラ対話 → Gap マーク）
    ↓
ファイル保存 + サマリー表示
    ↓
選択: もう一回 / 自動反復 / 実装する / 後で
    ↓
「実装する」→ Gap から Spec（Use Case, Tech Spec, ADR 等）を自動生成
```

フロー定義は YAML ファイルで、カスタマイズ可能。設計レビュー以外にも使えます。フロー定義を変えれば、同じエンジンで創作にも意思決定にも使える。

詳しくは [INTERNALS.md](https://github.com/opaopa6969/DGE-toolkit/blob/main/kit/INTERNALS.md) に mermaid のフロー図があります。

| 機能 | 説明 |
|------|------|
| 🎭 会話劇で Gap 発見 | キャラが議論して仕様の穴を見つける |
| 📋 Gap → Spec 変換 | Use Case, Tech Spec, ADR を自動生成 |
| 🔄 自動反復 | 収束するまで角度を変えて繰り返す |
| 📁 プロジェクト管理 | 複数テーマを TreeView で一覧 |
| 🎭 カスタムキャラ | 好きなキャラを追加。wizard でオリジナルも |
| ⚙ フローカスタマイズ | YAML でフロー定義を変更 |
| 🔧 API サーバー | キャラ管理 + 推奨エンジン（オプション） |

## まとめ: 仕事を楽しくするツール

DGE は「生産性を 10x にするツール」ではありません。

**仕事を楽しくするツール** です。

設計レビューという退屈な作業に、キャラクターの個性と対話の楽しさを持ち込む。結果として穴も見つかる。でも一番大事なのは、「DGE して」と言った後に出てくる会話劇を読むのが楽しい、ということ。

千石が怒り、ヤンが「要らなくない？」と返し、僕が「すべてがダメになる予感...」とつぶやく。

その中に、あなたの仕様書に書かれていなかった何かが、きっとある。

```bash
npm install @unlaxer/dge-toolkit
npx dge-install
```
