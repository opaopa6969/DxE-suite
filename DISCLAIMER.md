# DGE 免責事項・利用上の注意

> 対象 Gaps: 3-001, 3-002, 3-010, 3-026

---

## 1. DGE の性質と限界

DGE（Design-Gap Exploration / Dialogue-driven Gap Extraction）は**網羅的なレビュー手法ではない**。

- DGE は「書いてないことの発見」を目的とするが、すべての gap を発見することを保証しない
- 会話劇で発見されなかった gap が存在しないことを意味しない。LLM の知識・context window・キャラクター構成により、発見範囲は毎回異なる
- DGE の session 結果は**「レビュー完了」の証明にはならない**。「一定の角度から検討した記録」として扱うこと
- gap の数が多いことは品質の高さを意味しない。偽陽性（false positive）が含まれる可能性がある

---

## 2. 専門的判断の代替ではない

DGE の出力は、以下の専門的レビューの**代替にはならない**:

| 領域 | DGE ができること | DGE ではできないこと |
|------|----------------|-------------------|
| **セキュリティ** | 脅威モデルの見落としの示唆 | ペネトレーションテスト、脆弱性診断、SOC 2 監査 |
| **法務** | 法的リスクの論点提示 | 弁護士による法的助言、契約書レビュー、規制対応の判断 |
| **品質保証** | テストケースの着想 | 自動テスト、CI/CD、負荷テスト、E2E テスト |
| **アーキテクチャ** | 設計上の考慮漏れの指摘 | パフォーマンス検証、可用性の実測、障害注入テスト |

DGE の出力に基づいて判断を下す場合、該当領域の専門家の確認を経ること。

---

## 3. LLM 生成物に関する注意

DGE の会話劇と gap は LLM が生成する。以下のリスクがある:

- **Hallucination**: LLM が事実と異なる前提で gap を指摘する可能性がある。技術的に不正確な指摘、存在しないリスクの誇張が含まれうる
- **合意バイアス**: LLM はキャラクター間の対立を「丸く収める」傾向がある。本来もっと深刻な矛盾が弱められることがある
- **知識の限界**: LLM の学習データに含まれない領域（最新技術、ニッチな業界規制等）では gap 発見能力が低下する
- **再現性の欠如**: 同じ入力でも LLM のバージョン、プロバイダー、temperature 設定により異なる結果が出る

**人間によるレビューは必須**（method.md Step 2）。会話劇を読み、gap の妥当性を判断するのは人間の責任である。実績データでは、人間レビュー中に LLM が見落とした gap の 22% が追加発見されている。

## 4. 知的財産に関する注意

### キャラクター名の使用について

DGE toolkit のキャラクター名（今泉、千石、ヤン等）は、**思考パターンの参照ラベル**として使用している。キャラクターの二次創作・ファンフィクション・キャラクターグッズ等の制作を目的としたものではない。

- キャラクター名は「このような思考パターンで問いを立てる」という機能の識別子である
- DGE session の会話劇は設計レビューの手段であり、創作物としての頒布を意図していない

### 商業利用について

- MIT ライセンスは **DGE toolkit のコード・文書**に対して適用される
- MIT ライセンスは、キャラクター名の元となった作品の著作権・商標権には一切影響しない
- 商業利用時は、**各自の法的判断と責任**においてキャラクター名の使用可否を確認すること
- 商業利用に伴う知的財産権の侵害について、DGE toolkit の作者は責任を負わない

## 5. 推奨する併用手法

DGE は単独で完結する手法ではない。以下と併用することを推奨する:

| 手法 | DGE との関係 |
|------|-------------|
| **形式的レビュー**（ADR, RFC, Design Doc） | DGE で発見した gap を形式的な設計文書に反映する |
| **脅威モデリング**（STRIDE, DREAD） | DGE のセキュリティ session を脅威モデリングの入力として使う |
| **テスト駆動開発 / 自動テスト** | DGE の gap をテストケースに変換し、CI/CD で検証する |
| **ペネトレーションテスト** | DGE はセキュリティ専門家による検証の代替にはならない |
| **ユーザーテスト / ユーザビリティ調査** | DGE は実ユーザーの行動を代替しない |
| **コードレビュー** | DGE は設計レベルのレビュー。実装レベルは別途コードレビューが必要 |
| **チェックリストベースのレビュー** | DGE は「チェックリストにない項目」の発見に強い。チェックリストは既知の項目の漏れ防止に強い。相互補完 |
| **法務レビュー** | DGE の Legal gap は弁護士の確認なしに法的判断の根拠にしないこと |

---

## まとめ

DGE は「設計の穴を対話で見つける」手法であり、それ以上でもそれ以下でもない。会話劇は思考の触媒であって、最終判断ではない。発見された gap は人間が検証し、専門家が確認し、テストで裏付ける必要がある。
